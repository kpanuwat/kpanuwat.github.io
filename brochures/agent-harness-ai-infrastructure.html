Overview

Agent Harness คืออะไร และทำไมถึงสำคัญกว่าโมเดล

Agent Harness คือ software infrastructure ที่ล้อมรอบ AI Model และจัดการทุกอย่างยกเว้น reasoning ของโมเดลเอง — ตั้งแต่การเรียกใช้เครื่องมือ การจัดเก็บ memory การควบคุม context ไปจนถึงการป้องกัน error ที่อาจลุกลาม ถ้าโมเดลคือเครื่องยนต์ที่ฉลาด harness คือทั้งระบบรางรถไฟ สัญญาณไฟ และตารางเวลาเดินรถ ที่ทำให้ความฉลาดนั้นถูกส่งไปยังปลายทางที่ถูกต้องในทุกครั้ง

ปัญหาที่แก้ได้

Context Rot

เมื่อ context window เต็ม โมเดลจะเริ่มละเลยข้อมูลสำคัญที่อยู่ตรงกลาง Harness แก้ด้วย compression และ selective retrieval

ปัญหาที่แก้ได้

Hallucinated Tool Calls

โมเดลอาจเรียก function ที่ไม่มีอยู่จริงหรือส่ง parameter ผิด Harness validate ทุก call ก่อน execute จริง

ปัญหาที่แก้ได้

State Loss on Failure

Network timeout หรือ restart ทำให้ agent ลืมงานที่ทำไปแล้วทั้งหมด Harness บันทึก state ทุก checkpoint อัตโนมัติ

💡 Key insight: ข้อมูลจาก ICML 2025 แสดงว่า agent ที่ใช้ harness มี win-rate สูงกว่าอย่างมีนัยสำคัญบนโมเดลเดียวกัน — โดยไม่เปลี่ยน model weight หรือ prompt เลย
★ สาระสำคัญ: Harness ไม่ได้ทำให้โมเดลฉลาดขึ้น แต่ทำให้ความฉลาดที่มีอยู่แล้วถูกนำไปใช้ได้ตรงเป้าหมายทุกครั้ง

Section 1 — Deep-Dive Protocol

ส่วนประกอบหลัก สมมติฐาน และสิ่งที่ต้องรู้

ก่อนลงลึกในรายละเอียด ส่วนนี้สรุปโครงสร้างของ Agent Harness: ส่วนประกอบหลักที่ต้องเข้าใจ สมมติฐานที่ซ่อนอยู่ซึ่งผู้เริ่มต้นมักมองข้าม และ insight สำคัญที่จะช่วยให้มี mental model ที่ถูกต้องตั้งแต่ต้น

ส่วนประกอบหลัก (Components)

Component 1

Execution Loop

วงจร observe-think-act ที่แปลง probabilistic reasoning ของ LLM ให้กลายเป็น deterministic action ที่ตรวจสอบได้และทำซ้ำได้

Component 2

Tool Integration Layer

ชั้นเชื่อมต่อเครื่องมือภายนอกทั้งหมด พร้อม validate parameters ก่อน execute และ sanitize ผลลัพธ์ก่อนส่งกลับเข้า context

Component 3

Context & Memory

จัดการ working context, session state, และ long-term memory ข้ามหลาย session เพื่อให้ agent จำงานที่ทำไปแล้วได้เสมอ

Component 4

Control Flow & Guardrails

Phase-gating, timeout, budget cap, approval gate ที่ป้องกัน cascade failure และบังคับให้ agent ทำตามขั้นตอนที่กำหนด

Component 5

Observability & Persistence

Audit log, tracing, session checkpoint ที่ทำให้ debug ได้ ติดตามต้นทุนได้ และ resume งานได้หลัง failure โดยไม่เริ่มจากศูนย์

5 ส่วนประกอบหลักของ Agent Harness — แต่ละชั้นทำงานร่วมกันเพื่อแปลง probabilistic reasoning เป็น deterministic action

สมมติฐานที่ซ่อนอยู่ (Implicit Assumptions)

สิ่งเหล่านี้คือสิ่งที่หัวข้อ Agent Harness ถือว่าผู้อ่านรู้อยู่แล้ว — แต่ในความเป็นจริง ผู้เริ่มต้นมักไม่ทราบ การเข้าใจก่อนจะช่วยหลีกเลี่ยงความสับสนในภายหลัง

  • สมมติฐานที่ 1: LLM เป็น stateless โดยธรรมชาติ — แต่ละ session เริ่มใหม่ทั้งหมดโดยไม่จำอะไรจาก session ที่แล้ว ถ้าไม่เข้าใจเรื่องนี้ จะไม่เข้าใจว่าทำไม memory management ถึงเป็นปัญหาหลักของ agent ที่ทำงานยาวนาน
  • สมมติฐานที่ 2: Harness ≠ Framework — LangChain, CrewAI, AutoGen เป็น framework ที่ให้ components แต่ harness คือระบบ runtime ที่ assemble ทุก component และบังคับให้ทำงานถูกต้องตลอดเวลา เปรียบเหมือน framework คือกล่องเครื่องมือ แต่ harness คือสายการผลิตที่ใช้กล่องนั้น
  • สมมติฐานที่ 3: คนส่วนใหญ่คิดว่า Hallucination แก้ได้ด้วย prompt ที่ดีขึ้น — แต่ harness แก้ที่ระดับ system โดยทำให้ behavior ที่ผิดพลาดเป็นไปไม่ได้ตั้งแต่แรก ไม่ใช่แค่บอกให้โมเดล "ระวัง"

สิ่งสำคัญที่ต้องรู้ (Key Takeaways)

เจ็ด insight ต่อไปนี้เรียงจาก high-level intuition ลงสู่ actionable detail อ่านก่อนลงลึกเพื่อให้มี mental model ที่ถูกต้อง

Takeaway 1 — Big Picture

Harness คือระบบรางรถไฟของ AI Agent

โมเดลคือเครื่องยนต์ที่ฉลาด แต่ harness คือราง สัญญาณ และตารางเดินรถที่กำหนดว่าความฉลาดนั้นจะถูกใช้อย่างไร ถ้าไม่มีราง รถวิ่งไปทุกทิศทางโดยไม่ถึงปลายทาง

Takeaway 2 — Mental Model

{Agent} = {Model} + {Harness}

ความสามารถของ agent ไม่ได้มาจากโมเดลอย่างเดียว — harness เป็นตัวคูณที่ทำให้โมเดลระดับเดิมทำงานได้ดีขึ้นมากโดยไม่ต้องอัปเกรดหรือ fine-tune ใดๆ

Takeaway 3 — Key Mechanism

Initialization + Execution Loop = ความต่อเนื่อง

Harness ทำงาน 2 phase: ตั้งค่า state ครั้งเดียว แล้ววนซ้ำ load-do-verify-save ทำให้ agent ทำงานได้ข้ามหลาย session โดยไม่สูญเสีย progress

Takeaway 4 — Common Trap

Prompt ดีขึ้น ≠ Hallucination หายไป

Hallucination ที่ระดับ system เกิดจาก tool ที่ขาด validation, memory ที่เสื่อม, และ state ที่หาย — แก้ด้วย harness architecture ไม่ใช่ prompt tuning

💡 Takeaway 5–7:
  • เมื่อไหรที่ไม่ต้องการ harness: Single-turn task (summarize, Q&A) ไม่ต้องการ harness — เพิ่มความซับซ้อนโดยไม่จำเป็น ใช้เมื่อ agent ต้องการ tools หรือ memory ข้ามหลาย turn
  • เริ่มต้นอย่างไร: สร้าง Tool Layer ก่อน แล้วเพิ่ม Memory → Context Compression → Guardrails → Persistence ทีละ layer ตามปัญหาที่เจอจริง ไม่ต้องสร้างทุก component พร้อมกัน
  • ROI ที่ซ่อนอยู่: 65% ของ enterprise AI failures ใน 2026 มาจาก Harness Defects ไม่ใช่โมเดล — ลงทุนใน harness engineering มี return สูงกว่าการ upgrade โมเดล
★ Mental model: Harness คือ operating system ของ agent — โมเดลคือ CPU ที่ฉลาด แต่ OS คือสิ่งที่ทำให้ความฉลาดนั้นกลายเป็นงานที่เสร็จสมบูรณ์

Section 2 — Core Architecture

Execution Loop และ 5 ส่วนประกอบที่ทำให้ Agent ทำงานได้จริง

Agent Harness ทำงานผ่านวงจรที่วนซ้ำอย่างเป็นระบบ — ไม่ใช่แค่เรียก LLM แล้วรอคำตอบ แต่เป็นกระบวนการที่ load state, สั่งให้โมเดลคิด, ตรวจสอบผลลัพธ์, execute ด้วยเครื่องมือจริง, แล้วบันทึก progress ก่อนเริ่มรอบถัดไป วงจรนี้คือหัวใจที่แยก agent ที่ทำงานได้จริงในระยะยาว ออกจาก chatbot ที่ลืมทุกอย่างหลังปิดหน้าต่าง

Phase 1: Initialization — ตั้งค่าหนึ่งครั้ง ใช้ได้ทุก session

ก่อนที่ agent จะเริ่มทำงานจริง harness จะรัน Initialization Phase หนึ่งครั้งเพื่อตั้งค่า environment ให้พร้อม ขั้นตอนนี้สร้าง structured task list ที่เก็บไว้ใน persistent storage, initialize version control, และเขียน progress files ที่ session ต่อๆ ไปจะอ่านเพื่อรู้ว่างานถึงไหนแล้ว ความสำคัญของ phase นี้คือทำให้ state ของงานมีอยู่จริงนอก context window ของโมเดล แม้ระบบจะ restart หรือ timeout กลางทาง งานที่ทำไปแล้วก็ไม่สูญหาย

ตัวอย่างที่เห็นได้ชัด: coding agent ที่ต้องสร้าง feature ใหม่จะมี task list ที่บันทึกว่า "ขั้นตอน 1 เขียน unit test เสร็จแล้ว, ขั้นตอน 2 implement function อยู่ระหว่างดำเนินการ, ขั้นตอน 3 ยังไม่เริ่ม" ไม่ว่า session จะถูกตัดกี่ครั้ง session ใหม่จะเริ่มต่อจากจุดที่ค้างไว้เสมอ ไม่ต้องทำงานซ้ำ

เจาะลึก Initialization Pattern — สิ่งที่ต้องตั้งค่าและทำไม

ใน production harness ที่ดี Initialization Phase ควรสร้าง 4 สิ่งต่อไปนี้: (1) Task manifest — ไฟล์ JSON ที่แสดงทุก task พร้อม status และ dependency, (2) Environment snapshot — บันทึก version ของ tools และ dependencies ที่ใช้, (3) Checkpoint directory — โฟลเดอร์สำหรับเก็บ state ของแต่ละ execution step, (4) Audit log — ไฟล์ที่บันทึกทุก action พร้อม timestamp เพื่อ debug ภายหลัง

งานวิจัยของ Anthropic ที่ publish ใน 2025 แสดงว่า agent ที่ใช้ initializer-executor architecture สามารถสร้าง web application ที่ทำงานได้จริงข้ามหลาย context window — เพราะ executor แต่ละตัวอ่าน task manifest และรู้ทันทีว่าต้องทำอะไรต่อ โดยไม่ต้องมี handoff จากโมเดลตัวอื่น

Phase 2: Execution Loop — วงจรที่วนซ้ำจนงานเสร็จ

หลัง initialization Execution Loop จะทำงานซ้ำๆ จนกว่างานทั้งหมดจะเสร็จ: (1) Load state — อ่าน task manifest และ checkpoint ล่าสุด, (2) Plan & act — โมเดลตัดสินใจขั้นตอนถัดไปและเรียก tool ที่เหมาะสม, (3) Verify — harness ตรวจสอบว่าผลลัพธ์ถูกต้องก่อน mark ว่า complete, (4) Save & exit — บันทึก progress ก่อนปิด session เสมอ วงจรนี้สำคัญเพราะ step 3 (verify) คือสิ่งที่แยก agent ที่เชื่อถือได้ออกจาก agent ที่แค่ "บอกว่าเสร็จ"

ตัวอย่างที่จับต้องได้: agent ที่เขียนโค้ดต้องไม่แค่เขียนฟังก์ชันเสร็จแล้วบอกว่า done แต่ harness ต้อง run test suite จริง ถ้า test ล้มเหลว loop จะไม่ mark task เป็น complete แต่จะวนกลับไปให้โมเดลแก้ bug จนกว่า test จะผ่านทั้งหมด นี่คือความแตกต่างระหว่าง agent ที่ทำงานได้จริงกับ agent ที่แค่ดูเหมือนทำงาน

Execution Loop — Load State → Plan & Act → Verify → Save; เส้นประสีแดงคือ retry เมื่อ verify ล้มเหลว, เส้นประสีเทาคือ loop สู่ task ถัดไป

เจาะลึก: ทำไม Verify Step ถึงสำคัญที่สุดใน Loop

สถิติที่น่าสนใจ: ถ้า agent แต่ละ step มี accuracy 85% และต้องทำ 10 step โอกาสที่ทุก step จะถูกต้องคือ (0.85)^10 ≈ 20% เท่านั้น หมายความว่า 80% ของเวลา agent จะ deliver ผลลัพธ์ที่มี error อย่างน้อยหนึ่งจุด

Verify step แก้ปัญหานี้โดยตรวจจับ error ในแต่ละ step ก่อนที่มันจะสะสม แทนที่จะรอให้งานทั้งหมดเสร็จแล้วค่อยพบว่า step 3 ผิดพลาดซึ่งทำให้ step 4-10 พังตามกัน การ verify ทีละ step ตัด error propagation ได้ตั้งแต่ต้น

Pattern ที่เรียกว่า Plan-Execute-Verify (PEV) Loop ใช้ advanced model สำหรับ planning และ verification แต่ใช้ cheaper model สำหรับ intermediate execution steps — ลดต้นทุนได้มากโดยไม่เสียคุณภาพ

★ สาระสำคัญ: Execution Loop ที่ดีต้องมี Verify step — agent ที่ไม่ตรวจสอบผลลัพธ์ตัวเองแค่ "บอกว่าเสร็จ" แต่ไม่ได้ทำงานจริง

Section 3 — Context Engineering

Context & Memory Management: แก้ปัญหา Context Rot และ Lost in the Middle

LLM มี context window จำกัด และยิ่งข้อมูลเยอะขึ้น โมเดลจะเริ่มละเลยสิ่งที่อยู่ตรงกลาง ปรากฏการณ์นี้เรียกว่า "Lost in the Middle" และนำไปสู่ Context Rot ที่เป็นสาเหตุหนึ่งใน 65% ของ enterprise AI failures Harness แก้ปัญหานี้ด้วย 3 memory layer และกลยุทธ์ injection ที่ส่งข้อมูลที่ใช่ไปยังโมเดลในเวลาที่ใช่ แทนที่จะยัดทุกอย่างเข้าไปพร้อมกัน

3 ชั้นของ Memory ที่ Harness จัดการ

ระบบ memory ของ harness แบ่งเป็น 3 ชั้น ที่แต่ละชั้นมีวัตถุประสงค์และ lifecycle ต่างกัน: ชั้นแรกคือ Working Context ซึ่งคือ prompt ที่ส่งให้โมเดลในแต่ละ call — มีอายุเพียง 1 request และถูก curate อย่างระมัดระวังเพื่อให้มีแต่ข้อมูลที่ relevance สูงที่สุด ชั้นที่สองคือ Session State ซึ่งเป็น durable log ของ task ปัจจุบัน บันทึกเป็น JSON file เพื่อให้ session ถัดไปอ่านต่อได้โดยไม่ต้องเริ่มใหม่ ชั้นที่สามคือ Long-term Memory ซึ่งเก็บความรู้ข้ามหลาย task เช่น pattern ที่ใช้บ่อย, preference ของ user, หรือข้อมูลที่ดึงมาจาก knowledge base

เหตุผลที่ harness ใช้ JSON แทน Markdown สำหรับ state files: ในการทดลองจริง JSON พิสูจน์ว่า reliable กว่ามากเมื่อโมเดลต้องทั้งอ่านและเขียนไฟล์ข้าม session เพราะ Markdown มี syntax ที่ ambiguous ทำให้โมเดลบางครั้งตีความผิดเมื่อ parse กลับมาเป็น structured data

เจาะลึก: Lost in the Middle และวิธีที่ Harness แก้ไข

งานวิจัยพบว่า LLM มีความสามารถในการดึงข้อมูลที่สูงที่สุดเมื่อข้อมูลนั้นอยู่ที่ต้นและท้ายของ context แต่ความสามารถตกลงอย่างมีนัยสำคัญสำหรับข้อมูลที่อยู่ตรงกลาง ปรากฏการณ์นี้ทำให้ agent ที่ dump ทุกอย่างเข้า context พร้อมกันทำงานได้แย่ลงเมื่องานซับซ้อนขึ้น

Harness แก้ปัญหานี้ด้วย 2 กลยุทธ์: (1) Compaction — summarize ข้อมูลเก่าให้สั้นลงก่อนที่ context จะเต็ม เก็บแค่ essence ไม่ใช่ raw conversation history ทั้งหมด (2) Just-in-time injection — ดึงเฉพาะ document ที่ relevant ต่อ task ปัจจุบันเข้ามา แทนที่จะ preload ทุก document ตั้งแต่ต้น การวาง critical information ที่ขอบของ context (ต้นหรือท้าย) ยังช่วย attention ได้อีกด้วย

Context Compression: ลด Token ไม่ใช่ข้อมูล

Context Compression ทำหน้าที่ต่างจาก truncation ธรรมดา — แทนที่จะตัดข้อมูลเก่าออก มันสรุปข้อมูลเหล่านั้นให้อยู่ในรูป compressed representation ที่ยังคง semantic เดิมไว้ แต่ใช้ token น้อยกว่ามาก เปรียบเหมือนการย่อหนังสือทั้งเล่มให้เป็นสรุป 5 หน้า แทนที่จะฉีกหน้า 1-200 ออกเพราะ "ไม่มีที่วาง"

ผลลัพธ์ที่วัดได้จาก production systems: การใช้ Context Compression ร่วมกับ KV-cache locality และ semantic routing ลดต้นทุน token จาก $3.00 ต่อ million token เหลือ $0.30 ซึ่งเป็นการลด 10 เท่าโดยไม่เปลี่ยนโมเดลหรือคุณภาพผลลัพธ์ พร้อมกับลด latency ลง 4 เท่าเนื่องจาก cache hits ที่มากขึ้น

เจาะลึก: KV-cache และ Semantic Routing ทำงานอย่างไร

KV-cache locality ทำงานโดยรักษา prefix ของ system prompt ให้ stable ข้ามหลาย request เมื่อ prefix เหมือนกัน model สามารถ reuse cached computation แทนที่จะ process ซ้ำทุกครั้ง ถ้า system prompt เปลี่ยนทุก request cache จะ miss ทุกครั้งและเสีย latency

Semantic routing ใช้ embedding เพื่อจับคู่ query กับ relevant context documents แทนที่จะใส่ทุก document เข้า prompt โดยตรง harness จะ vectorize query, ค้นใน vector store, แล้วดึงเฉพาะ top-k results ที่ semantically similar ทำให้ context ที่ส่งให้โมเดลมีความ relevance สูงแม้ knowledge base จะมีเนื้อหามากแค่ไหน

★ สาระสำคัญ: Context ที่ดีไม่ใช่ context ที่เยอะที่สุด แต่คือ context ที่ relevant ที่สุดต่อ task ปัจจุบัน — harness คือระบบที่รู้จักความแตกต่างนั้น

Section 4 — Control & Safety

Guardrails: ลด Hallucination ที่ระดับ System ไม่ใช่ Prompt

วิธีที่ผู้คนส่วนใหญ่พยายามแก้ Hallucination คือเขียน prompt ที่ดีขึ้น — บอกโมเดลให้ "ระวัง" หรือ "ตรวจสอบก่อนตอบ" แต่ approach นี้ไม่ scalable เพราะมันพึ่งพาความน่าเชื่อถือของโมเดลเพียงอย่างเดียว Harness แก้ปัญหาในมุมที่ต่างออกไปโดยสิ้นเชิง: แทนที่จะบอกให้โมเดล "ระวัง" harness สร้างระบบที่ทำให้ behavior ที่ผิดพลาดเป็น structurally impossible ตั้งแต่แรก

Tool Validation: ป้องกัน Hallucinated Tool Calls

Hallucination ที่พบบ่อยที่สุดในระบบ agentic ไม่ใช่โมเดลบอกข้อมูลผิด — แต่คือโมเดลพยายามเรียก function ที่ไม่มีอยู่จริง หรือส่ง parameter ผิด type ให้ function ที่มีอยู่ ปัญหานี้เกิดเพราะโมเดลอาจ "จินตนาการ" tool ที่มันคิดว่าควรมี และเรียกมันโดยไม่รู้ว่ามันไม่มีจริง

Tool Validation Layer ใน harness แก้ปัญหานี้โดยสร้าง strict schema สำหรับทุก tool call ก่อนที่ harness จะ execute call ใดๆ มันจะตรวจสอบ: ชื่อ function มีอยู่จริงหรือไม่, parameter types ถูกต้องหรือไม่, required parameters ครบหรือไม่, และค่าอยู่ใน allowed range หรือไม่ ถ้า validation ล้มเหลว harness จะ return error message ที่อธิบายชัดเจนว่าผิดตรงไหนและควรแก้อย่างไร โมเดลจะใช้ข้อมูลนี้แก้ call ใหม่แทนที่จะ waste token กับ broken call ที่ไม่มีวันสำเร็จ

เจาะลึก: วิธีที่ OpenAI และ Anthropic ออกแบบ Error Messages เพื่อสอนโมเดล

OpenAI documented pattern ที่น่าสนใจ: เขียน linter error messages ที่ออกแบบมาเพื่อสอนโมเดลให้แก้ได้เอง แทนที่จะบอกแค่ "ERROR: invalid parameter" harness จะ return "ERROR: parameter 'timeout' ต้องเป็น integer (milliseconds) แต่ได้รับ string '30s' — ใช้ 30000 แทน" ทุก failure message กลายเป็น learning context ที่ทำให้ agent แก้ได้ในรอบถัดไปโดยอัตโนมัติ

แนวคิดนี้ขยายไปสู่ "environment engineering" — แทนที่จะทำให้โมเดลฉลาดขึ้น เราทำให้ environment ฉลาดขึ้นโดยออกแบบ API responses, error messages, และ tool schemas ให้ "agent-legible" คือโมเดลอ่านแล้วรู้ทันทีว่าต้องทำอะไรต่อ

Approval Gates: Human-in-the-Loop สำหรับ High-risk Operations

ไม่ใช่ทุก operation ที่ควรให้ agent ทำเองโดยอัตโนมัติ Approval Gate คือ checkpoint ที่ harness หยุดรอการยืนยันจากมนุษย์ก่อนดำเนินการต่อ สำหรับ operation ที่มีความเสี่ยงสูง เช่น การลบข้อมูล, การ publish content, การ modify production database, หรือการส่ง external request ที่มีผล irreversible

ใน Supervisor Pattern ที่ใช้ใน production systems การ circuit breaker จะ trigger อัตโนมัติ เมื่อ agent พยายามทำ operation ที่อยู่ใน "always-ask" list ระบบจะบันทึก proposed action ทั้งหมดพร้อม context ไว้ใน audit log แล้วส่ง notification ให้ผู้ดูแลระบบ approve หรือ reject ก่อนที่ harness จะ execute จริง pattern นี้ทำให้ระบบ agentic สามารถทำงาน autonomous ได้ในส่วนใหญ่ โดยยังคงมี human oversight ในจุดที่สำคัญที่สุด

เจาะลึก: Budget Cap และ Phase-gating ป้องกัน Runaway Agent

Budget Cap กำหนด token budget ต่อ session และ per-task ถ้า agent ใช้เกิน budget harness จะหยุดและ require human review ก่อนดำเนินการต่อ นี่ป้องกัน "grep-spree" — agent ที่วนหา information โดยไม่มีทิศทางจนเผาผลาญ token มหาศาล

Phase-gating บังคับให้ agent ทำตาม workflow ที่กำหนดไว้ล่วงหน้า โดยไม่อนุญาตให้ข้ามขั้นตอน เช่น research agent ต้องผ่าน "gather → analyze → verify → report" ตามลำดับ ไม่สามารถ jump ไป report โดยตรงโดยไม่ verify ก่อน Phase-gating ทำให้ audit ง่าย: ทุก decision มี explicit record ว่าเกิดขึ้นที่ phase ไหน

⚠ ข้อควรระวัง: ระวัง over-guardrailing — guardrail ที่มากเกินไปทำให้ agent ทำงานช้าและต้องมี human approval บ่อยจนไม่ต่างจาก manual workflow ออกแบบ guardrail เฉพาะ high-risk operations เท่านั้น
★ สาระสำคัญ: Harness แก้ Hallucination โดยทำให้ incorrect behavior เป็น structurally impossible — ไม่ใช่โดยบอกให้โมเดล "ระวัง" มากขึ้น

Section 5 — Architecture Patterns

3 รูปแบบ Harness Architecture ที่ใช้ใน Production

ไม่มี harness architecture ที่ดีที่สุดสำหรับทุก use case — ทางเลือกขึ้นอยู่กับ ขนาดของงาน ระดับ autonomy ที่ต้องการ และ risk tolerance ของระบบ ทั้ง 3 pattern ด้านล่างมี trade-off ที่ชัดเจน และแต่ละอันเหมาะกับบริบทที่ต่างกัน

Single-Agent Supervisor: หนึ่ง Agent ทุกอย่าง

Pattern ที่ง่ายที่สุด: โมเดลหนึ่งตัวทำงานในวงจร loop พร้อม tools, memory, และ verification Harness จัดการทุกอย่างรอบโมเดลตัวเดียว ทั้ง routing, context management และ error recovery Pattern นี้เหมาะสำหรับ bounded tasks ที่มีขอบเขตชัดเจนเช่น customer support agent ที่มี knowledge base, code review agent, หรือ research summarizer ที่ทำงานกับเอกสารชุดเดียว

ข้อดีหลักคือความเรียบง่าย — ไม่มี handoff ระหว่าง agent ทำให้ debug ง่าย และ state management ตรงไปตรงมา ข้อจำกัดคือ context window ของโมเดลเดียวอาจไม่พอสำหรับงานที่ยาวมาก และ bottleneck ที่ขั้นตอน reasoning หลักไม่สามารถ parallelize ได้

เหมาะสำหรับ: Bounded tasks, customer support, code review, research summarization — งานที่มีขอบเขตชัดเจนในหนึ่ง session

Initializer-Executor: แยก Setup ออกจาก Execution

Pattern นี้แยก agent ออกเป็น 2 ประเภทที่มีหน้าที่ต่างกัน: Initializer agent ทำงานหนึ่งครั้งเพื่อสร้าง project structure, task list, และ environment ที่สมบูรณ์ จากนั้น Executor agents หลายตัวสามารถรัน independently แต่ละตัวทำ incremental progress บน task list เดียวกัน, run tests, commit changes, และ update progress files ทุก session ของ executor share project environment เดียวกันเป็น distributed memory

ความงามของ pattern นี้คือ executor ทุกตัวสามารถ fail แล้วรีสตาร์ทได้โดยไม่เสีย progress เลย เพราะ task list บอกทุกอย่าง Anthropic ใช้ pattern นี้สาธิตการสร้าง web application ที่สมบูรณ์ข้ามหลาย context window — ทีม enterprise ที่ต้องการ multi-day coding workflows หรือ large-scale refactoring ควรเริ่มที่ pattern นี้ก่อน

เหมาะสำหรับ: Long-running projects, multi-day coding tasks, large refactoring — งานที่ต้องข้ามหลาย session และต้องการ fault tolerance สูง

Multi-Agent Coordination: Specialist Agents ทำงานร่วมกัน

Pattern ที่ซับซ้อนที่สุด: Specialist agents หลายตัวทำงานคนละบทบาท เช่น researcher agent, writer agent, reviewer agent และ coordinator agent ที่จัดการ handoff ระหว่างตัว Harness ทำหน้าที่สำคัญมากในที่นี้: จัดการว่าแต่ละ agent ได้รับ context อะไร โดยให้แต่ละตัวเห็นเฉพาะผลงานที่ relevant ต่อ role ของตัวเอง ไม่ใช่ประวัติทั้งหมด

ข้อดีคือ parallel processing ที่ทำให้งานซับซ้อนเสร็จเร็วขึ้นมาก และแต่ละ agent ถูก fine-tune สำหรับ task ที่เฉพาะเจาะจง ทำให้คุณภาพสูงกว่าการให้ agent เดียวทำทุกอย่าง ข้อจำกัดคือ complexity สูงมาก โดยเฉพาะในเรื่อง error recovery เมื่อ agent ตัวหนึ่ง fail กลางกระบวนการ — ต้องออกแบบ retry logic และ compensation mechanism อย่างระมัดระวัง

เหมาะสำหรับ: Complex research workflows, competitive intelligence pipelines, content production systems — งานที่ parallel processing ให้ประโยชน์ชัดเจน

Section 6 — Ecosystem & Practice

Ecosystem ของ Harness Frameworks และคู่มือเริ่มต้น

ใน 2026 มี framework ที่ช่วย build agent harness หลายตัว แต่ละตัวมีจุดแข็งต่างกัน สิ่งสำคัญที่ต้องเข้าใจก่อนเลือกคือ framework ให้ components แต่ harness คือสิ่งที่คุณต้อง build จาก components เหล่านั้น รวมถึง business logic, validation rules, และ domain-specific integrations ที่เฉพาะเจาะจงกับระบบของคุณ ไม่มี framework ไหนทำ harness ให้คุณ 100% ได้

Framework หลักในปี 2026 และจุดแข็งของแต่ละตัว

LangGraph โดย LangChain ใช้ graph-based state machine ที่ให้ explicit control flow ผ่าน conditional edges ทำให้ workflow ที่ซับซ้อนมีความชัดเจนและ debug ง่าย benchmark ปี 2026 แสดง task success rate 87% ซึ่งสูงที่สุดในกลุ่ม เหมาะสำหรับทีมที่ต้องการ fine-grained control ของ execution flow และ workflow ที่มีหลาย conditional branches ขึ้นอยู่กับผลลัพธ์ของแต่ละ step

CrewAI ใช้ role-based architecture ที่ agent แต่ละตัวมี persona ชัดเจน (researcher, writer, critic) ทำให้ตั้งค่า multi-agent team ได้เร็วที่สุด average latency 1.8 วินาทีสำหรับ multi-agent execution เหมาะสำหรับ content production, research pipelines, และทีมที่ต้องการ prototype ออกมาเร็ว ก่อนที่จะ optimize สำหรับ production

AutoGen / AG2 โดย Microsoft เน้น Docker-native sandboxing และ multi-turn debate pattern ที่ให้ agent หลายตัว argue กันเพื่อ refine ผลลัพธ์ มี 54,000+ GitHub stars ณ เดือนพฤษภาคม 2026 เหมาะสำหรับ code-heavy workflows ที่ต้องการ execution isolation และทีมที่ integrate กับ Azure

เจาะลึก: MCP และ A2A Protocol — มาตรฐานใหม่ของ Harness Interoperability

Model Context Protocol (MCP) โดย Anthropic กำหนดมาตรฐาน vertical interaction ระหว่าง agent กับ tools เปรียบได้กับ "USB-C สำหรับ AI" — ทำให้ tool ที่สร้างครั้งเดียวใช้งานได้กับ harness ทุกตัว โดยไม่ต้อง rewrite integration ใหม่ทุกครั้ง

Agent-to-Agent (A2A) Protocol โดย Linux Foundation กำหนดมาตรฐาน horizontal interaction ระหว่าง agent กับ agent ข้ามต่าง framework ทำให้ agent ที่สร้างด้วย LangGraph สามารถ delegate task ให้ agent ที่สร้างด้วย CrewAI ได้โดยตรง โดยไม่ต้องมี adapter พิเศษ ทั้งสอง protocol รวมกันทำให้ ecosystem ของ harness เป็น vendor-independent อย่างแท้จริง

คู่มือเริ่มต้น: ขั้นตอนสร้าง Harness แบบ Minimal

สำหรับผู้ที่ต้องการเริ่มสร้าง agent harness จากศูนย์ วิธีที่ดีที่สุดคือเริ่มจาก minimal viable harness แล้วเพิ่ม layer ตามปัญหาที่เจอจริง ไม่ใช่สร้างทุก component พร้อมกันตั้งแต่ต้น ตัวอย่างด้านล่างแสดง pattern พื้นฐานของ Python harness ที่มี tool validation และ state persistence: 

import json, os
from pathlib import Path

class AgentHarness:
    def __init__(self, agent_fn, tools: dict, state_path: str):
        self.agent_fn = agent_fn
        self.tools = tools
        self.state_path = Path(state_path)

    def load_state(self) -> dict:
        if self.state_path.exists():
            return json.loads(self.state_path.read_text())
        return {"tasks": [], "completed": [], "context": []}

    def save_state(self, state: dict):
        self.state_path.write_text(json.dumps(state, indent=2, ensure_ascii=False))

    def execute_tool(self, name: str, params: dict):
        if name not in self.tools:
            return {"error": f"Tool '{name}' not found. Available: {list(self.tools)}"}
        schema = self.tools[name]["schema"]
        for req in schema.get("required", []):
            if req not in params:
                return {"error": f"Missing required param '{req}' for tool '{name}'"}
        return self.tools[name]["fn"](**params)

    def run_loop(self, task: str):
        state = self.load_state()
        state["tasks"].append(task)
        for step in range(20):  # max steps = budget cap
            action = self.agent_fn(task, state["context"])
            if action["type"] == "done":
                state["completed"].append(task)
                break
            result = self.execute_tool(action["tool"], action["params"])
            state["context"].append({"step": step, "action": action, "result": result})
            self.save_state(state)  # checkpoint every step

โค้ดด้านบนแสดงโครงสร้างหลัก 4 ส่วน: load_state() และ save_state() ทำหน้าที่ State Persistence — บันทึก checkpoint ทุก step เพื่อให้ resume ได้หลัง failure execute_tool() ทำหน้าที่ Tool Validation — ตรวจสอบว่า tool มีอยู่และ parameters ถูกต้องก่อน execute จริง และ run_loop() คือ Execution Loop ที่มี budget cap (max 20 steps) เพื่อป้องกัน runaway agent ทุก step ที่รันจะถูก checkpoint ก่อนเริ่ม step ถัดไปเสมอ

ขั้นตอนถัดไปหลังจาก minimal harness นี้ทำงานได้แล้ว: เพิ่ม Context Compression ใน step ที่ context ยาวเกิน 4,000 tokens, เพิ่ม Approval Gate สำหรับ tools ที่ทำ write operations, และเพิ่ม Observability ด้วย structured logging ที่บันทึก token usage ต่อ step เพื่อ track ต้นทุน

★ สาระสำคัญ: เริ่มจาก 3 บรรทัด: tool validation + state persistence + budget cap — แล้วเพิ่ม layer ตามปัญหาที่เจอจริง ไม่ใช่ตาม feature list

Key Takeaways

  • Harness = Operating System ของ Agent: โมเดลคือ CPU แต่ harness คือ OS ที่แปลง probabilistic output เป็น deterministic action ที่เชื่อถือได้และทำซ้ำได้
  • 65% ของ failures มาจาก Harness ไม่ใช่โมเดล: ลงทุน optimize harness ก่อน upgrade โมเดล — มี ROI สูงกว่าและเร็วกว่า
  • 3 ชั้น Memory ที่ต้องมี: Working Context (per request), Session State (per task, JSON), Long-term Memory (across tasks) — ขาดชั้นใดชั้นหนึ่งจะทำให้ agent ลืมและ redo งานซ้ำ
  • Verify ทุก Step ไม่ใช่ทุก Task: ถ้า step accuracy 85% และมี 10 steps โอกาสสำเร็จทั้งหมดคือ 20% เท่านั้น — verify ระหว่างทางตัด error propagation
  • เริ่มจาก Minimal แล้วเพิ่ม Layer: Tool Validation + State Persistence + Budget Cap คือ harness ขั้นต่ำที่ใช้งานได้จริง — อย่าสร้างทุก component พร้อมกัน
  • Framework ≠ Harness: LangGraph, CrewAI, AutoGen เป็น components ที่ดี แต่ harness คือสิ่งที่คุณต้อง build จาก business logic และ domain integrations ของตัวเอง
  • Context Compression ลดต้นทุน 10x: KV-cache locality + semantic routing ลด $3.00 → $0.30 ต่อ MTok และลด latency 4x โดยไม่ต้องเปลี่ยนโมเดล

Glossary

Agent Harness
Software infrastructure ที่ล้อมรอบ LLM และจัดการ tool execution, memory, context, และ error recovery ทั้งหมด
Context Rot
การที่ LLM เริ่มละเลยข้อมูลสำคัญเมื่อ context window เต็ม โดยเฉพาะข้อมูลที่อยู่ตรงกลาง (Lost in the Middle)
Execution Loop
วงจร Load → Plan & Act → Verify → Save ที่วนซ้ำจนงานเสร็จ พร้อม checkpoint ทุก step
Guardrails
ระบบ validation และ approval ที่ทำให้ incorrect behavior เป็น structurally impossible แทนที่จะพึ่ง prompt
MCP (Model Context Protocol)
มาตรฐาน open-source โดย Anthropic สำหรับ agent-to-tool interaction ทำให้ tool ที่สร้างครั้งเดียวใช้ได้กับ harness ทุกตัว
★ ถ้าจำได้อย่างเดียว: {Agent} = {Model} + {Harness} — ลงทุน harness ก่อน upgrade โมเดล เพราะ 65% ของ failures มาจาก infrastructure ไม่ใช่ reasoning
