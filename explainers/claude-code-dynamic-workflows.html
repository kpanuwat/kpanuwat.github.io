Dynamic Workflows ต่างจากฟีเจอร์อื่นอย่างไร?

Claude Code มีวิธีจัดการงานหลายแบบ แต่ละแบบเหมาะกับปัญหาต่างกัน การเลือกผิดประเภทหมายถึงเสียเวลาและ Token โดยไม่จำเป็น ตารางด้านล่างเปรียบเทียบตรงๆ:

ฟีเจอร์ เน้น เหมาะกับ ไม่เหมาะกับ
Skills กระบวนการซ้ำได้ (HOW) งานมีสูตรตายตัว เช่น deploy, test งานที่ต้องการความกว้าง
Sub-agents (เดี่ยว) งานเดี่ยวแบบ Off-load งานเดี่ยวที่แยกออกจาก Context หลัก งานที่ต้องรันหลายตัวพร้อมกัน
Agent Teams ทีมขนาดเล็ก บทบาทตายตัว งานที่รู้บทบาทล่วงหน้า เช่น Frontend+Backend งานที่ยังไม่รู้ว่าต้องใช้ทักษะอะไร
/goal ความลึก (DEPTH) — วนลูปจนจบ ปัญหาที่ต้องลองซ้ำหลายรอบ งานที่แยกส่วนได้แล้วทำพร้อมกันได้
Dynamic Workflows ★ ความกว้าง (WIDTH) — ขนานจำนวนมาก งานชิ้นย่อยอิสระจำนวนมาก สเกลใหญ่ งานต้องทำตามลำดับ หรืองานเล็กเกิน
★ สาระสำคัญ: Dynamic Workflows = WIDTH play; /goal = DEPTH play — เลือกจากว่า bottleneck คือ "ต้องทำหลายอย่างพร้อมกัน" หรือ "ต้องคิดซ้ำจนได้คำตอบ"

เจาะลึก — สถาปัตยกรรม Fan-out / Fan-in

วิธีที่ง่ายที่สุดในการเข้าใจ Dynamic Workflows คือนึกภาพ "วาทยากร (Conductor) กับวงออร์เคสตรา" วาทยากร (Master Agent) ไม่ได้เล่นเครื่องดนตรีทุกชิ้นเอง แต่เขียนโน้ตให้นักดนตรีแต่ละคน (Sub-agents) เล่นพร้อมกัน แล้วรวบรวมเสียงทั้งหมดออกมาเป็นซิมโฟนีชิ้นเดียว — นั่นคือ Fan-out → Fan-in

ขั้นตอนที่ 1: Fan-out — กระจายงาน

เมื่อคุณส่งคำสั่งขนาดใหญ่ Claude (ในฐานะ Orchestrator) จะทำ 3 สิ่ง: (1) วิเคราะห์งานและตัดสินใจว่าส่วนไหนทำพร้อมกันได้, (2) เขียนสคริปต์ JavaScript ไดนามิกที่เก็บ "แผนงาน" ทั้งหมดไว้ในตัวแปร (ไม่ใช่ใน Context window ของคุณ), (3) สั่ง Sub-agents จำนวนมากออกไปทำงานพร้อมๆ กัน โดยแต่ละตัวได้รับไฟล์/งานส่วนของตัวเองและ Context window ที่สะอาด

สายแดงประ = Adversarial verification — เอเจนต์คู่โต้แย้งผลลัพธ์ของกันและกัน

ขั้นตอนที่ 2: Adversarial Verification — ระบบตรวจสอบคุณภาพในตัว

หลังจาก Sub-agents ทำงานเสร็จ ไม่ใช่ว่าผลลัพธ์จะถูกรวบรวมทันที Dynamic Workflows มีกลไกพิเศษคือการส่ง "เอเจนต์นักวิจารณ์" เพิ่มเติมเข้าไปพยายาม "หักล้าง (Refute)" ผลลัพธ์ที่ได้ กระบวนการนี้วนซ้ำจนกว่าทุกฝ่ายจะ Converge หรือตกลงกันได้ — นั่นคือเหตุผลที่โปรเจกต์ Bun สามารถแปลงโค้ด 750,000 บรรทัดจาก Zig ไป Rust แล้วยังผ่านเทสต์ได้ถึง 99.8%

เจาะลึก — Adversarial Verification ทำงานอย่างไร?

เมื่อ Sub-agent หนึ่งส่งผลลัพธ์กลับมา Orchestrator อาจสั่ง Sub-agent คู่โต้แย้ง (Adversarial agent) เข้าไปตรวจสอบโดยเฉพาะ โดยให้ Context เต็มของงานชิ้นนั้น พร้อมคำสั่งให้พยายามหาข้อผิดพลาดอย่างเต็มที่

ถ้า Adversarial agent พบข้อผิดพลาด Orchestrator จะส่ง Sub-agent ตัวเดิมกลับไปแก้ไข แล้วส่ง Adversarial agent ใหม่เข้าไปตรวจซ้ำ วงจรนี้จะวนจนผลลัพธ์ผ่านการโต้แย้ง กระบวนการทั้งหมดยังคงบันทึก Progress ไว้ทุกขั้นตอน ทำให้ถ้าเกิด Crash ระหว่างทาง ก็ไม่ต้องเริ่มใหม่ตั้งแต่ต้น

ขั้นตอนที่ 3: Fan-in — สังเคราะห์ผลลัพธ์เดียว

เมื่อ Sub-agents ทั้งหมดผ่านการตรวจสอบแล้ว ผลลัพธ์จะถูกส่งกลับมายัง Orchestrator ซึ่งจะทำหน้าที่ประมวลผลรวมและสร้าง Output สุดท้ายเดียวให้คุณ — คุณจะเห็นเพียงผลลัพธ์สรุปในหน้าต่างแชทหลัก ไม่ใช่ข้อมูลดิบจาก Sub-agents ทุกตัว ซึ่งช่วยให้ Context window ของแชทหลักยังสะอาดอยู่เสมอ

★ สาระสำคัญ: Fan-out คือ "กระจาย" Fan-in คือ "รวบ" Adversarial Verification คือ "โต้แย้งก่อนรวบ" — ทั้งสามคือกลไกหลักที่ทำให้ Workflows ทั้งเร็วและน่าเชื่อถือ

เจาะลึก — JavaScript Script ที่ Claude เขียนเอง

ส่วนที่น่าสนใจที่สุดของ Dynamic Workflows คือ Claude ไม่ได้เพียงแค่ "สั่ง" Sub-agents แต่มันเขียนโปรแกรม JavaScript ขึ้นมาทั้งชิ้นเพื่อควบคุมการทำงาน คิดว่ามันเหมือนกับ Claude กำลังเขียน Script สำหรับผู้กำกับภาพยนตร์ (Director) ที่จะบอกนักแสดง (Sub-agents) ว่าใครทำอะไร เมื่อไหร่ รอใคร และถ้าเกิดข้อผิดพลาดต้องทำยังไง

ทำไมต้องเป็น JavaScript? ไม่ใช่ Prompt ธรรมดา?

Prompt ธรรมดาอยู่ใน Context window ของ Claude — ซึ่งมีขนาดจำกัด และยิ่งยาวยิ่งช้า แต่ JavaScript Script อยู่ "นอก" Context window สคริปต์เก็บ State ของงานทั้งหมด: งานไหนเสร็จแล้ว, งานไหนกำลังรัน, ผลลัพธ์ชั่วคราวของใคร — ทั้งหมดนี้ไม่กิน Token ของแชทหลักเลย Claude จึงสามารถจัดการงาน 1,000 Sub-agents ได้โดยที่หน้าต่างสนทนาของคุณยังเบาและเร็ว

เจาะลึก — โครงสร้างสคริปต์มีหน้าตาเป็นอย่างไร?

เมื่อ Claude เขียนสคริปต์ขึ้นมา โดยทั่วไปจะมีองค์ประกอบดังนี้:

  • Task Queue — รายการงานทั้งหมดที่ต้องทำ พร้อม Dependencies ว่าชิ้นไหนต้องรอชิ้นไหน
  • Concurrency Controller — ควบคุมให้รัน Sub-agents พร้อมกันไม่เกิน 16 ตัว
  • Result Buffer — เก็บผลลัพธ์ชั่วคราวของแต่ละ Sub-agent
  • Checkpoint Writer — บันทึก Progress ทุกครั้งที่งานชิ้นหนึ่งเสร็จ
  • Synthesis Trigger — เรียก Fan-in เมื่องานทั้งหมดเสร็จ

สคริปต์นี้รันอยู่ใน Environment ที่ Claude Code จัดเตรียมให้โดยตรง ทำให้สามารถเรียก Claude API เพื่อสร้าง Sub-agent ใหม่ได้แบบ Programmatic

Progress Checkpointing — ทำไมงานถึงหยุดกลางคันได้โดยไม่เสียหาย

หนึ่งในคุณสมบัติที่ทรงคุณค่าที่สุดของ Dynamic Workflows คือการบันทึก Progress ไว้ตลอดเวลา เมื่อ Sub-agent แต่ละตัวทำงานเสร็จ Orchestrator จะเขียนบันทึกผลลัพธ์นั้นลงไปทันที ดังนั้นหากอินเทอร์เน็ตหลุดหรือ Session ถูกตัด คุณสามารถกลับมา Resume ได้จากจุดที่ค้างอยู่ — ไม่ต้องเสีย Token ในการรันซ้ำกับงานที่เสร็จไปแล้ว

★ สาระสำคัญ: สคริปต์ JS คือ "สมองนอกตัว" ของ Orchestrator — เก็บ State ทั้งหมดโดยไม่ใช้ Context window ทำให้ Workflow ใหญ่แค่ไหนก็ไม่ทำให้แชทหลักช้าลง

Use Cases — งานประเภทไหนที่ Dynamic Workflows เปลี่ยนเกม

Dynamic Workflows ไม่ใช่เครื่องมือสำหรับทุกงาน แต่เมื่อเจอปัญหาที่ใช่ มันสามารถทำในสิ่งที่เคยเป็นไปไม่ได้ให้กลายเป็นเรื่องของชั่วโมงหรือแม้แต่นาที หัวใจของทุก Use case ที่ได้ผลดีคือหลักการเดียวกัน: งานต้องแบ่งเป็นชิ้นย่อยที่เป็นอิสระต่อกันได้

กรณีที่ 1: กวาดล้าง Codebase ขนาดใหญ่ (Klarna)

บริษัท Klarna ใช้ Dynamic Workflows ในการค้นหา "Dead code" — โค้ดที่ไม่ได้ถูกเรียกใช้งานแล้วแต่ยังอยู่ในระบบ ซึ่งเครื่องมือ Static analysis แบบดั้งเดิมตรวจไม่พบ ปัญหาคือถ้าให้เอเจนต์ตัวเดียวอ่านไฟล์ทีละไฟล์ เมื่อถึงไฟล์ที่ 200 Context window จะเต็มและ "ลืม" เนื้อหาจากไฟล์แรกๆ แล้ว Dynamic Workflows แก้ปัญหานี้ด้วยการส่ง Sub-agents แยกกันไปตรวจสอบแต่ละโมดูล — แต่ละตัวมีความจำสะอาดและโฟกัสงานของตัวเอง

เจาะลึก — ทำไม Context Pollution ถึงร้ายแรง?

Context window ของ LLM ไม่ใช่หน่วยความจำที่ "จำได้ทุกอย่าง" ยิ่ง Context ยาวขึ้น โมเดลจะให้ความสำคัญกับเนื้อหาที่อยู่ใกล้ "ปัจจุบัน" มากกว่าเนื้อหาแรกๆ เมื่อวิเคราะห์ไฟล์ที่ 200 โมเดลอาจ "จำ" Pattern ที่เห็นในไฟล์ที่ 1 ได้ไม่แม่นยำอีกต่อไป ทำให้การเชื่อมโยงข้อมูลข้ามไฟล์มีโอกาสผิดพลาดสูงมาก Dynamic Workflows แก้ปัญหานี้ที่ระดับสถาปัตยกรรม ไม่ใช่ด้วยการยืด Context ให้ใหญ่ขึ้น

★ สาระสำคัญ: เมื่อต้องวิเคราะห์ไฟล์ 100+ ชิ้นที่เป็นอิสระต่อกัน ส่ง Sub-agent แยกกันไปแทนการอ่านเรียงลำดับ — ได้ทั้งความแม่นยำและความเร็ว

กรณีที่ 2: Migration โค้ดระดับ 750,000 บรรทัด (Bun)

โปรเจกต์ Bun ต้องการแปลงโค้ดจากภาษา Zig ไปเป็น Rust จำนวน 750,000 บรรทัด งานนี้ถ้าทำด้วยมนุษย์อย่างเดียวอาจใช้เวลาหลายเดือน แต่ด้วย Dynamic Workflows ใช้เวลาตั้งแต่ Commit แรกจนถึงการ Merge เพียง 11 วัน และโค้ดที่ได้ผ่านชุดการทดสอบ (Test suite) ถึง 99.8% — ตัวเลขนี้เป็นไปได้เพราะ Adversarial Verification: Sub-agents คู่โต้แย้งเข้าไปตรวจสอบโค้ดที่แปลงแล้วในแต่ละไฟล์ก่อนที่จะยอมรับผลลัพธ์

เจาะลึก — Adversarial Verification ในงาน Migration ทำงานอย่างไร?

ในกรณีนี้ Orchestrator ออกแบบ Workflow แบบ 3 Phase:

  1. Translator agents แปลง Zig → Rust ทีละไฟล์แบบขนาน
  2. Verifier agents รับโค้ดต้นทาง + โค้ดปลายทาง แล้วพยายามหาความแตกต่างทาง Semantic ที่ทำให้พฤติกรรมเปลี่ยน
  3. ถ้า Verifier พบปัญหา → ส่ง Translator กลับไปแก้ไข → Verify อีกครั้ง วนซ้ำจนผ่าน

กระบวนการทั้งหมดนี้เกิดขึ้น "นอก" Context window ของผู้ใช้ คุณเห็นเพียง Progress bar และสรุปผลสุดท้าย

Use Cases — Research เชิงลึกและการประเมินสเกลใหญ่

นอกจาก Codebase analysis Dynamic Workflows ยังแก้ปัญหาในงานวิจัยและประเมินผลที่ต้องรวบรวมข้อมูลจำนวนมากพร้อมกัน งานเหล่านี้มีลักษณะร่วมกันคือ "ต้องดูหลายอย่างพร้อมกัน แต่แต่ละอย่างเป็นอิสระ" — ซึ่งเป็น Sweet spot ของ Dynamic Workflows พอดี

กรณีที่ 3: Deep Research (/deep-research command)

ฟีเจอร์ /deep-research ใน Claude Code ทำงานบน Dynamic Workflows โดยตรง เมื่อคุณถามคำถามที่ซับซ้อน มันจะส่ง Sub-agents หลายตัวออกไปค้นหาข้อมูลจากเว็บไซต์ในหลากหลายมุมมองพร้อมกัน เอเจนต์บางกลุ่มทำหน้าที่ตรวจสอบความน่าเชื่อถือของแหล่งข้อมูล เอเจนต์บางกลุ่มหาข้อมูลโต้แย้ง ก่อนที่ Orchestrator จะสังเคราะห์รายงานที่มีการอ้างอิงแหล่งที่มาอย่างถูกต้อง — กระบวนการที่ถ้าทำแบบ Sequential อาจใช้เวลาหลายชั่วโมง ลดเหลือไม่กี่นาที

กรณีที่ 4: ประเมิน 41 ทักษะพร้อมกัน (41 Parallel Skill Audits)

ในกรณีศึกษาที่โดดเด่นจากวิดีโอ มีการส่ง Sub-agents 41 ตัวออกไปพร้อมกัน แต่ละตัวรับผิดชอบประเมิน 1 ทักษะ โดยแต่ละตัวต้องอ่านโค้ด ตรวจสอบเงื่อนไข และให้คะแนน — รวม Input tokens ที่ใช้ในการรันครั้งนั้นสูงถึง 5 ล้าน Tokens นี่คือตัวอย่างสำคัญที่แสดงทั้งพลังและต้นทุนของ Dynamic Workflows ในทางปฏิบัติ

เจาะลึก — ทำไม 41 ตัวถึงใช้ 5 ล้าน Tokens?

เมื่อ Sub-agent แต่ละตัวถูกสร้างขึ้น มันต้องโหลด Context ของตัวเองทั้งหมดใหม่ตั้งแต่ต้น รวมถึง:

  • System prompt และคำสั่งจาก Orchestrator
  • โค้ดหรือไฟล์ที่ต้องตรวจสอบ
  • Context พื้นฐานของโปรเจกต์

สมมติว่าแต่ละ Sub-agent ใช้ประมาณ 100,000 Tokens สำหรับการทำงานของตัวเอง × 41 ตัว = 4.1 ล้าน Tokens บวกกับ Orchestrator อีก ~900,000 Tokens = ใกล้เคียง 5 ล้าน Tokens พอดี

🔬

Security Audit ทั้งระบบ

ส่ง Sub-agents ไปตรวจสอบช่องโหว่ความปลอดภัยในทุก Service พร้อมกัน แต่ละตัวรายงานผลกลับมา แล้วสังเคราะห์เป็นรายงานความเสี่ยงรวม

📊

วิเคราะห์ Review 200 รายการ

แต่ละ Sub-agent อ่านและวิเคราะห์ 1 Review ออก Sentiment + Category ก่อนส่งกลับให้ Orchestrator รวบรวมแนวโน้มภาพรวม

🏗️

Plan Stress-testing

ส่งเอเจนต์หลายกลุ่มไปวิเคราะห์สถาปัตยกรรมจากมุมมองต่างกัน กลุ่มหนึ่งสนับสนุน อีกกลุ่มโต้แย้ง ได้แผนงานที่ผ่านการกลั่นกรองมาอย่างแข็งแกร่ง

★ สาระสำคัญ: กฎเหล็ก — งานที่เหมาะต้อง "แบ่งชิ้นย่อยอิสระได้" และ "สเกลใหญ่เกินกว่าเอเจนต์ตัวเดียวจะจำทุกอย่างได้" ถ้าทั้งสองเงื่อนไขไม่ครบ ให้ใช้เครื่องมืออื่นแทน

Token Cost — ความจริงที่ต้องรู้ก่อนกดรัน

Dynamic Workflows คือหนึ่งในฟีเจอร์ที่แพงที่สุดใน Claude Code มันไม่ได้ "ประหยัด" Token จากการแบ่งงาน แต่ ทวีคูณ (Multiply) การใช้ Token ขึ้นอย่างมหาศาล ความเข้าใจผิดนี้สามารถทำให้โควตาเดือนละ $200 หายไปครึ่งหนึ่งภายใน 30 นาที จากการรันงานเดียว

ทำไมถึงแพง? — Context Window Reloading

สาเหตุหลักคือ Sub-agent ทุกตัวที่ถูกสร้างขึ้นต้องโหลด Context ของตัวเองใหม่ตั้งแต่ต้น ตั้งแต่คำสั่งเริ่มต้น รายการงาน ไปจนถึงไฟล์ที่เกี่ยวข้อง ต้นทุนนี้บวกรวมกันจากทุก Sub-agent ไม่ใช่หาร

สูตรประมาณต้นทุน

ต้นทุนรวม ≈
  Orchestrator: 2,000–5,000 tokens
  + (จำนวน Sub-agents
     × 1,000–10,000 tokens/ตัว)

ตัวอย่าง:
  100 Sub-agents × 5,000 tokens
  = 500,000 tokens ขั้นต่ำ
  (ยังไม่รวมงานจริงของแต่ละตัว)

ตัวเลขจริงจากกรณีศึกษา

  • 41 Sub-agents ประเมิน 41 ทักษะ = 5 ล้าน Input Tokens
  • การกวาด Local files ทั้งโปรเจกต์ 30 นาที = เผา ครึ่งหนึ่งของโควตา $200/เดือน
  • Bun Migration 750k บรรทัด = หลายสิบล้าน Tokens รวม

3 กลยุทธ์ลดต้นทุน

🎯

1. Model Tiering

ใช้ Claude Haiku (หรือ Sonnet) สำหรับ Sub-agents ที่ทำงานซ้ำๆ เก็บ Opus ไว้เฉพาะ Orchestrator วิธีนี้ลดต้นทุนได้ 60-80%

🔬

2. Scope First

อย่าส่ง 300 ไฟล์ทีแรก ทดสอบด้วย 20 ไฟล์ก่อน เพื่อดูว่า Orchestrator แตกงานอย่างไร ใช้กี่ตัว และ Token เท่าไหร่ แล้วค่อยขยาย

📦

3. Context Pruning

ออกแบบ Prompt ให้แต่ละ Sub-agent ได้รับเฉพาะ Context ที่จำเป็น ไม่ใช่ทั้งโปรเจกต์ แต่ละ KB ที่ลดได้ × จำนวน Sub-agents = ประหยัดนับล้าน Tokens

เจาะลึก — เมื่อไหร่ Dynamic Workflows ถึงไม่ "คุ้ม" แม้งานจะใช้ได้?

แม้งานจะแยกส่วนได้ แต่ถ้า จำนวน Sub-agents น้อยเกินไป (เช่น 3-5 ตัว) ค่า Overhead ของ Orchestration (เขียนสคริปต์, ประสานงาน, สังเคราะห์) อาจมากกว่าประโยชน์ที่ได้จากการขนาน ให้ใช้ Dynamic Workflows เมื่องานต้องการ Sub-agents อย่างน้อย 10+ ตัว ไม่อย่างนั้น Sub-agent เดี่ยวหรือ Agent Teams จะประหยัดกว่า

★ สาระสำคัญ: คำนวณก่อนรัน: N Sub-agents × Est. tokens/ตัว = งบที่ต้องมี; Model Tiering (Haiku sub-agents + Opus orchestrator) คือ lever ที่ทรงพลังที่สุดในการลดต้นทุน 60-80%

คู่มือตัดสินใจ — ใช้ หรือ ไม่ใช้?

Dynamic Workflows เป็นเครื่องมือที่ทรงพลังแต่มีต้นทุนสูง การตัดสินใจว่าควรใช้หรือไม่ขึ้นอยู่กับลักษณะของงานเป็นหลัก ไม่ใช่ขนาดของงาน งานใหญ่แต่ Sequential ควรใช้เครื่องมืออื่น ส่วนงานเล็กแต่ขนานได้อาจยังไม่คุ้มค่าที่จะใช้

ใช้ Dynamic Workflows เมื่อ ✓

งานแยกส่วนได้ + สเกลใหญ่

ตรวจสอบ 100+ ไฟล์, แปลง 500+ ไฟล์, วิเคราะห์ 200+ รายการ — แต่ละชิ้นเป็นอิสระต่อกัน

ยังไม่รู้บทบาทล่วงหน้า

ไม่รู้ว่าต้องใช้ทักษะอะไรบ้าง ให้ Orchestrator ออกแบบ Sub-agent structure เองแบบไดนามิก

ต้องการ Adversarial QA

งาน Migration หรือ Audit ที่ต้องการคุณภาพสูง เอเจนต์โต้แย้งกันจนได้ผลลัพธ์ที่ผ่านการกลั่นกรองแล้ว

Context Pollution เป็นปัญหา

งานที่เอเจนต์ตัวเดียวจะ "ลืม" ข้อมูลแรกๆ เมื่อต้องดูไฟล์จำนวนมาก

อย่าใช้ Dynamic Workflows เมื่อ ✗

งาน Sequential เคร่งครัด

ขั้นตอน 2 ต้องรอขั้นตอน 1 เสมอ — การขนานไม่มีประโยชน์ เสีย Token ค่า Overhead ฟรีๆ

งานเล็ก จบได้ใน Pass เดียว

งานที่เอเจนต์ตัวเดียวทำได้ใน 2-3 นาที ค่า Orchestration overhead ไม่คุ้มเลย

แก้ไฟล์ร่วมกัน (Shared State)

ถ้า Sub-agents หลายตัวต้องเขียนไฟล์เดียวกัน จะเกิด Conflict — ต้องออกแบบให้แต่ละตัวมีไฟล์เป็นของตัวเอง

รู้บทบาทล่วงหน้าชัดเจน

ถ้ารู้แน่ว่าต้องการ Frontend + Backend + QA ใช้ Agent Teams แทน — ประหยัดกว่าและควบคุมได้ดีกว่า

เจาะลึก — Flow Chart ตัดสินใจแบบ Step-by-step
  1. งานแบ่งชิ้นย่อยอิสระได้ไหม? — ไม่ → ใช้ /goal หรือ Single agent
  2. ชิ้นย่อยมีกี่ชิ้น? — น้อยกว่า 10 → ใช้ Sub-agent เดี่ยวหรือ Agent Teams
  3. รู้บทบาทล่วงหน้าไหม? — รู้ → ใช้ Agent Teams; ไม่รู้ → Dynamic Workflows
  4. งบประมาณ Token เพียงพอไหม? — คำนวณ N × per-agent cost ก่อนรัน
  5. ถ้าทุกข้อผ่าน → Dynamic Workflows + Scope test (20 ชิ้นก่อน)
★ สาระสำคัญ: ถ้างานทำตามลำดับหรือเล็กเกินไป ฟีเจอร์อื่นจะถูกกว่าและเร็วกว่า Dynamic Workflows — ใหญ่ + แยกส่วนได้ คือ Sweet spot เดียวที่คุ้มค่า

สรุปสาระสำคัญ — 7 ข้อที่ต้องจำ

Dynamic Workflows เปลี่ยนวิธีที่ AI จัดการงานสเกลใหญ่ แต่พลังนี้มาพร้อมต้นทุนที่ต้องเข้าใจ 7 ข้อด้านล่างนี้สรุปทุกสิ่งที่จำเป็นต้องรู้ตั้งแต่แนวคิดพื้นฐานจนถึงกลยุทธ์ปฏิบัติ

1
WIDTH vs DEPTH — เลือกให้ถูก

Dynamic Workflows = WIDTH play (ทำหลายอย่างพร้อมกัน); /goal = DEPTH play (ลองซ้ำจนได้คำตอบ) เลือกจากว่า Bottleneck คืออะไร

2
Parallel ≠ ถูกกว่า — มันแพงกว่าเสมอ

แต่ละ Sub-agent โหลด Context ใหม่ทั้งหมด ต้นทุนรวม = N × per-agent cost ไม่ใช่ cost รวมหาร N เสมอประมาณก่อนรัน

3
Adversarial Verification คือกลไกคุณภาพ

เอเจนต์โต้แย้งกันจนได้คำตอบที่ Converge — เหตุผลที่ Bun ได้ 99.8% pass rate บน 750k บรรทัด

4
Independent + Data-isolated คือเงื่อนไขสำคัญ

งานต้องแยกชิ้นได้โดยที่แต่ละชิ้นไม่ต้องรอผลจากชิ้นอื่น และไม่แก้ไฟล์ร่วมกัน

5
Model Tiering = lever ลดต้นทุน 60-80%

Haiku (หรือ Sonnet) สำหรับ Sub-agents + Opus สำหรับ Orchestrator เท่านั้น

6
ทดสอบ 20 ชิ้นก่อน ไม่ใช่ 300

Small-scope test ช่วยให้เห็นว่า Orchestrator แตกงานอย่างไรและใช้ Token เท่าไหร่ก่อนลงทุนเต็มระบบ

7
รู้บทบาทแล้ว? → Agent Teams; ยังไม่รู้? → Dynamic Workflows

Agent Teams ประหยัดและควบคุมได้ดีกว่าเมื่อโครงสร้างทีมชัดเจน Dynamic Workflows เก่งในการออกแบบ Workflow เองเมื่อยังไม่รู้
