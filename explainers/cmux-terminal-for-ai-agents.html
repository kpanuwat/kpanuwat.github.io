Overview

cmux คืออะไร และทำไมนักพัฒนา AI ถึงต้องสนใจ

cmux (อ่านว่า "see-mux") คือ terminal app บน macOS ที่ถูกสร้างขึ้นมาตั้งแต่ต้นเพื่อ AI coding agents โดยเฉพาะ ไม่ใช่ terminal ทั่วไปที่ถูก patch ให้รองรับ AI ในภายหลัง ความแตกต่างหลักคือ cmux ให้ agent ควบคุม UI ของตัวเอง — เปิด pane ใหม่ สั่ง browser ทำงาน รับ-ส่ง notification — ทั้งหมดผ่าน shell commands ธรรมดา ผลคือนักพัฒนาสามารถดูแล AI agent หลายตัวพร้อมกันได้โดยไม่จมอยู่กับ terminal ที่รก และไม่พลาดเมื่อ agent ต้องการความสนใจ

ความเร็วระดับ Native

Swift + libghostty

เขียนด้วย Swift + AppKit ใช้ libghostty ของ Ghostty เป็น rendering engine ทำให้เร็วและเบากว่า Electron หลายเท่า ไม่มี web layer ไม่มี overhead พิเศษ

Agent Control API

cmux CLI + Unix Socket

agent สั่ง cmux new-split, cmux browser snapshot, หรือ cmux notify ได้เหมือน shell command ธรรมดา GUI ตอบสนองทันทีผ่าน Unix domain socket

Multi-Agent Visibility

Notification Rings

เมื่อ agent ทำงานเสร็จหรือต้องการ input วงแหวนแจ้งเตือนปรากฏรอบ pane นั้น ไม่ต้องจ้องหน้าจอตลอด ทำงานอื่นได้ระหว่างรอ

💡 Core insight: cmux เปลี่ยน terminal จาก "ที่รัน command" เป็น "workspace ที่ AI agent ควบคุมตัวเองได้" — ความแตกต่างนี้เล็กในรูปลักษณ์ แต่ใหญ่มากในทางปฏิบัติ
★ สาระสำคัญ: cmux = terminal ที่ AI agent ควบคุม UI ของตัวเองได้ผ่าน shell commands — ไม่ใช่แค่รัน agent

Section 1 — Deep-Dive Protocol

ส่วนประกอบหลัก สมมติฐานที่ซ่อนอยู่ และสิ่งที่ต้องเข้าใจ

ก่อนลงลึก ส่วนนี้สร้าง mental model ที่ถูกต้องตั้งแต่ต้น: cmux ประกอบด้วยชั้นสถาปัตยกรรม 6 ชั้นที่ทำงานร่วมกัน มีสมมติฐานที่ซ่อนอยู่หลายอย่างที่ผู้เริ่มต้น มักข้ามไป และมี 7 insight ที่ถ้าเข้าใจตั้งแต่ต้นจะประหยัดเวลาไปได้มาก

ส่วนประกอบหลัก (Architecture Components)

Rendering Layer

libghostty

Ghostty's terminal rendering library — GPU-accelerated, เร็วเท่า native app cmux ใช้เป็น library ไม่ใช่ fork ดังนั้นจึงยังอ่าน Ghostty config ได้

Layout Engine

Bonsplit

จัดการ Workspace → Surface (tab) → Pane (split) hierarchy ทำ animation smooth และ drag-and-drop reorder ได้ agent ใช้ surface ID เพื่อ target pane ที่ถูกต้อง

IPC Bus

Unix Domain Socket + CLI

agent รัน cmux command → CLI ส่ง JSON → socket → GUI อัพเดททันที เร็วกว่า HTTP เพราะเป็น local IPC ไม่ผ่านเครือข่าย

Feedback System

Notification Rings

ทำงานจาก OSC 9/99/777 escape sequences (auto) หรือ cmux notify (manual) แสดง visual ring รอบ pane + badge + macOS notification พร้อมกัน

Browser Engine

WebKit Pane

Safari-grade browser ฝังใน pane — cmux browser snapshot คืน DOM tree แบบ text-based พร้อม element refs (e10, e14) ให้ agent สั่ง click/fill

Orchestration

Sub-Agent Layer

Primary agent ใช้ cmux send + cmux new-split spawn sub-agents ใน pane แยก assign งาน อ่าน output แล้วปิด pane เมื่อเสร็จ

ขั้นตอนการสื่อสาร: AI Agent → cmux CLI → Unix Domain Socket (IPC) → cmux GUI อัพเดท UI → Notification Ring แจ้งผู้ใช้

สมมติฐานที่ซ่อนอยู่ (Implicit Assumptions)

สิ่งเหล่านี้คือสิ่งที่ documentation ถือว่าผู้อ่านรู้แล้ว แต่ผู้เริ่มต้นมักข้ามไป การเข้าใจสมมติฐานเหล่านี้ตั้งแต่ต้นจะหลีกเลี่ยงความสับสนได้มาก

  • Agent ไม่ใช้ cmux API อัตโนมัติ: การที่ agent รันใน cmux ไม่ได้หมายความว่ามันจะใช้ cmux notify หรือ cmux browser เอง คุณต้องบอก agent ว่ามีเครื่องมือเหล่านี้อยู่ ผ่าน system prompt หรือ Claude Code hooks
  • cmux ≠ Ghostty fork: cmux เป็น Swift app แยกต่างหากที่ใช้ libghostty เป็น rendering library เท่านั้น ไม่ใช่ branch หรือ plugin ของ Ghostty — อ่าน Ghostty config ได้แต่ feature อื่นๆ เป็นของ cmux เอง
  • Unix socket = local only: agent API ของ cmux ทำงานผ่าน Unix domain socket ซึ่งเป็น local IPC เท่านั้น agent ต้องรันบนเครื่อง Mac เดียวกัน ไม่มี remote agent control
  • macOS only โดย design: ไม่ใช่ limitation ชั่วคราว — Swift + AppKit เลือกโดยเจตนาเพื่อ performance และ macOS integration ไม่มีแผน port ไป Linux/Windows ใน near-term

สิ่งสำคัญที่ต้องรู้ก่อนลงลึก (Key Takeaways)

Takeaway 1 — Big Picture

cmux เปลี่ยน terminal เป็น controllable workspace

ความแตกต่างจาก terminal ทั่วไปไม่ใช่แค่ feature เพิ่มขึ้น แต่เป็น paradigm shift — agent ไม่ใช่แค่รัน command แต่ควบคุม workspace ของตัวเองได้

Takeaway 2 — Mental Model

Workspace → Surface → Pane คือ 3 ชั้นที่ต้องรู้

Workspace = project หรือ context หนึ่งอัน, Surface = tab ภายใน workspace, Pane = split ภายใน surface ทุก cmux command ต้องระบุ target surface ID

Takeaway 3 — Common Trap

อย่าสับสน cmux กับ tmux

ชื่อคล้ายกันแต่ต่างกันสิ้นเชิง tmux = multiplexer ที่รันใน terminal ใดก็ได้, cmux = native macOS app ที่มี GUI, browser, และ socket API ในตัว

Takeaway 4 — Actionable

browser snapshot คือ superpower ที่คนมองข้าม

แทนที่จะ guess CSS selector, cmux browser snapshot คืน DOM refs เช่น e10, e14 ที่ agent ใช้ click/fill ได้ทันที ทำให้ web automation เชื่อถือได้มากขึ้น

💡 Takeaways 5–7: 5) Notification rings ทำงานอัตโนมัติถ้า terminal emit OSC sequences — Claude Code ทำได้ด้วย hooks 6) Sub-agent orchestration ไม่ต้องเขียน code — ใช้ cmux send + new-split ล้วนๆ 7) ฟรีและ open source (AGPL) ไม่มี subscription — ดาวน์โหลดจาก cmux.com ได้ทันที
★ Mental model: cmux = Swift terminal ที่ให้ AI agent ควบคุม UI ผ่าน socket API — ทุก feature รอบ concept นี้

Section 2 — Deep Dive

สถาปัตยกรรมและ Tech Stack: ทำไมถึงเร็วและ native จริงๆ

หนึ่งในคำถามแรกที่นักพัฒนาถามเมื่อเห็น cmux คือ "มันต่างจาก terminal ทั่วไปยังไง?" คำตอบอยู่ที่ การเลือก tech stack อย่างพิถีพิถัน แทนที่จะสร้างบน Electron (ซึ่งง่ายกว่าและ cross-platform) ทีม Manaflow AI เลือกใช้ Swift + AppKit เป็นฐาน ผลคือ terminal ที่ lightweight, responsive, และรวมเข้ากับ macOS ได้อย่างลึกซึ้ง — ทั้งระบบ notification, keybindings, และ accessibility

libghostty — Rendering Engine ที่ยืมมาจาก Ghostty

Ghostty เป็น terminal emulator ที่รู้จักกันดีในเรื่องความเร็วและ GPU-accelerated rendering cmux ไม่ได้ fork Ghostty แต่ใช้ libghostty ซึ่งเป็น core library ของ Ghostty เป็น rendering engine — เหมือนกับที่ browser อื่นๆ ใช้ WebKit โดยไม่ต้อง fork Safari ความสัมพันธ์นี้สำคัญมากเพราะ cmux ได้รับ GPU-accelerated rendering โดยไม่ต้องเขียน terminal emulator เองตั้งแต่ต้น และยังอ่าน config จาก ~/.config/ghostty/config ได้ด้วย ทำให้ผู้ที่ใช้ Ghostty อยู่แล้วย้ายมา cmux ได้โดยไม่ต้อง reconfigure keybindings ใหม่ทั้งหมด

ในทางปฏิบัติ libghostty ทำให้ cmux render text และ graphics ได้ smooth แม้ตอนที่มี หลาย pane รันพร้อมกัน ใน Electron-based terminal การมี 6-8 agent instances พร้อมกัน มักทำให้ RAM สูงขึ้น 500MB-1GB+ ใน cmux ตัวเลขนี้ต่ำกว่ามากเพราะไม่มี Chromium instance

เจาะลึก libghostty — ทำงานอย่างไรภายใน?

libghostty ใช้ Metal (macOS graphics API) สำหรับ GPU rendering ทำให้ text rendering ไม่ block main thread เหมือน CPU-based terminal ผลคือ scrollback buffer ขนาดใหญ่ หรือ output ที่หนักๆ จาก AI agent ไม่ทำให้ UI หยุดตอบสนอง

cmux อ่าน Ghostty config format ได้ตรงๆ เช่น font settings, color scheme, และ keybindings สำหรับ terminal operations cmux-specific features (workspaces, notifications) มี config แยกใน Settings ของ app

Bonsplit — Layout Engine สำหรับ Multi-Pane Workspace

การจัดการ layout ของ cmux ใช้ library ชื่อ Bonsplit ซึ่งรับผิดชอบ hierarchy ทั้งหมด: Workspace (กลุ่ม surfaces), Surface (tab หนึ่งอัน), และ Pane (split ภายใน surface) Bonsplit ทำให้การ drag-and-drop panes, split แนวตั้งและแนวนอน, และการ animate layout changes ทำงานได้ smooth ซึ่งสำคัญมากเมื่อ agent กำลัง spawn panes ใหม่หลายอันในเวลาสั้นๆ

ทุก pane ใน Bonsplit มี unique surface ID (เช่น surface:7, surface:8) ซึ่ง agent ใช้เพื่อ target command ไปยัง pane ที่ถูกต้อง ถ้า agent ต้องการส่ง text ไปยัง sub-agent ใน surface 8 ก็แค่รัน cmux send --surface surface:8 "task here" ไม่ต้องทำ screen scraping หรือ grep output

WebKit — Browser Engine ใน Terminal Pane

cmux ฝัง WebKit (Safari's browser engine) เป็น pane type พิเศษ เมื่อ agent รัน cmux new-pane --type browser --url https://example.com จะได้ full browser ที่รองรับ DevTools, JavaScript, และ form interactions ทุกอย่าง ไม่ใช่แค่ text renderer

สิ่งที่ทำให้ browser pane ทรงพลังจริงๆ คือ cmux browser snapshot command ซึ่งคืน simplified DOM tree ในรูป text พร้อม short references เช่น e10 สำหรับ input fields, e14 สำหรับ buttons agent ใช้ references เหล่านี้แทน XPath หรือ CSS selectors ที่ขึ้นอยู่กับ HTML structure ที่เปลี่ยนได้ ทำให้ web automation เชื่อถือได้มากขึ้น

ชั้น Tech Stack ของ cmux: ผู้ใช้และ agent อยู่บนสุด, Swift app เป็นกลาง, libghostty + WebKit + macOS APIs เป็นฐาน

★ สาระสำคัญ: cmux เร็วเพราะ native Swift ไม่ใช่ Electron — libghostty render GPU, WebKit ทำ browser, Bonsplit จัดการ panes

Section 3 — Deep Dive

Workspace, Surface, Pane: ระบบจัดการหน้าต่างที่ออกแบบมาสำหรับ Multi-Agent

เมื่อทำงานกับ AI agents หลายตัวพร้อมกัน สิ่งที่สร้างความสับสนมากที่สุดคือการจัดการ context agent ตัวไหนกำลังทำงานอะไร อยู่ที่ directory ไหน มี error ไหม? cmux แก้ปัญหานี้ด้วย hierarchy 3 ชั้นที่ให้แต่ละ agent มี "บ้าน" ของตัวเอง และ sidebar ที่แสดง context สำคัญ ของแต่ละ workspace ได้ทันทีโดยไม่ต้องสลับไปดู

Hierarchy 3 ชั้น: Workspace → Surface → Pane

ใน cmux ทุกอย่างอยู่ใน hierarchy ที่ชัดเจน Workspace คือกลุ่มใหญ่สุด เปรียบเหมือน "project" หนึ่งอัน เช่น workspace "frontend" กับ "backend" ใน sidebar แต่ละ workspace มี icon วงกลมที่แสดงสถานะ notification และ metadata ของ workspace นั้น

ภายใน workspace แต่ละอัน จะมี Surface ซึ่งเปรียบเหมือน tab ใน browser แต่ละ surface รัน session แยกกันอิสระ เก็บ working directory, environment variables, และ history ของตัวเอง เมื่อ agent สร้าง surface ใหม่ด้วย cmux new-pane มันจะได้ surface ID ที่ unique เช่น surface:7 ซึ่ง agent อื่นหรือ primary agent ใช้ส่ง command มาได้โดยตรง

ใน surface หนึ่งอัน สามารถ split เป็น Panes ได้ทั้งแนวตั้งและแนวนอน pane แต่ละอันรัน terminal session หรือ browser session แยกกัน ทำให้เห็น agent กำลัง run tests ใน pane หนึ่ง ในขณะที่ pane ข้างๆ แสดง browser ที่กำลัง verify output

ตัวอย่าง Workspace เดียวที่มี 2 Surfaces แต่ละ Surface มี 2 Panes — รองรับ agent 3+ ตัวทำงานพร้อมกัน

Vertical Sidebar — Context ที่มองเห็นได้โดยไม่ต้อง switch

sidebar แนวตั้งทางซ้ายของ cmux ไม่ใช่แค่รายชื่อ tab — มันแสดง context สำคัญของแต่ละ workspace: git branch ปัจจุบัน, สถานะ PR ที่เชื่อมอยู่, listening ports ที่ process กำลังใช้, และ latest notification text จาก agent ใน workspace นั้น ทำให้นักพัฒนาสามารถ monitor หลาย agent พร้อมกันโดยมองที่ sidebar เดียว โดยไม่ต้องสลับ context

เมื่อ agent ใน workspace ใดส่ง notification มา icon วงกลมของ workspace นั้นจะแสดง badge และถ้า agent emit OSC 9 escape sequence หรือเรียก cmux notify "task complete" ระบบจะแสดง notification ring สว่างรอบ pane นั้นและส่ง macOS notification ด้วย ทำให้ทำงานอื่นในขณะที่รอ agent ทำงานได้โดยมั่นใจว่าจะไม่พลาดเมื่อมีอะไรสำคัญ

เจาะลึก cmux CLI commands สำหรับ workspace management

คำสั่งหลักที่ agent ใช้ในการจัดการ workspace:

# ดู workspaces ทั้งหมด
cmux list-workspaces

# สร้าง workspace ใหม่
cmux new-workspace --name "my-project"

# split surface แนวนอน (side-by-side)
cmux new-split --direction right

# สร้าง pane ใหม่ใน surface ใหม่ (tab)
cmux new-pane

# ปิด surface ที่ระบุ
cmux close-surface --surface surface:7

# อ่าน screen output จาก surface ที่ระบุ
cmux read-screen --surface surface:8

command new-split กับ new-pane ต่างกัน: new-split แยก surface ปัจจุบันออกเป็น 2 ส่วน ส่วน new-pane สร้าง surface ใหม่แยกต่างหาก ทั้งสองคืน surface ID ที่ agent ใช้ target ใน command ถัดไป

★ สาระสำคัญ: Workspace→Surface→Pane hierarchy ทำให้ agent หลายตัวมี context ของตัวเอง sidebar แสดงสถานะทุกตัวโดยไม่ต้อง switch

Section 4 — Hands-on Guide

cmux CLI: ภาษาที่ Agent ใช้สื่อสารกับ Terminal

cmux CLI คือชุด shell commands ที่ agent รันได้เหมือน command ทั่วไป แต่ผลลัพธ์คือการเปลี่ยนแปลง UI ของ cmux ทันที ไม่ใช่แค่พิมพ์ text ลง terminal ก่อนใช้งาน agent ต้องรู้ว่ามีเครื่องมือเหล่านี้อยู่ ผ่าน system prompt หรือ Claude Code hooks เมื่อ agent รู้แล้ว มันจะใช้ cmux เป็น "arm" ที่ควบคุม environment ของตัวเองได้อย่างอิสระ

ติดตั้งและเริ่มต้นใช้งาน

cmux เป็น native macOS app ติดตั้งโดยดาวน์โหลด .dmg จาก cmux.com ลาก app ไปยัง Applications folder และเปิดใช้งาน เมื่อเปิดครั้งแรก cmux จะติดตั้ง cmux CLI ใน PATH โดยอัตโนมัติ ทำให้ทุก terminal session ที่รันใน cmux เรียกใช้ cmux --help ได้ทันที ไม่ต้องติดตั้งแยก

สิ่งสำคัญที่ต้องรู้: cmux CLI ใช้ได้ภายใน cmux เท่านั้น เพราะมันสื่อสารผ่าน Unix domain socket ที่ cmux app สร้างขึ้น ถ้ารัน cmux command ใน terminal อื่น (iTerm2, Terminal.app) จะได้รับ error เพราะไม่มี socket รอรับ

Commands ที่ใช้บ่อยในการทำงานกับ Agent

commands ของ cmux แบ่งเป็น 5 กลุ่มหลัก: workspace management, pane/surface management, interaction (ส่งข้อความไปยัง pane อื่น), browser control, และ feedback ด้านล่างคือ commands ที่ agent ใช้บ่อยที่สุดในการทำงานจริง

Workspace & Pane Management

# สร้าง pane ใหม่แนวนอน
cmux new-split --direction right

# สร้าง browser pane
cmux new-pane --type browser \
  --url https://localhost:3000

# ปิด pane ที่เสร็จแล้ว
cmux close-surface --surface surface:9

# highlight pane เพื่อดึงความสนใจ
cmux trigger-flash --surface surface:7

Interaction & Feedback

# ส่งข้อความไปยัง pane อื่น (sub-agent)
cmux send --surface surface:8 \
  "Analyze this module for bugs"

# กด Enter เพื่อ submit command
cmux send-key --surface surface:8 Return

# อ่าน output จาก pane
cmux read-screen --surface surface:8

# ส่ง notification + badge
cmux notify "Analysis complete — 3 issues found"

คำสั่ง cmux send ส่งข้อความไปยัง stdin ของ pane ที่ระบุ ซึ่งเหมือนกับการพิมพ์ใน terminal นั้น แต่ทำโดย programmatically ส่วน cmux send-key Return simulate การกด Enter ทั้งสองต้องคู่กันเสมอเมื่อต้องการ execute command ใน sub-agent

ตัวอย่างจริง: Agent ค้นหาข้อมูลและสรุปผล

ตัวอย่างด้านล่างแสดง workflow ที่ primary agent ใช้ cmux เปิด browser ค้นหาข้อมูล จาก web แล้วนำผลมาประมวลผลต่อ ทั้งหมดรันใน cmux workspace เดียว ไม่ต้อง สลับออกไปทำงานในแอปอื่น 

# 1. เปิด browser pane ไปยัง URL ที่ต้องการ
cmux new-pane --type browser --url https://docs.example.com

# 2. รอให้ page โหลดเสร็จ
cmux browser wait --event loaded

# 3. ดู DOM structure เพื่อหา element refs
cmux browser snapshot
# ได้ output เช่น: e5=search-input, e12=submit-btn

# 4. ค้นหาด้วย element refs ที่ได้จาก snapshot
cmux browser type --ref e5 --text "AI agents deployment"
cmux browser click --ref e12

# 5. รอ navigation เสร็จ แล้ว snapshot อีกรอบ
cmux browser wait --event loaded
cmux browser snapshot

# 6. แจ้งเมื่อเสร็จ
cmux notify "Web research complete — ready to summarize"

เหตุผลที่ใช้ element refs จาก snapshot แทน CSS selectors คือ refs เหล่านี้ ถูก assign โดย cmux ตาม DOM ปัจจุบัน ไม่ขึ้นอยู่กับ class names หรือ IDs ที่อาจเปลี่ยนได้ ทำให้ automation เสถียรกว่า ถ้า element ที่ต้องการหายไปจาก DOM, cmux จะแจ้ง error แทนที่จะ click ผิดที่โดยไม่รู้ตัว

★ สาระสำคัญ: cmux CLI คือ "arm" ของ agent — ใช้ new-split สร้างพื้นที่, send ส่งงาน, browser snapshot + click ทำ web automation, notify แจ้งเมื่อเสร็จ

Section 5 — Deep Dive

Multi-Agent Orchestration: Primary Agent ควบคุม Sub-Agents ได้อย่างไร

งานพัฒนา software จริงๆ มักซับซ้อนเกินกว่าที่ agent ตัวเดียวจะจัดการได้ดี cmux แก้ปัญหานี้ด้วยการให้ primary agent spawn sub-agents ใน pane แยก มอบหมาย งาน รออ่าน output แล้วรวมผล ทั้งหมดนี้ทำได้ด้วย shell commands ธรรมดา ไม่ต้องเขียน orchestration code หรือ framework พิเศษ

Pattern: Primary Agent สร้างและควบคุม Sub-Agents

ลองนึกภาพว่าต้องการ refactor codebase ขนาดใหญ่ primary agent อาจแบ่งงานดังนี้: sub-agent 1 รับผิดชอบอ่านและเข้าใจ architecture, sub-agent 2 วิเคราะห์ dependencies และ potential issues ส่วน primary agent รอ output จากทั้งสอง สรุปผล แล้วตัดสินใจว่าจะ refactor ส่วนไหนก่อน pattern นี้ทำงานได้โดยไม่ต้องมี orchestration framework — ใช้ cmux shell commands ล้วนๆ 

# Primary agent: สร้าง 2 sub-agent panes พร้อมกัน
# Split surface ปัจจุบันแนวนอน
cmux new-split --direction right   # ได้ surface:7

# Split surface:7 แนวตั้ง
cmux new-split --direction down --surface surface:7  # ได้ surface:8

# Launch sub-agents ใน pane ที่สร้าง
cmux send --surface surface:7 "claude"
cmux send-key --surface surface:7 Return

cmux send --surface surface:8 "claude"
cmux send-key --surface surface:8 Return

# มอบหมายงานแยกกัน
cmux send --surface surface:7 \
  "Read all files in ./src and describe the architecture"
cmux send-key --surface surface:7 Return

cmux send --surface surface:8 \
  "Analyze all imports in ./src and map dependencies"
cmux send-key --surface surface:8 Return

# รอ output (primary agent ทำงานอื่นได้ระหว่างนี้)
# เมื่อ sub-agents เสร็จ อ่าน output
cmux read-screen --surface surface:7
cmux read-screen --surface surface:8

# ปิด panes หลังเสร็จ
cmux close-surface --surface surface:7
cmux close-surface --surface surface:8

สิ่งสำคัญที่ต้องเข้าใจใน pattern นี้: primary agent ไม่ได้รอแบบ blocking — หลังส่ง task ให้ sub-agents แล้ว มันสามารถทำงานอื่นต่อได้ใน pane ของตัวเอง เมื่อต้องการผล ค่อย cmux read-screen อ่าน output นี่คือ parallel execution จริงๆ ไม่ใช่ sequential

เจาะลึก: sub-agent รู้ได้อย่างไรว่าตัวเองรันใน cmux?

cmux set environment variable CMUX_SURFACE ใน session ของทุก pane ทำให้ sub-agent สามารถตรวจสอบได้ว่ารันใน cmux หรือไม่ด้วย $CMUX_SURFACE และรู้ surface ID ของตัวเองเพื่อส่ง notification กลับมาได้

ยิ่งไปกว่านั้น agent สามารถ discover capabilities ทั้งหมดที่มีได้ด้วย cmux --help ซึ่ง Claude Code ถูก train ให้รู้ว่าควรเรียก cmux --help เมื่อรันใน cmux เพื่อ discover available commands — นี่คือ zero-config experience ที่ทีม cmux ตั้งใจออกแบบ

Notification System: รู้เมื่อ Agent เสร็จโดยไม่ต้องจ้องหน้าจอ

เมื่อ agent หลายตัวทำงานพร้อมกัน ปัญหาหลักคือจะรู้ได้อย่างไรว่า agent ตัวไหนเสร็จ หรือต้องการ input? cmux แก้ด้วย notification ring system ที่ทำงานได้ 2 วิธี:

วิธีที่ 1 — Automatic (OSC escape sequences): terminal emulator หลายอันรองรับการ emit OSC 9 (เช่น iTerm2 style) เมื่อ process เสร็จ Claude Code มี hook ที่ emit sequence เหล่านี้ได้โดยอัตโนมัติ ทำให้ cmux แสดง notification ring ทันทีที่ task เสร็จโดยไม่ต้อง modify agent code

วิธีที่ 2 — Manual (cmux notify): agent call cmux notify "message" ได้โดยตรง เหมาะสำหรับ custom checkpoints เช่น "phase 1 of analysis complete" หรือ "waiting for user input" notification จะแสดงใน ring รอบ pane นั้น + badge ใน sidebar + macOS desktop notification พร้อมกันทั้งหมด

Claude Code Hook ที่แนะนำ: เพิ่ม hook ใน Claude Code settings เพื่อให้ Claude Code emit cmux notify อัตโนมัติเมื่อ task หรือ subtask เสร็จ ทำให้ไม่ต้อง instruct Claude Code ทุกครั้ง — ดูตัวอย่าง hooks ที่ cmux docs
★ สาระสำคัญ: primary agent spawn sub-agents ด้วย cmux send + new-split, notification rings แจ้งเมื่อเสร็จ — parallel execution โดยไม่ต้องเขียน orchestration code

Section 6 — Comparison & FAQ

cmux เทียบกับ tmux, Ghostty, และ iTerm2: เลือกอะไรในสถานการณ์ไหน

เมื่อพูดถึง terminal บน macOS มีหลายตัวเลือกที่แต่ละคนคุ้นเคยต่างกัน คำถามที่ถามบ่อยที่สุดคือ "ต้องเปลี่ยนมาใช้ cmux ไหม?" คำตอบขึ้นอยู่กับ workflow ถ้าทำงานกับ AI agents เป็นหลัก cmux มีข้อได้เปรียบชัดเจน แต่ถ้า workflow ยังเป็น manual programming เป็นหลัก terminal ที่คุ้นเคยอยู่อาจดีพอแล้ว

cmux — Native macOS App สำหรับ AI Agent Era

cmux ออกแบบมาโดยเฉพาะสำหรับ AI agents บน macOS ข้อได้เปรียบหลักคือ socket API ที่ให้ agent ควบคุม UI ได้, browser pane ใน built-in, notification rings สำหรับ multi-agent monitoring, และ vertical sidebar ที่แสดง git/PR/port context ของแต่ละ workspace

ข้อจำกัดที่ต้องรู้ก่อนเลือก: macOS เท่านั้น ไม่มี Linux/Windows version agent ต้องได้รับการบอกว่า cmux API มีอยู่ ไม่ใช่ automatic ถ้าใช้ agent น้อยหรือใช้แบบ sequential (ทีละตัว) ข้อได้เปรียบของ cmux จะไม่ชัดเจนมากนัก

เหมาะสำหรับ: นักพัฒนาที่ใช้ Claude Code / Codex / Gemini CLI เป็นประจำ, ต้องการรัน agents หลายตัวพร้อมกัน, หรือต้องการ browser automation ใน workflow เดียวกับ terminal

tmux — Terminal Multiplexer ที่ใช้ได้ทุก Platform

tmux รันใน terminal ใดก็ได้และทำงานได้บน Linux, macOS, และ Windows (WSL) เป็นเครื่องมือที่นักพัฒนา backend และ DevOps รู้จักดี และมี ecosystem ของ plugins (tmux-plugins, tpm) ที่ใหญ่มาก ถ้าทำงานบน remote server หรือ SSH เป็นหลัก tmux ยังคงเป็นตัวเลือกที่ดีที่สุด

ข้อจำกัดสำหรับ AI agent workflow: tmux ไม่มี socket API ที่ agent ควบคุม GUI ได้, ไม่มี browser pane, และ notification ต้องทำผ่าน bell หรือ script แยก การ monitor หลาย agent พร้อมกันต้องอาศัย manual ดูมากกว่า

เหมาะสำหรับ: งานบน remote server / SSH, cross-platform workflow, หรือทีมที่ invest ใน tmux config มาแล้วมาก

Ghostty / iTerm2 — Excellent Terminals ที่ไม่ได้ออกแบบสำหรับ Agent

Ghostty และ iTerm2 เป็น terminals ที่ดีมากในฐานะ "terminal สำหรับมนุษย์" — Ghostty เร็วมาก, iTerm2 มี features เยอะและ mature มาก ทั้งสองทำงานได้ดีกับ AI agents ในฐานะ "ที่รัน" แต่ agents ไม่สามารถควบคุม UI ของ terminal เหล่านี้ได้ ไม่มี socket API สำหรับ agent

ถ้า workflow ยังเป็น manual เป็นหลัก และใช้ agent เป็นครั้งคราว Ghostty หรือ iTerm2 ก็ตอบโจทย์ได้ดี cmux จะ shine เมื่อ agent เป็น primary worker ไม่ใช่ assistant

เหมาะสำหรับ: นักพัฒนาที่ใช้ terminal แบบ manual เป็นหลัก, ต้องการ terminal ที่ stable และ mature, หรือยังไม่ได้ shift ไปสู่ AI-driven workflow เต็มตัว

ได้เลย — cmux เป็น terminal ดังนั้น agent ใดที่รันใน terminal ก็ทำงานใน cmux ได้ ไม่ต้องแก้ code หรือ config อะไรพิเศษ Claude Code, Codex CLI, OpenCode, Gemini CLI, Kiro, Aider, Goose, Amp, Cline, และทุก agent ที่รัน command line ทำงานได้ออกนอกกล่อง

ส่วน features พิเศษของ cmux เช่น browser pane, notification API, และ multi-pane control จะทำงานได้เมื่อ agent รู้ว่ามี cmux CLI อยู่ ซึ่งต้องบอกผ่าน system prompt หรือ configure hooks Claude Code มีการรองรับ cmux โดยตรงผ่าน integration examples ที่ cmux docs

cmux เบากว่า Electron-based terminals มากเพราะใช้ libghostty (GPU-accelerated, native) แทนที่จะ embed Chromium ในทางปฏิบัติ cmux app เองกิน RAM น้อย ส่วนที่กิน RAM คือ agent processes เช่น Claude Code instances ซึ่งเป็น memory cost ของ agent ไม่ใช่ cmux

ถ้ารัน Claude Code 4-6 instances พร้อมกัน RAM usage ส่วนใหญ่จะมาจาก Claude Code processes ไม่ใช่ cmux terminal เอง cmux เพิ่ม overhead เพียงเล็กน้อยต่อ pane

การย้ายไม่ได้ต้องเริ่มต้นใหม่ทั้งหมด cmux ยังเป็น terminal ดังนั้น shell scripts, dotfiles, และ aliases ทั้งหมดใช้ได้เหมือนเดิม สิ่งที่เปลี่ยนคือวิธีจัดการ windows: แทนที่ tmux prefix + c สำหรับ new window, ใน cmux ใช้ keyboard shortcut ของ app หรือ cmux new-pane command

Ghostty config (keybindings สำหรับ terminal) อ่านได้ตรงๆ cmux-specific shortcuts (workspaces, splits) customise ได้ใน Settings สิ่งที่ต้องเรียนใหม่คือ cmux CLI commands สำหรับ agent control ซึ่งไม่มี equivalent ใน tmux

ยัง — cmux ยังเป็น terminal ที่ดีมากสำหรับนักพัฒนาทั่วไป vertical sidebar พร้อม git branch/PR context มีประโยชน์แม้ทำงาน manual notification system ใช้ได้กับ long-running commands ใดก็ได้ไม่ใช่แค่ AI agents

แต่ถ้าจุดสำคัญของคุณคือ performance ล้วนๆ (ไม่ใช้ agent เลย) Ghostty อาจ ให้ raw terminal performance ที่ดีพอๆ กัน ในราคา "simpler app" ที่ focus กว่า cmux ให้ value สูงสุดเมื่อ agent integration คือ priority

⚠ ข้อควรรู้: cmux macOS only — ถ้า team ใช้ Linux หรือ Windows WSL บางส่วน ไม่สามารถ standardize บน cmux ได้ทั้งทีม ต้องพิจารณา workflow ที่ cross-platform ด้วย
★ สาระสำคัญ: เลือก cmux ถ้า AI agent คือ primary workflow บน macOS; tmux ถ้าต้อง cross-platform; Ghostty/iTerm2 ถ้ายังทำงาน manual เป็นหลัก

Closing — Key Takeaways

สิ่งที่ต้องจำและขั้นตอนถัดไป

cmux แสดงให้เห็นว่า terminal ในยุค AI agents ต้องการมากกว่าแค่ "ที่รัน command" — ต้องเป็น workspace ที่ agent ควบคุมได้, แจ้งได้, และทำงานร่วมกันได้ 7 ประเด็นด้านล่างคือ insight สำคัญที่ใช้เป็น mental model สำหรับ AI-native development

Key Takeaways

  • cmux ≠ terminal ธรรมดา: มันคือ workspace ที่ agent ควบคุม UI ตัวเองได้ผ่าน CLI — paradigm shift ที่เล็กในรูป แต่ใหญ่มากใน capability
  • libghostty คือ renderer ไม่ใช่ fork: cmux เป็น Swift app แยกที่ยืม rendering engine มา — อ่าน Ghostty config ได้แต่เป็น app คนละตัวสิ้นเชิง
  • Workspace→Surface→Pane ต้องจำ: ทุก cmux command target surface ID ที่ถูกต้อง — เข้าใจ hierarchy นี้ก่อนจะใช้ multi-agent ได้เต็มที่
  • browser snapshot คือ key insight: cmux browser snapshot คืน element refs (e10, e14) ที่ agent ใช้ click/fill — ทำให้ web automation เชื่อถือได้จริงๆ
  • notification hooks ควรตั้งตั้งแต่ต้น: configure Claude Code hooks ให้ emit cmux notify เมื่อ task เสร็จ — ไม่ต้องบอก agent ทุกครั้ง
  • sub-agent orchestration ไม่ต้องเขียน code: ใช้ cmux send + new-split ล้วนๆ primary agent ทำงานอื่นได้ระหว่างรอ sub-agent — parallel จริงๆ
  • ฟรีและ open source (AGPL): ไม่มี subscription ไม่มี lock-in ติดตั้งจาก cmux.com ได้เลย source code อยู่บน GitHub (manaflow-ai/cmux)
