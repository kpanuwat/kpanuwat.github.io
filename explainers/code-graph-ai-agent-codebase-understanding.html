Overview — The Problem

ทำไม AI Agent ถึงเสีย Token ส่วนใหญ่ไปกับการ "ค้นหา" ไม่ใช่ "เขียนโค้ด"

ก่อนที่ AI Agent จะตอบคำถามหรือแก้ bug ได้ มันต้องเข้าใจก่อนว่า codebase ทำงานอย่างไร กระบวนการนี้เรียกว่า Discovery Phase — และมันกิน Token budget ไปเป็นจำนวนมากโดยที่ผู้ใช้ไม่รู้ตัว Code Graph เกิดมาเพื่อแก้ปัญหานี้โดยตรง

ปัญหา — File-by-File

การสุ่มอ่านไฟล์

Agent อ่าน imports, grep patterns, trace function calls ทีละไฟล์ คำถามง่ายๆ อย่าง "อะไร call ProcessOrder?" เผาไป ~45,000 tokens ผ่าน read/grep หลายรอบ

ราคาแท้จริง — Context Waste

Discovery = 60–80% ของ Budget

งานวิจัยพบว่า agent อย่าง Claude Code, Cursor, Codex ใช้ token ส่วนใหญ่ใน "discovery phase" ไม่ใช่การเขียนโค้ด ยิ่ง codebase ใหญ่ยิ่งแย่

ทางออก — Pre-indexed Graph

Query ไม่ต้อง Explore

Code Graph index โครงสร้างโค้ดล่วงหน้า agent ถามกราฟโดยตรง ไม่ต้องอ่านไฟล์ คำถามเดิมใช้เพียง ~200 tokens ตอบใน <1ms

Key insight: ปัญหาไม่ได้อยู่ที่ AI ฉลาดหรือโง่ แต่อยู่ที่วิธีที่มันเข้าถึงข้อมูล — file system exploration เป็น O(n) ต่อ query ในขณะที่ graph query เป็น O(1) หลังจาก index
★ สาระสำคัญ: Code Graph เปลี่ยน AI จาก explorer ที่อ่านไฟล์สุ่ม เป็น navigator ที่รู้แผนผังโค้ดล่วงหน้า

Section 1 — Architecture Deep Dive

Tree-sitter + SQLite: วิธีสร้าง Knowledge Graph จากโค้ดจริง

Code Graph ไม่ได้เป็นแค่ "search index" ธรรมดา แต่เป็น property graph ที่เก็บโหนด (functions, classes, files) และ edges (CALLS, IMPORTS, INHERITS) ไว้ใน SQLite บนเครื่อง ทำให้ query ที่ต้องการความสัมพันธ์เชิงโครงสร้าง ตอบได้ในเวลาไม่ถึง 1 millisecond โดยไม่ต้องส่งข้อมูลออกไปยัง server ภายนอก

Tree-sitter: Parser ที่เข้าใจโค้ดในระดับ AST

Tree-sitter คือ library สำหรับ parse source code ให้กลายเป็น Abstract Syntax Tree (AST) ซึ่งเป็นการแสดงโครงสร้างของโค้ดในรูปแบบต้นไม้ ตัวอย่างเช่น ฟังก์ชัน processOrder(id) จะกลายเป็น node ประเภท "function_definition" ที่มี child nodes สำหรับชื่อฟังก์ชัน, parameter list, และ body ตามลำดับ ต่างจาก grep หรือ regex ที่มองโค้ดเป็น text ล้วน Tree-sitter เข้าใจว่าแต่ละส่วนของโค้ดมีบทบาทอะไรในเชิง syntax

โปรเจกต์ codebase-memory-mcp ซึ่งมีมากกว่า 30,000 GitHub stars รองรับ 66 ภาษาโดย vendor grammar ทั้งหมดเป็น compiled C source ไว้ใน binary เดียว ทำให้ไม่มี dependency ภายนอกและ grammar ไม่มีวันหายหรือ update กะทันหัน ครอบคลุมตั้งแต่ภาษาหลักอย่าง Python, TypeScript, Go, Rust, Java, C/C++ ไปจนถึง config format อย่าง YAML, TOML, Dockerfile และ scientific language อย่าง Julia, R

สิ่งที่ Tree-sitter extract ออกจาก AST ในแต่ละไฟล์ได้แก่: definitions (functions, classes, methods พร้อม signature), call sites (ทุกที่ที่มีการเรียก function อื่น), imports และ module references, HTTP route declarations (รองรับ 6 framework: Python/Flask/FastAPI, Go, Java/Spring, Kotlin/Ktor, Express.js, Laravel), และ trait/interface implementations ใน typed languages สำหรับ Go, C, C++ มี type-resolution pass เพิ่มเติมเพื่อแก้ปัญหา method dispatch และ pointer dereferencing ที่ทำให้ call graph ไม่ accurate ถ้า parse ด้วย string matching ล้วน

โค้ดต้นทาง → Tree-sitter parse เป็น AST → 6-Phase Build สร้าง graph ใน memory → บันทึกลง SQLite

SQLite Schema: Property Graph ที่เก็บทุกความสัมพันธ์ในโค้ด

ไม่เหมือน SQL database ทั่วไปที่จัดเก็บข้อมูลใน table แบบ tabular, Code Graph ใช้ property graph model บน SQLite ซึ่งประกอบด้วย typed nodes และ typed edges Node types ที่รองรับ ได้แก่: Project, File, Function, Class, Interface, Enum, Type, Route โดยแต่ละ node มี property เช่น qualified name, line number, signature, doc comment

ที่ทำให้ graph นี้ทรงพลังกว่า full-text search คือ edge types มากกว่า 15 ประเภทที่แสดงความสัมพันธ์เชิงโครงสร้าง: CALLS (ฟังก์ชัน A เรียก B), HTTP_CALLS (endpoint A เรียก endpoint B), ASYNC_CALLS (async/await call chain), IMPORTS (dependency), CONTAINS (file ประกอบด้วย class), DEFINES (class define method), IMPLEMENTS (class implement interface), INHERITS (class สืบทอด), USES_TYPE, THROWS, READS, WRITES, TESTS (test function cover function ใด), FILE_CHANGES_WITH (git co-change analysis), และ MEMBER_OF (community membership จาก Louvain algorithm)

Pipeline การสร้าง graph แบ่งเป็น 6 phase ภายใน database transaction เดียว: Phase 1-4 ทำงานบน parallel worker threads สร้าง in-memory hash maps indexed by qualified name Phase 5 ทำ bulk INSERT พร้อม deferred index creation (เร็วกว่า index-while-insert มากกว่า 10×) Phase 6 รัน Louvain community detection เพื่อหา module clusters และคำนวณ file-level change probability ผลลัพธ์คือ Django project (49,000 nodes) index เสร็จใน 6 วินาที, Linux kernel (2.1 ล้าน nodes) ใน 3 นาที และ incremental re-index เมื่อมีการแก้ไขไฟล์ใช้เวลาเพียง 1.2 วินาที

การตั้งค่าเพียงแค่ระบุ path ของ SQLite database ซึ่งจะถูกสร้างอัตโนมัติครั้งแรก ไม่ต้องการ server, cloud subscription, หรือ API key ใดๆ ทั้งสิ้น Background file watcher จะ monitor ไฟล์ที่เปลี่ยนแปลงและ trigger incremental re-index โดยอัตโนมัติ ทำให้ graph ไม่มีวัน stale แม้โค้ดจะเปลี่ยนบ่อยแค่ไหน

★ สาระสำคัญ: Tree-sitter แปลงโค้ดเป็น AST แล้ว 6-phase pipeline สร้าง property graph ใน SQLite — index ครั้งเดียว query ได้นับพัน

Section 2 — MCP Communication Layer

14 MCP Tools: วิธีที่ Agent ถามกราฟแทนการอ่านไฟล์

Model Context Protocol (MCP) คือ standard protocol ที่ทำให้ AI agent สื่อสารกับ tool server ได้โดยตรง Code Graph expose knowledge graph ทั้งหมดผ่าน 14 MCP tools แบ่งเป็น 4 หมวด ทำให้ agent ที่รองรับ MCP ไม่ว่าจะเป็น Claude Code, Cursor, Codex CLI, Gemini CLI สามารถ query โครงสร้างโค้ดได้ด้วย tool call เดียว โดยไม่ต้องปรับ prompt หรือ integration พิเศษ

หมวดที่ 1-2: Indexing และ Query Tools

ก่อนที่ agent จะ query ได้ ต้อง index codebase ก่อน ซึ่งทำผ่าน tool index_repository ที่รับ path และ trigger การ parse ทั้ง project ในพื้นหลัง status สามารถตรวจสอบได้ผ่าน get_index_status และ project ที่ index แล้วดูได้ผ่าน list_projects การ index เป็น one-time operation ที่ตามด้วย incremental update อัตโนมัติ

Query tools คือหัวใจของระบบ ประกอบด้วย:

  • search_symbols — ค้นหา function/class ตามชื่อหรือ regex pattern ส่งกลับ qualified name, file, line number ใน <1ms
  • trace_call_path — หา call chain ระหว่าง function A และ B ใช้ BFS traversal บน graph ตอบใน ~0.3ms แม้ depth=5
  • query_graph — รัน Cypher-like query บน graph โดยตรง สำหรับคำถามซับซ้อนที่ต้องการ multi-hop traversal
  • ingest_trace — นำ runtime trace มา overlay บน static graph เพื่อ enrich ด้วยข้อมูล dynamic

ตัวอย่างการใช้ที่เห็นความแตกต่างชัดที่สุด: เมื่อ agent ต้องการรู้ว่า "อะไรเรียก sendEmail() บ้าง?" แทนที่จะ grep ทุกไฟล์ (4-8 tool calls, ~45,000 tokens) agent เพียง call trace_call_path หนึ่งครั้ง ได้กลับมาเป็น list of callers พร้อม file path และ line number ทั้งหมดใน <1ms ใช้ไม่ถึง 500 tokens

หมวดที่ 3-4: Analysis และ Code Retrieval Tools

Analysis tools ช่วย agent วิเคราะห์ impact ของการเปลี่ยนแปลงก่อนลงมือแก้ไข detect_change_impact รับ list of files หรือ functions ที่จะแก้ไข แล้วคำนวณ transitive impact โดยเดิน edges ย้อนกลับบน graph ผลลัพธ์คือ ranked list ของ functions ที่อาจได้รับผลกระทบ พร้อม confidence score ซึ่งช่วยให้ agent ตัดสินใจได้ว่าต้องเขียน test เพิ่มที่ไหน

get_architecture สร้าง high-level overview ของ project ในทันที รวมถึง entry points, major components, dependency clusters, และ HTTP routes ทั้งหมด tool นี้มีประโยชน์มากเมื่อ onboard codebase ใหม่ — ข้อมูลที่ปกติใช้เวลาอ่าน README + ดู directory หลายนาที กลายเป็นคำตอบใน 1-2 tool calls

Code tools ได้แก่ get_code_snippet (ดึง source code ของ function ที่ระบุโดย qualified name) และ search_full_text (full-text search บน source content ที่ index ไว้แล้ว) ความสำคัญของ get_code_snippet คือ agent ไม่ต้องรู้ file path หรือ line number แค่รู้ชื่อ function ก็ดึง implementation ออกมาได้ ซึ่งลดการ hallucination เรื่อง path ได้มาก

Before: Agent อ่านไฟล์หลายรอบ ~45,000 tokens / After: MCP tool call เดียว ~200 tokens, ตอบใน <1ms

★ สาระสำคัญ: MCP เปลี่ยน "การสุ่มอ่านไฟล์" เป็น "structured query" — agent ถามกราฟ 1 ครั้งแทนการ grep 8 รอบ

Section 3 — Performance Benchmarks

ตัวเลขจริง: ประหยัด Token ได้มากแค่ไหนในแต่ละ Use Case

ตัวเลข "121× savings" ฟังดูใหญ่เกินจริง แต่มาจาก benchmark ที่ run บน real codebase ความสำคัญไม่ได้อยู่แค่จำนวนรวม แต่อยู่ที่ pattern — query ที่ต้องการ structural traversal ประหยัดได้มากที่สุด ในขณะที่ query ที่ต้องการ pattern matching แบบ exhaustive กราฟยังสู้ file exploration ไม่ได้ทุกกรณี

Token Savings ต่อ Query Type: ความแตกต่างที่ชัดเจน

ตาราง benchmark ด้านล่างเปรียบเทียบ token ที่ใช้ระหว่าง Knowledge Graph (Code Graph) กับ File Search (วิธีเดิมที่ agent grep และ read files) สำหรับ query ประเภทต่างๆ สังเกตว่า query ที่ประหยัดได้มากที่สุดคือ query ที่ต้องการ "traversal" เช่น trace call chain เพราะปกติ agent ต้องอ่านไฟล์หลายชั้นเพื่อตามหา caller ในขณะที่กราฟ pre-compute ไว้แล้ว

ที่น่าสังเกตคือ "Architecture overview" ประหยัดได้น้อยที่สุดในกลุ่ม (67×) เพราะ answer มี context มากกว่าจึงต้องใช้ token ในการ render ผลลัพธ์ด้วย ในขณะที่ "ค้นหา function ตาม pattern" ประหยัดได้ 225× เพราะคำตอบเป็นแค่ list ของ symbols ไม่ต้องอธิบาย context เพิ่ม

ผลการทดสอบจาก paper "Codebase-Memory" บน 31 repositories จริงยืนยันว่า MCP agent ใช้ tool calls เฉลี่ย 2.3 ครั้ง/คำถาม เทียบกับ file exploration ที่ใช้ 4.8 ครั้ง (ลดลง 2.1×) และใช้ tokens เฉลี่ย ~1,000/คำถาม เทียบกับ ~10,000 (ลดลง 10×) ส่วน enterprise deployment ของ codegraph รายงาน ~70% fewer tool calls และ ~35% lower cost บน production workload

Indexing Speed: โปรเจกต์ใหญ่แค่ไหนก็ index ได้

ประสิทธิภาพของ indexing เป็นหัวใจสำคัญ เพราะถ้า index ช้า developer จะไม่อยากรอ ผลการทดสอบบน Apple M3 Pro แสดงให้เห็นว่า:

Real-world Project

Django (49,000 nodes)

~6 วินาที สำหรับ initial index — เร็วกว่าการเปิด IDE ส่วนใหญ่

Large Open Source

Linux Kernel (2.1M nodes)

~3 นาที สำหรับ 28 ล้านบรรทัด 75,000 ไฟล์ — 4.9 ล้าน edges

Daily Use

Incremental Re-index

~1.2 วินาที เมื่อมีการแก้ไขไฟล์ — background update อัตโนมัติ

Query Latency

Sub-millisecond

Cypher query <1ms / BFS call-path (depth=5) ~0.3ms บน graph ทุกขนาด

ที่น่าทึ่งกว่า raw speed คือ query latency ไม่ได้ขึ้นกับขนาดของ codebase อย่างมีนัยสำคัญ เพราะ SQLite index ทำให้ lookup เป็น O(log n) ไม่ใช่ O(n) agent ที่ query Django กับ Linux kernel จะได้คำตอบเร็วพอๆ กัน ตรงกันข้ามกับ file exploration ที่ช้าขึ้นเป็น linear ตามจำนวนไฟล์

ข้อแลกเปลี่ยนที่ต้องรู้: quality score จากงานวิจัยพบว่า MCP agent ได้ 0.83 เทียบกับ file explorer ที่ 0.92 ความแตกต่าง 9% นี้ส่วนใหญ่มาจาก query ที่ต้องการ exhaustive pattern matching หรือ line-level source เช่น "หา comment ทุกที่ที่พูดถึง TODO" ซึ่ง full-text grep ยังดีกว่า graph traversal แต่สำหรับ query เชิงโครงสร้าง graph ชนะขาด

★ สาระสำคัญ: Query latency <1ms ไม่ว่า codebase จะใหญ่แค่ไหน เพราะ SQLite index ทำให้ graph lookup เป็น O(log n)

Section 4 — Conceptual Comparison

Code Graph ≠ RAG: ทำไมความคล้ายคลึงไม่พอสำหรับโค้ด

เมื่อพูดถึงการทำให้ AI เข้าใจ codebase หลายคนนึกถึง RAG (Retrieval-Augmented Generation) ซึ่งใช้ vector embedding เพื่อหาไฟล์ที่ "คล้าย" กับคำถาม แต่ Code Graph เลือกแนวทางที่ต่างออกไปโดยสิ้นเชิง การเข้าใจว่าทั้งสองต่างกันอย่างไรช่วยให้เลือกใช้ได้ถูกเวลา

ปัญหาหลักของ RAG กับ Codebase: ความคล้ายคลึงไม่ใช่ความสัมพันธ์

RAG ทำงานโดยการ embed document เป็น vector แล้วหา vector ที่ใกล้เคียงกับ query vector วิธีนี้เหมาะมากกับเนื้อหาที่เป็น prose เช่น เอกสาร, บทความ, หรือ FAQ เพราะ semantic similarity ของ text มักสัมพันธ์กับความเกี่ยวข้องของเนื้อหา

แต่โค้ดมีคุณสมบัติที่ต่างออกไปอย่างสิ้นเชิง: ฟังก์ชัน processPayment() และ refundPayment() อาจมี embedding ที่ใกล้กันมากเพราะชื่อคล้ายกัน แต่สิ่งที่ developer ต้องการรู้คือ "อะไรเรียก processPayment()?" ซึ่งเป็นคำถามเชิงโครงสร้าง ไม่ใช่เชิงความหมาย vector search จะค้นได้ว่าไฟล์ไหน "พูดถึง" payment แต่ตอบไม่ได้ว่าไฟล์ไหน "เรียก" processPayment ผ่าน call chain กี่ชั้น

อีกปัญหาหนึ่งคือ RAG สำหรับโค้ดมักต้อง embed ทั้ง function หรือทั้งไฟล์ ทำให้ precision ต่ำเมื่อต้องการความเฉพาะเจาะจง เช่น "ใน class UserService method ไหนที่มี side effect ต่อ database?" — RAG อาจคืน UserService กลับมา แต่ไม่รู้ว่า method ไหนในนั้นที่ทำ database write เพราะความสัมพันธ์นั้น ต้องอ่านจาก WRITES edge ใน property graph ไม่ใช่จาก text similarity

เปรียบเทียบ 3 แนวทาง: Vector/RAG, Repo-Map, และ Code Graph

Embedding-based Retrieval

วิธีที่แพร่หลายที่สุด: embed โค้ดเป็น vector เก็บใน vector database (Pinecone, Weaviate, pgvector) แล้วใช้ cosine similarity หา chunk ที่ใกล้เคียงกับ query ข้อดีคือ setup ง่าย ทำงานได้กับทุกภาษา และ handle prose + code ได้ในระบบเดียว

ข้อเสีย: ต้องการ vector database (infrastructure overhead), ต้องมี embedding model (API cost หรือ local model), semantic similarity ไม่ตรงกับ structural relationship สำหรับโค้ด และ precision สำหรับ exact call-graph query ต่ำกว่า graph approach มาก ใช้ token ~2-5K ต่อ query (ยังต้องโหลด context เพิ่มเพื่อ answer)

Repository Map (aider style)

สร้าง text summary ของ codebase structure (function signatures, class hierarchy) แล้วใส่เข้า context window ทั้งก้อน ไม่ต้องมี embedding หรือ vector DB ข้อดีคือ simple มาก ทำงานได้ดีกับ repo ขนาดกลาง

ข้อเสีย: repo-map ใช้ token ~1K แต่เป็น static text ไม่ query ได้ agent ต้องอ่าน map ทั้งหมดทุกครั้งแม้จะต้องการข้อมูลเพียงส่วนเล็กน้อย และไม่รองรับ structural query เช่น call-path tracing หรือ impact analysis สำหรับ repo ใหญ่ map อาจใหญ่เกิน context window

Code Graph (Property Graph + MCP)

Parse โค้ดเป็น AST ด้วย Tree-sitter สร้าง typed property graph ใน SQLite expose ผ่าน MCP tools ไม่ต้องการ vector DB หรือ embedding model ตอบ structural query ด้วย precision สูงสุด และ latency <1ms

เหมาะที่สุดสำหรับ: ตาม call chain, ค้นหา dependency, วิเคราะห์ impact, ดู architecture overview, หา dead code, และ understand unfamiliar codebase token cost ~200-1,500/query ขึ้นกับ query type

ข้อจำกัด: static analysis เท่านั้น — runtime behavior, reflection, dynamic dispatch ไม่ถูก capture ใน graph ภาษาที่ใช้ macro มาก (C) quality ลดลง (score 0.58 vs 0.83 average)

Section 5 — Decision Framework

เมื่อไหรที่ Code Graph คุ้มค่า และเมื่อไหรที่เป็น Overhead ไม่จำเป็น

ไม่มี tool ที่เหมาะกับทุกสถานการณ์ Code Graph คุ้มค่าที่สุดเมื่อ codebase ใหญ่พอที่ Discovery Phase เป็นปัญหาจริงๆ แต่สำหรับโปรเจกต์เล็กที่ context window รับ codebase ทั้งหมดได้ การ setup graph อาจเป็น overhead ที่ไม่คืนค่า framework ด้านล่างช่วย decide ได้

สัญญาณที่บอกว่าถึงเวลาใช้ Code Graph

เกณฑ์ที่ชัดที่สุดคือ token cost ของ Discovery Phase ถ้า agent ใช้ token ไปกับการ grep, read, และ trace มากกว่า 30% ของ budget ก่อนจะเริ่มตอบ นั่นคือสัญญาณที่ชัดเจนว่า graph จะคืนค่า ในทางปฏิบัติ สัญญาณเหล่านี้มักปรากฏพร้อมกัน:

  • ขนาด codebase > 50 ไฟล์หรือ > 5,000 บรรทัด: context window ไม่สามารถ load ทั้งหมดได้ agent จำเป็นต้องเลือกว่าจะอ่านไฟล์ไหน ซึ่งหมายถึง exploration overhead
  • Agent มักถาม call-graph question: "อะไรเรียก X?", "X ถูกใช้ที่ไหนบ้าง?", "ถ้าแก้ Y จะกระทบอะไร?" — นี่คือ query ที่ graph ตอบได้ดีที่สุด
  • Team ขนาดใหญ่กับ unfamiliar codebase: developer ใหม่ใช้ graph ทำความเข้าใจ architecture ได้เร็วกว่าอ่าน README + trace code มาก
  • Monorepo หรือ microservices ที่มี cross-service calls: HTTP call detection ช่วย map dependency ระหว่าง service ซึ่งปกติหาได้ยากมากจาก code alone
  • Refactoring ขนาดใหญ่ที่ต้องรู้ impact: detect_change_impact ช่วยให้รู้ว่าต้องแก้หรือ test ที่ไหนเพิ่มก่อน merge

จากการทดสอบบน 31 repositories จริง MCP agent ได้คะแนน quality 0.83 เทียบกับ file explorer 0.92 ความต่าง 9% นี้ยอมรับได้สำหรับ large codebase เพราะ token savings 10× และ speed ที่เพิ่มขึ้นคุ้มค่ากว่า แต่สำหรับโปรเจกต์เล็กที่ file explorer ยังทำงานได้ดีและ fast อยู่แล้ว การสูญเสีย 9% quality อาจไม่คุ้ม

เมื่อไหรที่ Code Graph เป็น Overhead ที่ไม่คุ้มค่า

สำหรับโปรเจกต์ขนาดเล็กถึงกลางที่ context window ยังรับทั้ง codebase ได้ การ index และ maintain graph อาจเพิ่มความซับซ้อนโดยไม่ได้ประโยชน์คืน กฎง่ายๆ: ถ้า agent อ่านทุกไฟล์ในโปรเจกต์ได้ใน 1-2 tool calls โดยไม่ hit token limit codebase นั้นยังไม่จำเป็นต้องใช้ graph

ข้อจำกัดเชิงเทคนิคที่ต้องรู้ก่อนตัดสินใจ:

Tree-sitter parse โค้ดก่อน preprocessing ทำให้ macro expansion ที่เปลี่ยน call graph ณ compile time ไม่ถูก capture สำหรับ C codebase ที่ใช้ macro เยอะ quality score ลดจาก 0.83 เฉลี่ยเหลือ 0.58 ซึ่งอาจไม่เพียงพอสำหรับงานที่ต้องการ accuracy สูง แนะนำให้ใช้ร่วมกับ file exploration แบบ hybrid สำหรับ C project

Code Graph วิเคราะห์ static structure เท่านั้น ถ้าโค้ดใช้ reflection, dependency injection, หรือ dynamic dispatch (เช่น Python's __getattr__, Java Spring beans, Ruby method_missing) call graph ที่ graph เห็นจะ incomplete ในกรณีเหล่านี้ tool ingest_trace ช่วยได้บางส่วนโดยรับ runtime trace มา overlay บน static graph แต่ต้องมี runtime data ก่อน

ถ้า codebase ทั้งหมดอยู่ใน 1-3 ไฟล์ หรือเป็น script ที่เขียนครั้งเดียวทิ้ง overhead ของการ setup MCP server, config, และ first-time indexing ไม่ได้รับความคุ้มค่ากลับมา สำหรับ prototype ที่ต่ำกว่า 200 บรรทัด ให้ใส่โค้ดทั้งหมดเข้า context โดยตรงยังดีกว่า

Query ที่ต้องการ scan ทุกบรรทัดเช่น "หา TODO comment ทั้งหมดใน repo" หรือ "หาทุกที่ที่ใช้ string literal 'error'" — full-text grep ยังดีกว่า graph traversal ใน case นี้ เพราะ graph ไม่ได้ index arbitrary text content ทุกบรรทัด แต่ index เฉพาะ structural elements (function, class, call) Code Graph มี full-text search tool แต่สำหรับ pattern ที่ arbitrary มากๆ grep โดยตรงยังครอบคลุมกว่า

Rule of Thumb: ถ้า agent ใช้เวลา >30% ไปกับ discovery ก่อนตอบ → Code Graph คุ้มค่า ถ้า codebase <50 ไฟล์และ agent ตอบได้โดยไม่ต้องอ่านไฟล์มากกว่า 3 ไฟล์ → ยังไม่จำเป็น
★ สาระสำคัญ: Code Graph คุ้มค่าเมื่อ codebase ใหญ่พอที่ Discovery Phase กิน token เยอะ — สำหรับโปรเจกต์ <50 ไฟล์ context window ยังพอ

Closing — Key Takeaways

สิ่งที่ต้องจำและขั้นตอนถัดไป

Code Graph เป็นแนวคิดที่เปลี่ยน fundamental ว่า AI Agent เข้าถึง codebase อย่างไร จาก exploration-based (อ่านไฟล์สุ่มจนเข้าใจ) เป็น query-based (ถามกราฟที่รู้โครงสร้างทั้งหมดแล้ว)

Key Takeaways

  • Discovery Phase คือต้นทุนที่ซ่อนอยู่: AI agent เสีย 60-80% ของ token budget ใน "การหา" ไม่ใช่ "การตอบ" Code Graph ตัด overhead นี้ออกด้วย pre-indexed graph
  • Tree-sitter + SQLite = precision + speed: AST parsing ให้ความเข้าใจโครงสร้างจริง ไม่ใช่ text matching; property graph ใน SQLite ตอบ structural query ใน <1ms ไม่ว่า repo จะใหญ่แค่ไหน
  • MCP คือ interface ที่ทำให้ universal: agent ทุกตัวที่รองรับ MCP (Claude Code, Cursor, Codex, Gemini) ใช้ graph ได้ทันที ไม่ต้องปรับ prompt หรือ custom integration
  • Graph ≠ RAG: ตอบคำถามคนละประเภท: RAG ตอบ "อะไรคล้ายกัน?" — Graph ตอบ "อะไรเชื่อมกัน?" คำถามโค้ดส่วนใหญ่ต้องการ structural relationship ไม่ใช่ semantic similarity
  • มี trade-off จริง: quality score ลดจาก 0.92 เหลือ 0.83 เมื่อเปลี่ยนจาก file exploration เป็น graph สำหรับ exhaustive search และ macro-heavy language กราฟยังสู้ grep ไม่ได้
  • ขนาดโปรเจกต์กำหนดความคุ้มค่า: ถ้า codebase <50 ไฟล์และ context window รับได้ — graph เป็น overhead ที่ไม่คืนค่า ถ้า >50 ไฟล์และ agent ถาม call-graph บ่อย — คุ้มมาก
  • Static analysis มีขีดจำกัด: runtime behavior, reflection, dynamic dispatch ไม่อยู่ใน graph สำหรับโค้ดที่ใช้ dependency injection หนัก ให้ใช้ ingest_trace เสริมด้วย runtime data

Glossary

AST (Abstract Syntax Tree)
โครงสร้างต้นไม้ที่แสดง syntax ของโค้ดในรูปแบบที่ program อ่านได้ — ระดับเดียวกับที่ compiler ใช้วิเคราะห์โค้ด
Tree-sitter
Library open-source สำหรับ parse source code ให้กลายเป็น AST รองรับ 100+ ภาษา เร็วพอสำหรับ real-time parsing ใน editor
MCP (Model Context Protocol)
Protocol มาตรฐานของ Anthropic สำหรับให้ AI agent สื่อสารกับ external tool server — คล้าย REST API แต่ออกแบบมาสำหรับ AI โดยเฉพาะ
Property Graph
Graph model ที่ทั้ง node และ edge มี properties (key-value pairs) ต่างจาก simple graph ที่ edge มีแค่ relationship type
RAG (Retrieval-Augmented Generation)
เทคนิคที่ดึง document ที่เกี่ยวข้องจาก vector database มาใส่ใน context ก่อนให้ LLM ตอบ — ใช้ semantic similarity เป็น retrieval mechanism
Louvain Community Detection
Algorithm สำหรับหา "community" หรือ cluster ใน graph — ใช้เพื่อหา module groupings ใน codebase โดยอัตโนมัติจากความหนาแน่นของ edges
★ ถ้าจำได้อย่างเดียว: Code Graph เปลี่ยน AI จาก "explorer ที่อ่านไฟล์สุ่ม" เป็น "navigator ที่ query graph" — ประหยัด token 121× ใน structural query
