Claude Code Agent

6 — Host Environment

VM / Proxmox

รัน Docker containers และ Claude Code — Pi Zero เชื่อมผ่าน Ethernet ทำให้ VM มี "virtual serial ports" ที่ชี้ไปยัง ESP32 จริงๆ บน workbench

7 — IoT Test Env

Test Simulation Layer

Pi สามารถสร้าง WiFi AP, ปิด MQTT broker, ตัด network กลางอากาศ — จำลอง real-world failure conditions เพื่อทดสอบ error handling ของ firmware

ESP32 Target Board

Big Picture

Common Trap

Actionable

Non-obvious

Hardware Architecture — Intuition

ทำไมถึงต้องมี Pi Zero Bridge

ก่อนที่จะเข้าใจ workbench นี้ได้ต้องเข้าใจก่อนว่า ปัญหาที่ระบบนี้แก้คืออะไร ในโลกของ embedded development การที่ AI ต้องการ "แตะ" hardware โดยตรงเป็นสิ่งที่ยากมากเพราะ hardware ต้องต่อเข้า USB หรือ serial port ของเครื่องที่รัน AI อยู่ แต่เมื่อ AI รันอยู่ใน VM หรือ cloud environment การต่อ USB ตรงๆ เป็นสิ่งที่ทำไม่ได้โดยง่าย Pi Zero 2W คือทางออกที่หรูหราที่สุดสำหรับปัญหานี้

ปัญหาที่ workbench แก้: Serial Port ใน VM ไม่พอ

Proxmox และ hypervisor ทั่วไปมีข้อจำกัดเรื่อง serial controller — โดยปกติ VM หนึ่งจะได้รับ serial controller ได้จำนวนจำกัด และการ passthrough USB device ไปยัง VM ก็มีความซับซ้อนพอสมควร ถ้าต้องการต่อ ESP32 สาม, สี่, หรือห้าบอร์ดพร้อมกัน เรื่องนี้กลายเป็นปัญหาใหญ่ทันที

นี่คือจุดที่ Raspberry Pi Zero 2W เข้ามาแก้ปัญหาได้อย่างสง่างาม: แทนที่จะพยายาม passthrough USB ทุกตัวเข้า VM โดยตรง ให้ Pi Zero เป็นตัวรับ USB device ทั้งหมดแทน จากนั้น Pi expose serial ports เหล่านั้นออกมาเป็น TCP socket ผ่าน Ethernet — VM เชื่อมต่อมาที่ TCP socket นั้น และมองเห็นเหมือนเป็น serial port ปกติ ปัญหาหมดไปทันที

analogyที่ดีที่สุดคือ USB print server ที่คนใช้กันในออฟฟิศ — แทนที่ทุกคนจะต้องเสียบสาย USB เข้า printer โดยตรง ก็ต่อ printer เข้า print server ครั้งเดียว แล้วทุกคนใช้ผ่าน network ได้ workbench นี้ก็ทำแบบเดียวกันกับ serial ports ของ ESP32

ทำไมต้อง Automate การ Flash และทดสอบ

ก่อนจะมีระบบ automated นี้ embedded developer ต้องทำขั้นตอนซ้ำๆ ทุกครั้งที่แก้โค้ด: เสียบ USB เข้าบอร์ด, เปิด terminal ใหม่, เลือก port ที่ถูก, รัน esptool, รอ flash เสร็จ, เปิด serial monitor, ดู boot log, แล้วก็ทำซ้ำ loop นี้ทุกครั้งที่แก้บรรทัดเดียว นี่คือ "context switch cost" ที่สูงมากและเป็นหนึ่งในสาเหตุหลักที่ embedded dev ใช้เวลานาน

เมื่อ AI รันอยู่ใน terminal และมีสิทธิ์เข้าถึง serial bridge ผ่าน network AI สามารถสั่ง flash ได้เองด้วย command เดียว จากนั้นเปิด serial monitor อ่าน output ประเมินผล และถ้ามี error ก็แก้โค้ดแล้ว flash ใหม่ทันที — ทั้งหมดนี้เกิดขึ้นภายใน terminal เดียวกันโดย developer ไม่ต้องลุกจากเก้าอี้

ผลลัพธ์ที่ได้คือ feedback loop ที่เร็วมาก ใน embedded development ปกติ feedback loop อาจใช้เวลา 3-5 นาทีต่อ iteration แต่ด้วยระบบ automated นี้ลดลงเหลือไม่กี่สิบวินาที เมื่อ AI ทำซ้ำได้หลายร้อยรอบในชั่วโมงเดียว ผลลัพธ์ก็คือโปรเจกต์เสร็จเร็วกว่ามากโดยธรรมชาติ

Analogy: USB-over-Network Server สำหรับ Embedded

ถ้าเคยใช้ NAS หรือ network printer จะเข้าใจ concept นี้ได้ทันที — Pi Zero 2W ทำหน้าที่เหมือน device server ในโลก industrial automation ซึ่งเป็นเทคโนโลยีที่มีมานานแล้ว แต่ที่นี่เราใช้กับ ESP32 developer board ราคาถูกแทน serial equipment ราคาแพง

ข้อดีของ approach นี้คือ scalable มาก ถ้าต้องการเพิ่ม ESP32 board อีกสิบบอร์ดก็แค่เพิ่ม USB Hub ให้ใหญ่ขึ้น ไม่ต้องแตะ VM configuration ใดๆ เลย Pi Zero จะ expose serial ports เพิ่มเข้ามาใน network โดยอัตโนมัติ

และที่สำคัญมากคือ การออกแบบแบบนี้ทำให้ workbench สามารถอยู่ได้แม้ไม่มี cloud — ทุกอย่างรันใน local network ข้อมูล firmware และ code ไม่ออกไปนอก premise เลย สำหรับ project ที่มีความ sensitive เรื่อง IP นี่เป็นข้อดีที่สำคัญมาก

Key Insight: Pi Zero 2W คือ "serial bridge" ที่ทำให้ VM สั่ง ESP32 ได้เหมือนเสียบ USB โดยตรง — แก้ปัญหา serial passthrough ของ VM ด้วยวิธีที่ scalable และไม่ต้องแตะ hypervisor configuration
Hardware Architecture — Mechanism

การตั้งค่าและ Wiring

ส่วนนี้เจาะลึกรายละเอียดของ hardware ที่ต้องใช้และวิธีที่ Pi Zero 2W expose serial ports ออกมาผ่าน network TCP socket เพื่อให้ VM และ Claude Code เชื่อมต่อได้ รายละเอียดเหล่านี้สำคัญสำหรับผู้ที่ต้องการ replicate ระบบนี้เอง

อุปกรณ์ Hardware ที่ต้องใช้

Raspberry Pi Zero 2W เป็น single-board computer ที่ใช้ ARM Cortex-A53 quad-core และรัน Raspberry Pi OS ได้เต็มๆ มี USB OTG port ที่สามารถทำงานเป็น USB host ได้ด้วย adapter ราคาอยู่ที่ประมาณ $15 USD ซึ่งถือว่าถูกมากเมื่อเทียบกับประโยชน์ที่ได้

USB Hub แบบ powered (มี power adapter ของตัวเอง) เป็นสิ่งจำเป็น เพราะ Pi Zero 2W ไม่สามารถจ่ายไฟให้ ESP32 หลายตัวพร้อมกันได้เพียงพอ ราคาอยู่ที่ $10-20 USD USB Hub แบบ 4-7 port ก็เพียงพอสำหรับการใช้งานทั่วไป

ESP32 boards ที่ต่อเข้าระบบต้องใช้ USB data cable เท่านั้น ไม่ใช่ power-only cable (สายที่มีแค่ 2 เส้นสำหรับชาร์จ) สายที่ดีจะมี 4 เส้น รวม D+ และ D- สำหรับ data transfer ถ้าเสียบแล้ว Pi ไม่เห็นบอร์ด ให้ตรวจสอบสายก่อนเสมอ นอกจากนี้ยังต้องมี Ethernet cable เชื่อมต่อ Pi กับ VM ด้วย Ethernet เสถียรกว่า WiFi สำหรับ serial forwarding

วิธีที่ Pi Expose Serial Ports ผ่าน Network

เมื่อ ESP32 เชื่อมต่อผ่าน USB Pi Linux จะเห็น device เป็น /dev/ttyUSB0, /dev/ttyUSB1 เป็นต้น ขึ้นอยู่กับจำนวนบอร์ดที่ต่อ วิธีที่ง่ายที่สุดในการ forward serial port ผ่าน network คือใช้ socat หรือเขียน Python script เองก็ได้

# บน Pi Zero 2W: forward /dev/ttyUSB0 ไปเป็น TCP port 8000
socat TCP-LISTEN:8000,reuseaddr,fork \
      /dev/ttyUSB0,raw,b115200

# หรือใช้ Python script
python3 -c "
import serial, socket, threading
ser = serial.Serial('/dev/ttyUSB0', 115200)
srv = socket.socket(); srv.bind(('0.0.0.0', 8000)); srv.listen(1)
conn, _ = srv.accept()
# forward bidirectional
"

บน VM หรือ Docker container สามารถ connect กลับมาได้ด้วย socat เช่นกัน:

# บน VM: สร้าง virtual serial port ที่ชี้ไปยัง Pi
socat PTY,link=/dev/ttyVUSB0,raw \
      TCP:192.168.1.100:8000

# Claude Code จากนั้นใช้ /dev/ttyVUSB0 เหมือน serial port ปกติ
esptool.py --port /dev/ttyVUSB0 --baud 115200 write_flash 0x0 firmware.bin

pattern นี้ทำงานได้อย่างโปร่งใส — Claude Code ไม่รู้ว่า serial port ที่ใช้อยู่เป็น "virtual" ที่อยู่ไกลออกไป 10 เมตรหรืออยู่ข้างๆ เครื่อง การ flash firmware และ monitor serial output ทำงานได้เหมือนกันทุกประการ

Wiring Diagram

แผนภาพ wiring: ESP32 (USB data cable) → USB Hub (powered) → Pi Zero 2W → Ethernet → VM Docker container — socat สร้าง virtual serial port ใน container ที่ชี้กลับไปยัง Pi ผ่าน TCP

Core Pattern: serial-over-network ไม่ใช่ของใหม่ — ยืมมาจาก industrial automation แต่นำมาใช้กับ ESP32 developer board ราคาถูกและ open-source tools เพื่อสร้าง workbench ที่ cost-effective มากๆ

ทำไม AI ต้องการ Workflow ที่ชัดเจน

ลองนึกภาพว่าจ้างช่างก่อสร้างมาสร้างบ้าน แต่บอกแค่ว่า "สร้างบ้านให้หน่อย" โดยไม่มีแบบแปลนและ specification ใดๆ เลย ผลลัพธ์ที่ได้จะขึ้นอยู่กับการตีความของช่างล้วนๆ และอาจไม่ตรงกับที่ต้องการเลย embedding development ก็เหมือนกัน — AI ไม่รู้ว่า project ควรใช้ MCU อะไร, communication protocol อะไร, pin ไหนต่อกับอะไร, และควร test อะไรบ้าง ถ้าไม่มีข้อมูลเหล่านี้

FSD (Functional Specification Document) คือ "blueprint" ที่เติมเต็มช่องว่างเหล่านั้น เมื่อ AI มี FSD ที่สมบูรณ์ก็สามารถทำงานได้อย่างมีทิศทางตั้งแต่ต้นจนจบ โดยไม่ต้องถามซ้ำแล้วซ้ำเล่า หรือเดาข้อมูลที่ไม่มี ความสำคัญของขั้นตอน FSD นี้จึงไม่ใช่แค่ "documentation" แต่เป็น "machine-readable instruction" สำหรับ AI โดยตรง

เปรียบเทียบกับ architectural blueprint ที่ architect วาดไว้ก่อนที่ contractor จะลงมือสร้าง — blueprint ดีทำให้ contractor ทำงานได้รวดเร็วและถูกต้อง blueprint แย่ทำให้เสียเวลาแก้ไขตลอด FSD ใน embedded dev ก็ทำหน้าที่เดียวกันทุกประการ

Workflow — Mechanism

FSD SOP Skill และ Code Generation Loop

FSD SOP Skill คือ custom AI skill ที่เป็นหัวใจของ workflow — มันแปลง idea ที่เขียนด้วยภาษามนุษย์ให้กลายเป็น structured specification ที่ AI ใช้ทำงานได้โดยตรง ส่วน code generation loop คือกลไกที่ทำให้ AI เขียนโค้ด, แก้ error, และ iterate จนได้ firmware ที่ทำงานได้โดยไม่ต้องมีมนุษย์เข้ามาแทรก

FSD SOP Skill ทำงานอย่างไร

SOP ย่อมาจาก Standard Operating Procedure — FSD SOP Skill คือ custom skill ที่สร้างขึ้นสำหรับ Claude Code โดยเฉพาะ เมื่อ developer รัน skill นี้ มันจะอ่าน idea.md และ generate FSD ออกมาในรูปแบบที่ AI สามารถใช้ทำงานต่อได้ทันที ไม่ใช่แค่ documentation สำหรับมนุษย์อ่าน

ตัวอย่าง idea.md สำหรับ iOS Voice Keyboard project อาจมีเนื้อหาประมาณนี้: "ต้องการสร้าง keyboard ที่รับ input จาก iPhone ผ่าน speech-to-text แล้วส่งข้อความผ่าน Bluetooth ไปยัง ESP32 ซึ่งทำหน้าที่เป็น USB HID keyboard รองรับภาษาไทยและภาษาอังกฤษ" จาก idea ไม่กี่บรรทัดนี้ FSD SOP Skill จะ generate spec ที่มีความละเอียดมากขึ้นอย่างมาก

FSD ที่ generate ออกมาจะประกอบด้วยหัวข้อสำคัญเช่น: MCU selection (ESP32 พร้อม rationale ว่าทำไม), communication protocols ที่ใช้ (BLE สำหรับรับจาก iPhone, USB HID สำหรับส่งไปยัง host), pin assignments (USB D+/D- ต่อ pin ไหน), component list, และ test cases ที่ครอบคลุม happy path และ edge cases ทั้งหมด

# idea.md (input)
# iOS Voice Keyboard
Phone does speech-to-text, sends text via BLE to ESP32.
ESP32 acts as USB HID keyboard to host computer.
Must support Thai and English characters.
Must reconnect automatically if disconnected.

# FSD output (excerpt, generated by AI skill)
## MCU Selection
- Target: ESP32-S3 (native USB, BLE 5.0)
- Rationale: native USB HID avoids CH340 chip complexity

## Protocols
- BLE (central: iOS app, peripheral: ESP32)
- USB HID (ESP32 as keyboard to host)

## Test Cases
- TC001: BLE pairing success within 10s
- TC002: Thai character U+0E01 transmitted correctly
- TC003: USB disconnect → auto-reconnect within 5s

Code Generation และ Self-Healing Compilation

เมื่อ AI มี FSD ที่สมบูรณ์แล้ว ขั้นต่อไปคือการเขียน firmware code สำหรับ ESP32 AI จะเขียน C++ code ที่ implement แต่ละ feature ตามที่ FSD กำหนดไว้ทีละ phase จากนั้นรัน compiler เพื่อตรวจสอบว่า code ถูกต้อง

ส่วนที่น่าทึ่งที่สุดคือ self-healing compilation loop — เมื่อ compiler ออก error message AI จะอ่าน error นั้น วิเคราะห์สาเหตุ แก้ไข code ที่เกี่ยวข้อง และ compile ใหม่ loop นี้ทำซ้ำจนกว่า compilation จะผ่าน 100% โดยไม่ต้องให้มนุษย์ดูเลย ต่างจาก traditional development ที่ developer ต้องนั่งอ่าน error เอง คิดเอง แล้วแก้เอง ซึ่งใช้เวลาและ mental energy มาก

กระบวนการนี้เร็วกว่า manual debugging มากเพราะ AI ไม่ต้องหยุดคิด ไม่ต้อง search ใน Stack Overflow ไม่ต้องรอให้สมองประมวลผล มัน iterate ด้วยความเร็วที่มนุษย์สู้ไม่ได้ และสามารถลอง approach ต่างๆ ได้หลายแนวทางในเวลาสั้นๆ

Enhancement Loop: Developer อัปเดต FSD → AI Iterate

หลังจาก project ทำงานได้แล้วและ developer ต้องการเพิ่ม feature ใหม่ ขั้นตอนก็ง่ายมาก แค่เพิ่ม requirement ใหม่เข้าใน FSD file — ไม่ว่าจะเป็น test case เพิ่มเติม, feature ใหม่, หรือข้อกำหนดด้าน performance ที่เพิ่มขึ้น

AI อ่าน diff ของ FSD แล้วระบุว่าส่วนไหนของ code ต้องเปลี่ยน implement changes เฉพาะส่วนที่เกี่ยวข้อง ทดสอบอีกครั้ง และ commit/push ไปยัง Git repository โดยอัตโนมัติ ทุก iteration มี commit message ที่อธิบาย change ที่เกิดขึ้น ทำให้ developer track progress ได้ง่าย

ความสวยงามของ pattern นี้คือ FSD กลายเป็น "single source of truth" ที่ทั้ง developer และ AI ใช้อ้างอิงร่วมกัน ถ้ามีความขัดแย้งระหว่าง code กับ spec AI จะยึด spec เสมอ ทำให้ code ตามทัน spec อยู่เสมอ

FSD = Single Source of Truth: ทำให้ AI และมนุษย์พูดภาษาเดียวกัน — developer ลงทุนใน spec ที่ดี AI ลงทุนใน code ที่ถูกต้องตาม spec นั้น
Environment

AI Agent Environment: Docker + Claude Code

การออกแบบ environment ที่ดีคือส่วนสำคัญที่ทำให้ AI ทำงานได้อย่างมีประสิทธิภาพและปลอดภัย Docker container ให้ isolation ที่จำเป็น ในขณะที่ Claude Code ทำหน้าที่เป็น AI brain ที่รับ task และ execute ในแต่ละ project มี container ของตัวเองพร้อม environment ที่ configure ไว้สำหรับ project นั้นโดยเฉพาะ

ทำไมต้อง Isolate ต่อ Project

ในโปรเจกต์ embedded development แต่ละโปรเจกต์มี dependency ที่แตกต่างกันมาก บางโปรเจกต์ต้องการ ESP-IDF version เก่า บางโปรเจกต์ต้องการ Arduino framework บางโปรเจกต์มี Python library ที่อาจชนกับโปรเจกต์อื่น Docker container แก้ปัญหานี้ได้หมดจด แต่ละ container มี filesystem และ environment ของตัวเอง ไม่มีทางที่ dependency ของโปรเจกต์หนึ่งจะไปกระทบอีกโปรเจกต์ได้

superclaude คือ shell alias ที่ถูก configure ไว้ให้ start Claude Code พร้อม environment ที่ครบถ้วนภายใน container — รวมถึง access ไปยัง serial bridge บน Pi Zero, Git credentials, และ environment variables ที่จำเป็นทั้งหมด developer แค่พิมพ์ superclaude ใน terminal ของ container แล้ว Claude Code ก็พร้อมทำงานทันที ไม่ต้องตั้งค่าอะไรเพิ่มเติม

การออกแบบ one container per project ยังมีประโยชน์ด้านการจัดการอีกด้วย developer สามารถ pause โปรเจกต์หนึ่ง หันไปทำอีกโปรเจกต์ แล้วกลับมาทำต่อได้โดยสภาพ environment ยังคงเดิมทุกอย่าง เหมือนการ save state ของงานทั้งหมด

Terminal-Driven Workflow

Claude Code ทำงานใน terminal ทั้งหมด — ไม่มี GUI ไม่มี point-and-click ไม่มี drag-and-drop developer สื่อสารกับ AI ผ่าน text ใน terminal Claude Code ตอบสนองด้วย text และ action ที่มองเห็นได้ทั้งหมด แนวทางนี้มีข้อดีชัดเจนสำหรับ automation: ทุก action ที่ AI ทำสามารถ log ได้, replay ได้, และ automate ได้ง่าย

ใน terminal session หนึ่ง AI มีสิทธิ์เข้าถึงหลายสิ่ง: filesystem ของ container ทั้งหมด, git commands สำหรับ version control, serial port ที่ tunnel มาจาก Pi bridge, internet connection สำหรับ download library หรือ documentation, และ system commands ทั่วไป การมี access ครบถ้วนนี้ทำให้ AI ทำ task ได้โดยไม่ต้องขอ permission ซ้ำๆ ซึ่งช่วยลด friction ของ workflow ลงได้มาก

สำหรับ developer ที่คุ้นเคยกับ pair programming นี่เป็น experience ที่คล้ายกัน แต่ "pair" คนนี้ทำงานได้ 24 ชั่วโมง ไม่เหนื่อย และ iterate ได้เร็วกว่ามนุษย์มาก developer อธิบายสิ่งที่ต้องการ AI ลงมือทำ developer review ผลลัพธ์ แล้ว iterate ต่อ

ข้อควรระวังเรื่อง Security

การให้ AI มี "full env access" ใน container หมายความว่า AI สามารถทำอะไรก็ได้ภายใน container นั้น — อ่านไฟล์ทุกอย่าง, เขียนไฟล์, รัน command ใดๆ, เชื่อมต่อ network ไปยังที่ใดก็ได้ นี่เป็นความสามารถที่จำเป็นสำหรับ autonomous work แต่ก็มาพร้อมกับ risk ที่ต้องตระหนัก

Docker isolation ช่วยจำกัด blast radius ได้อย่างมีนัยสำคัญ ถ้า AI ทำอะไรผิดพลาดใน container ผลเสียจะอยู่แค่ใน container นั้น ไม่กระทบ host system หรือ container อื่น แต่นี่ไม่ได้หมายความว่าควรปล่อยให้ AI ทำงานโดยไม่ดูแล การ review git diff ก่อน push ทุกครั้งเป็นนิสัยที่ควรมีเสมอ เพื่อ verify ว่า code ที่ AI เขียนสมเหตุสมผลและไม่มี security issue แฝงอยู่

นอกจากนี้ควรระวังเรื่อง credentials ภายใน container — อย่าเก็บ API keys หรือ passwords ใน environment variables ที่ AI เข้าถึงได้ โดยไม่จำเป็น ใช้ Docker secrets หรือ vault pattern แทน ถ้า project มี sensitive data ควรพิจารณาใช้ read-only file system สำหรับส่วนที่ไม่จำเป็นต้องแก้ไข

Best Practice: Review git diff ก่อน push เสมอ — AI อาจเขียน code ที่ทำงานได้แต่มี logic ที่ไม่ตรงกับ intent จริงๆ การ review เป็น safety net ที่สำคัญ
Docker + Claude Code: sandbox ที่ให้ AI ทำงานได้เต็มที่โดยไม่กระทบ host system — isolate ต่อ project, superclaude shortcut สำหรับ full env access, review diff ก่อน push เสมอ
Flash & Test — Intuition

Autonomous Flash คืออะไร

คำว่า "autonomous flash" หมายความว่า AI สามารถ flash firmware เข้า ESP32 board ได้เองโดยที่ developer ไม่ต้องแตะ physical hardware เลย ไม่ต้องเสียบสาย ไม่ต้องเปิด serial monitor ไม่ต้องรอดู boot sequence เอง ทั้งหมดนี้เกิดขึ้นโดย AI ผ่าน Pi Zero bridge ที่ทำหน้าที่เป็น "remote hands"

Flash แบบ Manual vs. Automatic เปรียบเทียบ

ใน workflow แบบ manual เมื่อ developer แก้โค้ดเสร็จต้องทำขั้นตอนต่อไปนี้ทุกครั้ง: บันทึกไฟล์, เปิด terminal ใหม่, ตรวจสอบว่า ESP32 เชื่อมต่อ port อะไร, รัน esptool.py --port /dev/ttyUSB0 ..., รอ 15-30 วินาที, เปิด serial monitor, ดู boot log, ตรวจสอบว่าทำงานถูกต้อง แล้วกลับไปแก้โค้ดต่อ ถ้าต้องทำแบบนี้ 50-100 รอบใน session เดียว เวลาที่เสียไปสะสมได้อย่างมาก

ใน workflow แบบ automatic AI รัน command เดียวผ่าน serial bridge — Pi Zero รับ command แล้วส่งต่อไปยัง ESP32 จริงๆ ใช้เวลา flash เท่ากัน แต่ AI ไม่ต้องรอ context ใดๆ ทำได้ทันทีที่ compile ผ่าน จากนั้น AI เปิด serial monitor อ่าน boot log วิเคราะห์ output และ report ผลได้เองโดยอัตโนมัติ developer ได้รับ summary ของผลการ flash โดยไม่ต้องทำอะไรเพิ่มเติม

เปรียบเทียบกับ lab assistant ที่คอยทำงาน manual ให้ตลอดเวลา — developer ออกแบบ experiment แล้วส่งให้ assistant ทำ assistant รายงานผล developer วิเคราะห์ผลแล้วออกแบบ experiment ต่อไป นี่คือ model ที่ embedded AI workbench ใช้อยู่ และเป็น model ที่นักวิทยาศาสตร์และวิศวกรรู้จักดีในบริบทของ laboratory automation

Serial Monitoring อัตโนมัติ

หลังจาก flash เสร็จ AI เปิด serial monitor เพื่อดู boot sequence ของ ESP32 ทันที การดู boot log สำคัญมากเพราะ ESP32 บางครั้งอาจ panic หรือ crash ระหว่าง boot ซึ่งบ่งชี้ว่ามีปัญหาใน firmware แต่ปัญหานี้จะไม่ปรากฏใน compile log — ต้องรัน firmware จริงๆ บน hardware จริงๆ ถึงจะเห็น

เมื่อ AI เห็น panic message หรือ error ใน serial log มันจะ parse log นั้น ระบุสาเหตุ (เช่น stack overflow, null pointer dereference, hardware initialization failure) แล้วกลับไปแก้ code ที่เกี่ยวข้อง compile ใหม่ flash ใหม่ ดู log ใหม่ ทั้งหมดนี้เกิดในลูปอัตโนมัติจนกว่า boot sequence จะสมบูรณ์ไม่มี error

ความเร็วของ loop นี้คือจุดที่สร้างความแตกต่างที่ยิ่งใหญ่ที่สุด ใน manual workflow developer อาจใช้ 5-10 นาทีต่อ iteration (คิด + แก้ + flash + ดู log) แต่ AI iteration อาจเหลือแค่ 1-2 นาที เมื่อมี bug ที่ต้องแก้หลายรอบ ความแตกต่างนี้จะขยายเป็นชั่วโมงที่ประหยัดได้

Autonomous Flash Loop: developer เขียนแค่ spec, AI เป็นคนกด flash และอ่าน log เอง — feedback loop จาก หลายนาที เหลือ ไม่กี่สิบวินาที ต่อ iteration

หนึ่งในคุณสมบัติที่โดดเด่นที่สุดของ workbench คือความสามารถในการทดสอบ edge cases ที่เกิดขึ้นจริงในสภาวะ IoT จริงๆ ได้ — ไม่ใช่แค่ simulation บน desktop Pi Zero มีความสามารถในการสร้าง WiFi Access Point, kill MQTT broker, ตัด network connection, และ monitor serial output พร้อมกัน ทำให้ test cases สะท้อน real-world condition ได้อย่างแท้จริง

การจำลอง Network Failure

IoT device ในโลกจริงต้องทำงานในสภาวะที่ network ไม่เสถียร WiFi อาจหาย, MQTT broker อาจ restart, power อาจดับ ถ้า firmware ไม่ handle กรณีเหล่านี้ได้อย่างถูกต้อง device จะ hang หรือ crash โดยไม่ recover การทดสอบกรณีเหล่านี้บน desktop simulator มักไม่สมบูรณ์เพราะ simulator ไม่สามารถ reproduce ความซับซ้อนของ network stack จริงๆ ได้ครบถ้วน

workbench แก้ปัญหานี้ได้อย่างตรงไปตรงมา Pi Zero สามารถสร้าง WiFi AP ได้ด้วย hostapd ทำให้ ESP32 connect เข้ามาได้เหมือน real WiFi network จากนั้น AI สั่งให้ Pi ปิด AP กลางอากาศ และ monitor ดูว่า ESP32 มี behavior อะไร: reconnect ได้ภายในกี่วินาที, มี error message ที่ถูกต้องใน serial log หรือไม่, state ของ application ยังคงสมบูรณ์หรือเปล่า

ในทำนองเดียวกัน AI สั่ง kill MQTT broker process บน Pi แล้วดูว่า ESP32 handle การขาด connection ของ broker ได้อย่างไร ทั้งหมดนี้เป็น automated test ที่ AI รันเองโดยไม่ต้องให้ developer มานั่งปิด router เอง หรือ kill process เอง

ตัวอย่าง Test Cases จาก iOS Keyboard Project

ในโปรเจกต์ iOS Voice Keyboard test cases ที่ AI รันอัตโนมัติครอบคลุม scenarios ที่สำคัญทั้งหมด:

TC001 — BLE Pairing

BLE Pairing Failure Recovery

จำลอง pairing ล้มเหลว (signal weak / timeout) แล้วดูว่า ESP32 retry หรือ fallback gracefully ไม่ hang อยู่ใน pairing state ตลอดไป

TC002 — Unicode

Thai Character Transmission

ส่งอักขระ Unicode ภาษาไทย (เช่น U+0E01 ก) ผ่าน BLE → ตรวจสอบว่า USB HID report ส่ง keycode ที่ถูกต้องไปยัง host computer

TC003 — USB Disconnect

Host Computer Disconnect

host computer ปิด USB port กลางอากาศ — ESP32 ต้องตรวจจับ disconnect และ reconnect อัตโนมัติเมื่อ USB กลับมา ไม่ค้างอยู่ใน error state

TC004 — Multi-Language

Language Switch

สลับระหว่าง Thai → English → Thai ผ่าน BLE command — ตรวจสอบว่า HID keymap เปลี่ยนตามและ output ถูกต้องทั้งสองภาษา

AI รัน test ทั้งหมดเหล่านี้โดยอัตโนมัติผ่าน Pi workbench ผลลัพธ์ถูก log ไว้และ summary ถูกส่งกลับมาให้ developer อ่าน ถ้า test ใด fail AI จะ loop กลับไป debug แก้โค้ด flash ใหม่ แล้ว rerun test จนผ่านทั้งหมด

ทำไม Realistic Testing สำคัญสำหรับ IoT

IoT devices ทำงานใน environment ที่ไม่แน่นอนอย่างยิ่ง — network ขาดโดยไม่มีการเตือน, power อาจหายกะทันหัน, temperature เปลี่ยน, signal strength ผันผวน device ที่ทำงานได้ดีใน controlled environment ของ lab อาจ fail ทันทีที่ deploy จริงถ้าไม่ได้ test กรณีเหล่านี้ไว้ก่อน

simulator บน desktop สร้าง happy path test ได้ดี แต่ fail ในการจำลอง real-world chaos เพราะ network stack จริงๆ มีความซับซ้อนและ timing behavior ที่ simulator ไม่สามารถ reproduce ได้ครบถ้วน การทดสอบบน real hardware ใน real network environment จึงเป็น gold standard ที่ workbench นี้ทำได้

ยิ่งไปกว่านั้น workbench ให้ทดสอบเหล่านี้รันอัตโนมัติทุก iteration โดยไม่ต้องตั้งค่าเพิ่มเติม ทุกครั้งที่ code เปลี่ยน test suite ทั้งหมดรันใหม่อัตโนมัติ นี่คือสิ่งที่ CI/CD ทำสำหรับ web development แต่สำหรับ embedded hardware ซึ่งปกติยากกว่ามาก

Realistic IoT Testing: workbench ทดสอบ error condition จริงๆ (network failure, power loss simulation, reconnect recovery) — ไม่ใช่แค่ happy path ที่ simulator ทำได้
Demo Project

Demo: iOS Voice Keyboard ใน 1.5 ชั่วโมง

โปรเจกต์ iOS Voice Keyboard คือหลักฐานที่จับต้องได้ชิ้นสำคัญที่สุดของ workbench นี้ มันแสดงให้เห็นว่าในเวลาเพียง 1.5 ชั่วโมง AI สามารถสร้าง product ที่ซับซ้อนพอสมควรได้โดยสมบูรณ์ ตั้งแต่ BLE pairing, Unicode text handling, ไปจนถึง USB HID keyboard emulation รองรับหลายภาษา

แนวคิดและ Motivation

Dictation software ที่ดีอย่าง Dragon NaturallySpeaking มีราคาแพงมากและมักจำกัดอยู่กับ platform เฉพาะ Apple Dictation ใน macOS ดีพอสมควรแต่ใช้ได้แค่กับ Mac เท่านั้น สำหรับ Windows หรือ Linux มีทางเลือกน้อยมากที่รองรับหลายภาษาได้ดี โดยเฉพาะภาษาไทยซึ่งมักเป็นปัญหาใหญ่ใน speech recognition บน desktop

แนวคิดง่ายมาก: smartphone ปัจจุบันมี speech-to-text engine ที่ยอดเยี่ยมในตัวอยู่แล้ว ทั้ง iOS Siri และ Android มีความแม่นยำสูงและรองรับหลายภาษา แทนที่จะพยายาม implement speech recognition ใหม่บน desktop ให้ใช้สิ่งที่มีอยู่แล้ว — รับ text ที่ transcribe แล้วจาก phone แล้วส่งไปยัง computer ผ่าน ESP32 ที่ทำหน้าที่เป็น USB keyboard

ผลลัพธ์คือ keyboard ที่ทำงานได้กับทุก OS (Windows, Linux, Mac) รองรับทุกภาษาที่ iOS รองรับ และไม่ต้องลง driver ใดๆ เพราะ host computer เห็น ESP32 เป็น standard USB HID keyboard device ที่ทุก OS รู้จักอยู่แล้ว

Technical Architecture

Architecture ของโปรเจกต์นี้แบ่งออกเป็นสามส่วนหลักที่ทำงานร่วมกัน:

Layer 1

iPhone App

iOS app ใช้ SFSpeechRecognizer framework รับเสียงจาก microphone แปลงเป็น text แบบ real-time จากนั้นส่ง text string ผ่าน BLE characteristic ไปยัง ESP32 ที่ advertise อยู่

Layer 2

ESP32 (BLE + HID)

ESP32-S3 รับ BLE data จาก iPhone แปลง Unicode text string เป็น USB HID keyboard report (scan codes) แล้วส่ง report ไปยัง host computer ผ่าน USB native interface

Layer 3

Host Computer

Windows/Linux/Mac เห็น ESP32 เป็น standard USB keyboard — ไม่ต้องลง driver ใดๆ text จาก voice ปรากฏใน active window เหมือนพิมพ์จาก keyboard จริงๆ

ส่วนที่ซับซ้อนที่สุดคือการ convert Unicode text ให้เป็น HID keyboard reports เพราะ Unicode character หนึ่งตัวอาจต้องใช้ HID keycode หลายตัวร่วมกัน (modifier keys + key codes) โดยเฉพาะสำหรับอักขระ Unicode ที่ไม่ได้อยู่ใน Basic ASCII range อย่างตัวอักษรไทย AI เขียน mapping table ที่ครอบคลุมกรณีเหล่านี้ไว้ใน firmware

เวลาที่ใช้และ Role ของ AI Workbench

โปรเจกต์ทั้งหมดสร้างเสร็จภายใน ~1.5 ชั่วโมง ซึ่งถือว่าน้อยมากเมื่อเทียบกับความซับซ้อนของ project นี้ใน breakdown ของเวลา ส่วนใหญ่ใช้ไปกับการที่ AI อ่าน FSD, เขียน BLE code, เขียน HID code, debug, compile, flash, และทดสอบ developer ใช้เวลาจริงๆ แค่ในการเขียน idea.md และ review ผลลัพธ์

workbench flash firmware ใหม่ทุกครั้งที่ AI แก้โค้ดโดย developer ไม่ต้องแตะ ESP32 board เลยแม้แต่ครั้งเดียวตลอดกระบวนการพัฒนา 1.5 ชั่วโมง ทุก compilation, ทุก flash, ทุก serial log reading เกิดขึ้นใน terminal ที่ AI ควบคุม

ถ้าทำโปรเจกต์เดียวกันด้วย manual workflow ตามปกติ การ setup BLE stack, เขียน HID descriptor, debug Unicode mapping อาจใช้เวลา 2-3 วัน ไม่รวมเวลาที่เสียกับการ flash manual ทุก iteration และเปิด serial monitor ดู log เอง ความแตกต่างของเวลา 1.5 ชั่วโมง vs. หลายวัน คือ proof ที่ชัดเจนที่สุดของ value ที่ workbench นี้สร้าง

Proof of Concept: iOS Voice Keyboard สร้างเสร็จใน 1.5 ชั่วโมง รองรับภาษาไทยและภาษาอังกฤษ ทำงานได้กับ Windows และ Linux โดยไม่ต้องลง driver — developer แค่เขียน idea.md
ใน 1.5 ชั่วโมง: AI สร้าง multi-language voice keyboard ที่ทำงานข้าม OS ได้ โดย developer แค่เขียน idea — โปรเจกต์เดียวกันแบบ manual ใช้เวลา 2-3 วัน
Investment

ต้นทุนและการตั้งค่าครั้งแรก

ก่อนตัดสินใจสร้าง workbench นี้ควรเข้าใจ investment ที่ต้องการทั้งด้านเวลาและเงิน แม้ว่า hardware จะถูกมากเมื่อเทียบกับ professional test equipment แต่มี cost ที่มักถูกมองข้ามในส่วนของ software subscription และเวลา setup ครั้งแรก การวางแผน investment เหล่านี้ล่วงหน้าจะช่วยให้เห็น ROI ที่ชัดเจน

Hardware Costs

Raspberry Pi Zero 2W ราคาอยู่ที่ประมาณ $15 USD ซึ่งถือว่าถูกมากสำหรับ single-board computer ที่รัน Linux ได้เต็มๆ มีประสิทธิภาพเพียงพอสำหรับ serial forwarding และ network bridging ที่ workbench ต้องการ ข้อเสียเดียวคือ availability อาจผันผวนขึ้นอยู่กับ chip shortage สถานการณ์

USB Hub แบบ powered (มี DC adapter ของตัวเอง) สำคัญมาก ราคาอยู่ที่ $10-20 USD สำหรับ 4-7 port ต้องเป็นแบบ powered เพราะ Pi Zero ไม่สามารถจ่ายไฟให้ ESP32 หลายตัวพร้อมกันได้เพียงพอ USB cables สำหรับแต่ละ ESP32 ต้องเป็น data cable ราคารวมประมาณ $5-10 USD สำหรับ 3-4 เส้น

รวม hardware ทั้งหมดอยู่ที่ประมาณ $30-50 USD สำหรับ workbench ที่รองรับ 3-4 ESP32 boards พร้อมกัน ถ้าเทียบกับ commercial hardware testing equipment ที่ราคาเป็นหลักพันหรือหลักหมื่น USD นี่ถือว่า affordable มากๆ และถ้ามี ESP32 board อยู่แล้วก็ไม่ต้องซื้อเพิ่ม

Software Costs

Claude Code เป็น component ที่มี cost ต่อเนื่อง ต้องมี Anthropic API key หรือ Claude Pro subscription เพื่อใช้งาน ค่าใช้จ่ายขึ้นอยู่กับ usage แต่สำหรับ developer ที่ทำ IoT project บ่อยๆ cost นี้มักคุ้มค่ามากเมื่อเทียบกับเวลาที่ประหยัดได้ในแต่ละ project

ส่วน software อื่นๆ เป็น open source ทั้งหมด: Proxmox (hypervisor, free), Docker (free), esptool (free), Python serial libraries (free), socat (free, มีใน repository ทุก Linux distro) ซึ่งหมายความว่า ongoing cost หลักคือ Claude Code subscription เท่านั้น

วิธีคิด ROI ง่ายๆ: ถ้า Claude Code ช่วยประหยัดเวลา 1 วันต่อ project และทำ project เดือนละ 2-3 project ค่า subscription ถือว่าคุ้มค่ามากเมื่อเทียบกับ opportunity cost ของเวลาที่ใช้ไปกับ manual flash และ debug

Initial Setup Time Investment

การตั้งค่า Pi Zero 2W ครั้งแรกใช้เวลาประมาณ 2-4 ชั่วโมง รวมถึง: ติดตั้ง Raspberry Pi OS, configure network (Ethernet + static IP), ติดตั้ง socat และ Python serial libraries, ทดสอบ serial forwarding กับ ESP32 board จริงๆ เพื่อยืนยันว่าระบบทำงาน ขั้นตอนนี้ทำครั้งเดียวแล้วไม่ต้องทำซ้ำ

การสร้าง FSD SOP AI skill ครั้งแรกใช้เวลาประมาณ 1-2 ชั่วโมง skill นี้เป็น custom prompt template ที่บอก Claude Code ว่าต้องอ่าน idea.md และ generate FSD ในรูปแบบที่กำหนด เมื่อสร้างเสร็จแล้วใช้ได้กับทุก project ไม่ต้องแก้ไขอีก

การตั้งค่า Docker environment ใช้เวลาประมาณ 1 ชั่วโมง รวมถึง: สร้าง base Docker image ที่มี toolchain ของ ESP32 (ESP-IDF หรือ Arduino framework), configure network access ไปยัง Pi bridge, สร้าง superclaude alias ใน container รวมเวลา setup ทั้งหมดประมาณ 4-8 ชั่วโมง ลงทุนครั้งเดียว ใช้ได้ตลอดไป

ไม่จำเป็น! Proxmox เป็นแค่ hypervisor ที่ผู้สร้าง workbench นี้ใช้อยู่เอง คุณสามารถใช้ VirtualBox, VMware, Hyper-V หรือแม้แต่ Docker บน Windows/Mac โดยตรงก็ได้ สิ่งสำคัญคือ Claude Code ต้องเข้าถึง network ที่ Pi Zero อยู่ได้ ซึ่งทำได้ผ่าน bridge network ใน Docker หรือ VM

ไม่ต้องใช้เลย! ถ้า Claude Code รันโดยตรงบน machine ที่มี USB port และมองเห็น ESP32 เป็น /dev/ttyUSB0 ได้โดยตรง ก็ไม่จำเป็นต้องมี Pi bridge เลย Pi bridge ถูกออกแบบมาเพื่อแก้ปัญหา serial passthrough ของ VM โดยเฉพาะ

ขึ้นอยู่กับ usage pattern ถ้าทำ IoT project บ่อยๆ และแต่ละ project ประหยัดเวลาได้ 1-3 วัน ค่า subscription คุ้มค่ามากอย่างแน่นอน แต่ถ้าทำ project นานๆ ครั้ง อาจลอง API pay-per-use แทน subscription ก็ได้ เริ่มจาก project เล็กๆ แล้ววัด ROI จริงๆ ก่อนตัดสินใจ

ต้องรู้พื้นฐาน embedded dev พอสมควร — เช่น รู้ว่า ESP32 คืออะไร, รู้จัก serial port, รู้จัก esptool, เข้าใจ concept ของ BLE หรือ WiFi ในระดับหนึ่ง AI จะทำ coding และ debugging ให้ แต่ developer ยังต้องเขียน idea ที่มี technical direction ที่ถูกต้อง และ review ผลลัพธ์ได้ ไม่ใช่ zero-knowledge tool

Investment Summary: hardware ~$30-50 USD (ครั้งเดียว) + setup ~4-8 ชั่วโมง (ครั้งเดียว) + Claude Code subscription (ต่อเนื่อง) — คืนทุนตั้งแต่ project แรกที่ประหยัดได้ 1-2 วัน

1. Serial Bridge Architecture
workbench = Pi Zero เป็น serial bridge ให้ AI ใน VM เข้าถึง ESP32 ผ่าน network — แก้ปัญหา serial passthrough ของ Proxmox ได้อย่างหรูหรา

custom AI skill ที่แปลง idea Markdown → structured spec อัตโนมัติ ทำให้ developer ลงทุนเวลาใน "อยากได้อะไร" แทนที่จะเสียเวลากับ "เขียนยังไง"

AI flash + test + iterate เองโดยไม่มีมนุษย์เข้ามาแทรก — ตัด context switch ที่เป็น cost ใหญ่ที่สุดของ embedded dev ออกไปได้ทั้งหมด

7. Limitations ที่ต้องรู้
Claude Code เป็น paid service, initial setup 4-8 ชั่วโมง, Pi bridge จำเป็นเฉพาะกรณี VM — ไม่ใช่ magic solution ที่ไร้ cost
