Overview

RTK-GPS คืออะไร และ DePIN เปลี่ยนอะไร

GPS ทั่วไปที่ใช้ในมือถือให้ความแม่นยำระดับ 1–5 เมตร ซึ่งเพียงพอสำหรับการนำทางรถยนต์ แต่ไม่เพียงพอสำหรับหุ่นยนต์ โดรน หรือการสำรวจที่ดิน RTK-GPS แก้ปัญหานี้ด้วยการวัด "phase" ของคลื่นพาหะแทนที่จะวัดแค่เวลาของสัญญาณ ทำให้ความแม่นยำกระโดดจาก ±5 เมตรสู่ ±1–2 เซนติเมตร DePIN (Decentralized Physical Infrastructure Networks) เข้ามาเปลี่ยนโมเดลธุรกิจ ทำให้ใครก็ตามสามารถสร้าง Base Station ของตัวเอง แบ่งปัน Correction Data และรับ Crypto Token เป็นรางวัล แทนที่จะต้องพึ่งบริการจากบริษัทพาณิชย์ที่มีค่าใช้จ่ายสูง

เทคโนโลยีหลัก

Carrier Phase RTK

วัด phase ของคลื่น L1/L2 (wavelength ~20 cm) แทนการวัด timing เพียงอย่างเดียว ทำให้ได้ความแม่นยำระดับ sub-centimeter

โครงสร้างพื้นฐาน

Base + Rover + Internet

Base Station ที่รู้ตำแหน่งแน่นอนคำนวณ Error แล้วส่ง Correction ผ่าน Internet (NTRIP protocol) ให้ Rover แก้ไข Ambiguity

โมเดลธุรกิจใหม่

DePIN Tokenomics

ผู้ run Base Station รับ Crypto (ONO จาก Onocoy หรือ GEOD จาก GEODNET) เป็น Reward ตามคุณภาพ Signal ที่ส่งออก

💡 Core insight: RTK แก้ปัญหา "Integer Ambiguity" — ไม่รู้ว่า Rover อยู่บน wave ที่เท่าไหร่ — ด้วยการเปรียบเทียบกับ Base Station ที่รู้ตำแหน่งแน่นอน การทำ DIY ด้วย ESP32 + u-blox ZED-F9P ทำให้ต้นทุนลดจาก $700+ เหลือต่ำกว่า $300
★ สาระสำคัญ: RTK = GPS + Correction Signal จาก Base → ±1 cm; DePIN = Base Station ที่สร้างรายได้ crypto

Section 1 — Deep-Dive Protocol · ส่วนประกอบ

ส่วนประกอบหลักของระบบ RTK-GPS และ DePIN

ก่อนลงลึกในรายละเอียด ส่วนนี้แผนที่ส่วนประกอบทั้งหมดของระบบ RTK-DePIN เพื่อให้ผู้อ่านเห็นภาพรวมก่อนว่าแต่ละชิ้นส่วนทำหน้าที่อะไร และเชื่อมต่อกันอย่างไร ความเข้าใจโครงสร้างตั้งแต่ต้นจะช่วยให้ทุก section ที่ตามมามีความหมายมากขึ้น

ส่วนประกอบหลัก — ฝั่งการรับสัญญาณ

Component 1 — Signal Source

ดาวเทียม GNSS

ส่งคลื่นพาหะ L1 (1575.42 MHz) และ L2 (1227.60 MHz) มา Earth โดยมี wavelength ~20 cm แต่ละดาวเทียมส่ง timing code + navigation message + carrier wave พร้อมกัน

Component 2 — Standard GPS

Pseudorange Measurement

วัดเวลาที่สัญญาณเดินทางจากดาวเทียมถึง Receiver แล้วคูณด้วยความเร็วแสง ได้ระยะห่าง ±1–5 m ขึ้นอยู่กับ atmospheric delay, multipath, clock error

Component 3 — RTK Layer

Carrier Phase Measurement

วัด phase ของ carrier wave (ตำแหน่งบนคลื่น) ซึ่งมี resolution ที่สูงกว่าการวัด timing มาก แต่มีปัญหา "Integer Ambiguity" ที่ต้องแก้ไขก่อนจึงจะใช้งานได้จริง

Component 4 — Reference Point

Base Station

เครื่องรับ GNSS ที่ตำแหน่งทราบแน่นอน (survey-grade หรือ calibrated) คำนวณความแตกต่างระหว่างตำแหน่งที่ "ควรจะเป็น" กับที่ "วัดได้" แล้วส่ง Correction Data ต่อเนื่องทุก 1 วินาที

Component 5 — Mobile Receiver

Rover

เครื่องรับ GNSS ที่เคลื่อนที่ได้ รับ Correction Data จาก Base แล้วใช้ algorithm (LAMBDA) แก้ไข Integer Ambiguity จนได้ตำแหน่งที่แม่นยำ ±1–2 cm ภายใน 5–60 วินาที

Component 6 — Transport Layer

NTRIP / Internet

Protocol สำหรับส่ง RTCM3 Correction Messages ผ่าน HTTP/TCP ทำให้ Base Station และ Rover ไม่ต้องอยู่ใกล้กัน แค่มี Internet ก็ส่ง Correction ข้ามทวีปได้

ส่วนประกอบหลัก — ฝั่ง DePIN

Component 7 — Network Layer

Correction Marketplace

บริการอย่าง Onocoy (token: ONO) และ GEODNET (token: GEOD) รวบรวม Correction Data จาก Base Station ทั่วโลก แล้วขายให้ผู้ใช้ปลายทาง (enterprise, drones, robots)

Component 8 — Incentive Layer

Token Rewards

Operator ที่ run Base Station รับ Crypto Token ตามคุณภาพ Signal ที่ส่งออก: จำนวน satellite bands, uptime, geographic coverage — ยิ่งคุณภาพสูงยิ่งได้ reward มาก

แผนผังระบบ: ดาวเทียม → Base Station คำนวณ Error → ส่งผ่าน NTRIP/Internet → Rover แก้ Ambiguity → ±2 cm

★ ภาพรวม: RTK ต้องการ 2 receiver (Base + Rover) + Internet — Base คำนวณ error แทน Rover ทำให้ Rover ได้ตำแหน่งแม่นยำโดยไม่ต้องซับซ้อนเอง

Section 1 — Deep-Dive Protocol · สมมติฐาน & Takeaways

สิ่งที่ซ่อนอยู่และ Insights สำคัญ

สมมติฐานที่ซ่อนอยู่ (Implicit Assumptions)

สิ่งเหล่านี้คือสิ่งที่ระบบ RTK-GPS ถือว่าผู้ใช้รู้อยู่แล้ว แต่คนเริ่มต้นมักไม่ทราบและส่งผลให้เกิดความสับสนหรือผลลัพธ์ที่ไม่คาดคิด

  • สมมติฐานที่ 1 — Base ต้องอยู่ใกล้ Rover ≤ 20 km: Correction Data ที่ Base ส่งมาจะ "ถูกต้อง" สำหรับ Rover ก็ต่อเมื่อทั้งคู่อยู่ในสภาพ atmospheric error ที่คล้ายกัน ถ้าห่างเกิน 20 km ค่า ionospheric delay จะต่างกันมากจน Correction ไม่แม่นยำ
  • สมมติฐานที่ 2 — Base Station ต้องทราบตำแหน่งของตัวเองก่อน: ถ้า Base ไม่รู้ตำแหน่งของตัวเองอย่างแม่นยำ (survey-in process อาจใช้เวลา 1–24 ชั่วโมง) ความผิดพลาดจะถูกส่งต่อไปยัง Rover ทั้งหมด ทำให้ถึงแม้ Rover จะได้ตำแหน่งที่ "เสถียร" แต่ก็อาจ offset จากตำแหน่งจริง
  • สมมติฐานที่ 3 — Multi-band antenna ≠ ทางเลือก แต่คือความจำเป็น: การรับสัญญาณ L1 เพียง band เดียวสามารถ resolve ambiguity ได้ช้ากว่ามาก (หรือล้มเหลวในสภาพ sky ที่บังบาง) multi-band (L1+L2+L5) ให้ redundancy และความเร็วในการ fix สูงกว่ามาก

Key Takeaways — เรียงจาก High-level → Actionable

Takeaway 1 — Big Picture

RTK วัด Phase ไม่ใช่แค่ Timing

GPS ทั่วไปวัดว่าสัญญาณใช้เวลานานเท่าไหร่ RTK วัดว่าสัญญาณอยู่ที่ "ตำแหน่งไหนบนคลื่น" ซึ่งมี resolution สูงกว่าเป็นร้อยเท่า นั่นคือความแตกต่างระหว่าง ±5 m กับ ±2 cm

Takeaway 2 — Mental Model

Base = เพื่อนที่รู้ตัวเองว่าอยู่ที่ไหน

เปรียบ Base Station เหมือนเพื่อนที่ยืนอยู่ตรงจุดที่รู้แน่นอน แล้วบอกว่า "GPS บอกว่าฉันอยู่ผิดไป 3 เมตร" Rover ได้ยินแล้วก็รู้ว่าตัวเองน่าจะผิดไปในทิศทางเดียวกัน จึงแก้ไขได้ถูกต้อง

Takeaway 3 — Common Trap

ราคา Hardware ≠ ความแม่นยำ

Base Station ราคา $700+ จาก Leica หรือ Trimble ไม่ได้แม่นยำกว่า DIY ESP32 + ZED-F9P อย่างมีนัยสำคัญ ความแตกต่างอยู่ที่ ruggedness และ support เท่านั้น สำหรับ hobbyist/maker DIY เพียงพอ

Takeaway 4 — Actionable

DePIN เปลี่ยน CAPEX เป็น Revenue Stream

แทนที่ Base Station จะเป็น "ค่าใช้จ่าย" ที่ต้องจ่ายเพื่อใช้ RTK เอง การเชื่อมต่อกับ Onocoy/GEODNET เปลี่ยนมันเป็น "แหล่งรายได้" ที่ได้ Token ขณะที่ไม่ได้ใช้งาน

💡 Takeaway 5–7:
  • T5: Fix time (TTFF) คือ 5–60 วินาทีหลัง startup — ไม่ใช่ instant ต้องรอให้ Ambiguity resolve ก่อนจึงได้ cm accuracy
  • T6: Selling RTK correction data คือธุรกิจมูลค่า multi-million dollar แล้วตอนนี้ — DePIN democratize การเข้าถึงตลาดนี้
  • T7: Signal quality = reward: ยิ่งมี multi-band antenna + clear sky view + stable internet ยิ่งได้ token มาก เพราะ network ให้ reward ตาม contribution quality

Section 2 — RTK-GPS พื้นฐาน · สัญชาตญาณ

ทำไม GPS ทั่วไปถึงไม่แม่นยำ และ RTK แก้ปัญหานี้อย่างไร

เพื่อเข้าใจว่า RTK ทำงานอย่างไร ต้องเริ่มจากเข้าใจก่อนว่า GPS ทั่วไปมีข้อจำกัดอะไร สัญญาณ GPS เดินทางจากดาวเทียมสู่โลกผ่านชั้น ionosphere และ troposphere ซึ่งทำให้สัญญาณช้าลงและเบี่ยงเบน ยิ่งไปกว่านั้น clock ของดาวเทียมและ receiver ก็ไม่เคย sync กันสมบูรณ์ 100% ความผิดพลาดเหล่านี้รวมกันทำให้ความแม่นยำอยู่ที่ ±1–5 เมตร ไม่ใช่ความผิดพลาดของอุปกรณ์ แต่เป็น fundamental limit ของวิธีการวัด

GPS ทั่วไป: วัดเวลาเพื่อคำนวณระยะทาง

GPS receiver ทำงานโดยรับสัญญาณจากดาวเทียมอย่างน้อย 4 ดวง แต่ละดาวเทียมส่ง "timestamp" บอกว่าสัญญาณออกเดินทางเมื่อไหร่ receiver วัดว่าสัญญาณมาถึงเมื่อไหร่ แล้วคูณด้วยความเร็วแสง (300,000 km/s) เพื่อได้ระยะทาง วิธีนี้เรียกว่า pseudorange measurement

ปัญหาคือ "เวลา" ที่วัดได้นั้นไม่แม่นยำสมบูรณ์ แสงเดินทางช้าลง ~240 ns เมื่อผ่านชั้น ionosphere และอีก ~7 ns เมื่อผ่าน troposphere ซึ่งแปลเป็นความผิดพลาดของระยะทางหลาย meter ไม่มีทางรู้ว่าวันนี้ ionosphere หนาแค่ไหนโดยไม่มีข้อมูลอ้างอิง นั่นคือที่มาของ ±5 m error

GPS ทั่วไป: Satellite ส่ง timestamp → ผ่าน Ionosphere ที่ทำให้ช้าลง → Receiver วัดเวลา → ตำแหน่งมีความผิดพลาด ±1–5 m

RTK: วัด Phase แทนการวัดแค่เวลา

GPS receiver ที่รองรับ RTK ไม่ได้วัดแค่ว่าสัญญาณมาถึงเมื่อไหร่ แต่วัดด้วยว่าสัญญาณ "อยู่ที่ตำแหน่งไหนบนคลื่น" ณ ขณะนั้น เหมือนกับที่คุณสามารถบอกได้ไม่เพียงแค่ว่าเส้นไหนบนไม้บรรทัดที่ตรงกับขอบกล่อง แต่ยังบอกได้ว่าอยู่ที่ fraction ไหนของช่วงระหว่างขีด

Carrier wave ของ L1 มี wavelength 19.05 cm สามารถวัด phase ได้ด้วยความแม่นยำ ~1% ของ wavelength หมายความว่าความแม่นยำของการวัด phase เพียงอย่างเดียวอยู่ที่ ~2 mm ดีกว่า pseudorange มาก แต่มีปัญหาหนึ่งที่ต้องแก้ก่อน: Integer Ambiguity

เจาะลึก: Integer Ambiguity คืออะไร และทำไมถึงแก้ยาก?

สมมติว่า Rover รับสัญญาณจากดาวเทียมดวงหนึ่งซึ่งอยู่ห่าง 20,000 km receiver วัดได้ว่า carrier wave อยู่ที่ phase angle 0.37 cycle แต่ระยะ 20,000 km นั้นมี carrier wave กี่รอบ? คำตอบคือประมาณ 100 ล้านรอบ แต่ receiver ไม่รู้ว่ารอบที่เท่าไหร่กันแน่ — รู้แค่ว่า fractional phase คือ 0.37

ตัวเลขจำนวนเต็มของรอบที่ไม่รู้นี้คือ "Integer Ambiguity" (N) ถ้าแก้ได้ว่า N = 100,000,000 แล้ว phase measurement จะให้ความแม่นยำ sub-centimeter แต่ถ้าเดาผิดแม้แต่ 1 cycle จะ offset ตำแหน่งไป 19 cm ทันที

RTK แก้ปัญหานี้ด้วย LAMBDA algorithm (Least-squares AMBiguity Decorrelation Adjustment) ซึ่งใช้ data จากหลายดาวเทียมพร้อมกัน และ Correction Data จาก Base Station เพื่อ narrow down ว่า N ที่เป็นไปได้มากที่สุดคือเท่าไหร่ โดยทั่วไปใช้เวลา 5–60 วินาที

★ Beginner takeaway: RTK ไม่ได้ "แก้" GPS — มันเพิ่ม layer ใหม่ที่วัด phase ของคลื่น แล้วใช้ข้อมูลจาก Base Station ที่รู้ตำแหน่งแน่นอนเพื่อแก้ค่า error ร่วมกัน

Section 2 — RTK-GPS พื้นฐาน · กลไก

กระบวนการทำงานของ RTK ทีละขั้นตอน

เมื่อเข้าใจ "ทำไม" แล้ว ส่วนนี้อธิบาย "อย่างไร" — กระบวนการที่เกิดขึ้นจริงในทุก epoch (1 วินาที) ตั้งแต่ Base Station วัดสัญญาณ ส่ง Correction ผ่าน Internet จนถึง Rover คำนวณตำแหน่ง ±2 cm

ขั้นตอนการทำงาน RTK ทุก Epoch

  1. 1
    Base Station วัด Error

    Base รับสัญญาณจากดาวเทียมเหมือนกัน แล้วเปรียบเทียบตำแหน่งที่ "วัดได้" กับตำแหน่งที่ "รู้แน่นอน" ความแตกต่างคือ aggregate error ของระบบ ณ ขณะนั้น (ionospheric delay, satellite clock error, orbital error) ซึ่งแม่นยำกว่าการประมาณจากโมเดล

  2. 2
    Encode เป็น RTCM3 Message แล้วส่งผ่าน NTRIP

    Base encode ค่า correction เป็น RTCM3 binary format (ขนาด ~250 bytes/message) ส่งผ่าน NTRIP Caster (เช่น RTK2GO.com หรือ Onocoy) ด้วยอัตรา 1 message/วินาที Rover ต่อ internet แล้วดึง stream ผ่าน HTTP persistent connection

  3. 3
    Rover รับ Correction + Resolve Ambiguity

    Rover รับ RTCM3 message แล้วใช้ร่วมกับ carrier phase measurement ของตัวเอง algorithm LAMBDA iterates จนหาจำนวนเต็มของ cycle ที่เป็นไปได้มากที่สุด เมื่อ "fix" สำเร็จ ความแม่นยำกระโดดจาก "float solution" (~30 cm) เป็น "fixed solution" (~2 cm) ทันที

  4. 4
    Track Fix Status ต่อเนื่อง

    หลัง fix แล้ว Rover track การเปลี่ยนแปลงของ phase ต่อเนื่อง (cycle slip detection) ถ้าสัญญาณถูกบัง (อาคาร, ต้นไม้) และ cycle slip เกิดขึ้น Rover ต้องทำ re-initialize ซึ่งใช้เวลา 5–60 วินาทีอีกครั้ง นี่คือข้อจำกัดสำคัญในสภาพแวดล้อม urban ที่มีตึกล้อมรอบ

ความสำคัญของระยะห่าง Base–Rover

Correction Data จาก Base "แม่นยำ" สำหรับ Rover ก็ต่อเมื่อทั้งคู่อยู่ในสภาพ atmospheric error ที่คล้ายกัน ถ้าอยู่ห่างกัน 20 km ค่า ionospheric delay ที่ Base วัดได้ยังคล้ายกับที่ Rover เจออยู่พอ ถ้าห่างเกิน 20 km error ของทั้งคู่เริ่มแตกต่างกัน ทำให้ Correction ไม่ถูกต้อง

นี่คือเหตุผลว่าทำไม Network RTK (หลาย Base Station ครอบคลุม grid) จึงสำคัญ — Rover สามารถเลือกใช้ Base ที่ใกล้ที่สุดได้เสมอ Onocoy และ GEODNET build network นี้โดยให้ community เป็นคนติดตั้ง Base Station แล้วจ่าย Token เป็น incentive

เจาะลึก: RTCM3 Message Types — ส่งข้อมูลอะไรบ้าง?

RTCM3 มี message type หลายร้อยประเภท แต่ที่ใช้บ่อยใน RTK คือ: Type 1005/1006 — ตำแหน่ง Base Station (ARP: Antenna Reference Point); Type 1074–1127 — MSM (Multiple Signal Messages) สำหรับ GPS, GLONASS, Galileo, BeiDou ทั้ง pseudorange และ carrier phase; Type 4076 — State Space Representation (SSR) สำหรับ network corrections

Onocoy และ GEODNET รับ RTCM3 จาก Base Station แล้วแจกจ่ายให้ Rover ทั่วโลก Rover เพียงแค่ configure endpoint URL + credentials ก็เริ่มได้รับ Correction ทันที ไม่ต้องรู้ว่า Base อยู่ที่ไหนในโลก

★ Intermediate takeaway: Fix time 5–60s คือ cold-start delay ที่หลีกเลี่ยงไม่ได้ — วางแผน workflow ให้ Rover warm-up ก่อนเริ่มงานที่ต้องการ cm accuracy

Section 3 — Correction Network · สัญชาตญาณ

NTRIP, RTK2GO และ Onocoy: Internet ส่ง Correction ได้อย่างไร

หนึ่งในความก้าวหน้าที่สำคัญที่สุดของ RTK สมัยใหม่คือการส่ง Correction Data ผ่าน Internet แทนที่จะต้องใช้ Radio ระยะสั้น ก่อนหน้านี้ Base Station และ Rover ต้องอยู่ใกล้กันและมี Radio Link ตรง ทำให้ต้นทุนสูงและยืดหยุ่นน้อย NTRIP Protocol เปลี่ยนเกมทั้งหมด — ตอนนี้ Base Station ของคุณอยู่กรุงเทพฯ สามารถส่ง Correction ให้ Rover ในเชียงใหม่ได้โดยตรงผ่าน Internet

NTRIP Protocol: HTTP สำหรับ GPS Correction

NTRIP (Networked Transport of RTCM via Internet Protocol) ออกแบบมาในปี 2004 โดย Federal Agency for Cartography and Geodesy ของเยอรมนี ทำงานบน HTTP/TCP Port 2101 โดยมีสามส่วนหลัก:

  • NTRIP Source (Broadcaster): Base Station ที่ส่ง RTCM3 stream ขึ้นไปยัง Caster ต่อเนื่อง ตราบใดที่มี Internet
  • NTRIP Caster: Server กลาง (เช่น RTK2GO.com, EUREF-IP) รับ stream จาก Source หลายตัว แล้วแจกจ่ายให้ Client ตามที่ขอ — เหมือน "radio station" บน Internet
  • NTRIP Client (Rover): ส่ง HTTP GET request พร้อม mountpoint ของ Base ที่ต้องการ แล้วรับ RTCM3 stream ต่อเนื่อง forward เข้า GNSS receiver ผ่าน serial port

NTRIP flow: Base → Caster (RTK2GO/Onocoy/GEODNET) → Client (ESP32/phone) → Rover → ±2 cm

RTK2GO vs Onocoy vs GEODNET: เลือกอะไร?

RTK2GO — NTRIP Caster ฟรี

RTK2GO.com เป็น NTRIP Caster สาธารณะที่ให้บริการฟรี เหมาะสำหรับการทดสอบและ hobbyist ที่ต้องการ Correction Data สำหรับใช้เองโดยไม่ได้ต้องการ earn rewards ผู้ใช้สามารถ register Base Station ได้ฟรีและ share ให้คนอื่นใช้ได้

ข้อเสีย: ไม่มี incentive ให้ Base Station operator เพราะไม่ใช่ DePIN model, coverage ขึ้นอยู่กับ volunteer, ไม่รับประกัน uptime หรือ quality

เหมาะสำหรับ: ทดสอบ setup ครั้งแรก, DIY project ที่ไม่ต้องการ production reliability

Onocoy — DePIN GNSS Marketplace

Onocoy เป็น DePIN network ที่ focus เฉพาะ GNSS/RTK correction data ผู้ run Base Station รับ $ONO token เป็น reward ตาม contribution quality Onocoy ขาย Correction Data ให้ enterprise ด้วย subscription model แล้วนำรายได้ส่วนหนึ่งกลับไปจ่าย operator

ข้อเสีย: ต้องการ compatible hardware (multi-band GNSS receiver), network ยังอยู่ระหว่าง growth phase, token price volatility

เหมาะสำหรับ: ต้องการ earn crypto จาก Base Station ที่ตั้งถาวร, อยู่ในพื้นที่ที่ยังขาด coverage

GEODNET — Blockchain-Powered GNSS

GEODNET เป็น DePIN network ที่ใหญ่กว่า Onocoy และมีฐาน investor จาก crypto world ผู้ run station รับ $GEOD token โดย GEODNET ใช้ hex-based reward zone system — station แต่ละตัวครอบคลุม hex cell ถ้า cell นั้นมี station น้อย reward สูงกว่า ถ้ามี station มากแล้ว reward ลด

ข้อเสีย: hardware ที่รองรับอย่างเป็นทางการมักราคาสูงกว่า ($300+), reward system ซับซ้อน, ต้องศึกษา tokenomics ก่อน

เหมาะสำหรับ: ต้องการ ecosystem ที่ใหญ่กว่า, พื้นที่ที่ยังไม่มี station ใน hex cell
★ Beginner takeaway: NTRIP คือ "HTTP streaming สำหรับ GPS correction" — Base push, Caster relay, Rover pull ผ่าน TCP connection เดียว

Section 3 — Correction Network · กลไก

การตั้งค่า NTRIP Connection และ Network Coverage

ส่วนนี้อธิบายขั้นตอนการเชื่อมต่อ Rover เข้ากับ NTRIP Caster ในทางปฏิบัติ รวมถึงวิธีตรวจสอบว่ามี Base Station ครอบคลุมพื้นที่ที่ต้องการหรือไม่ และ config ที่ต้องตั้งใน GNSS receiver

Config ที่ต้องมีสำหรับ NTRIP Client

การเชื่อมต่อ Rover กับ NTRIP Caster ต้องการข้อมูลสี่อย่าง ซึ่งทั้งหมดได้มาจากการ register บนเว็บไซต์ของ Caster (RTK2GO, Onocoy, GEODNET) ก่อนใช้งาน:

# NTRIP Client config (u-blox ZED-F9P via ESP32)
Host:        rtk2go.com      # or ntrip.onocoy.com
Port:        2101
Mountpoint:  MY_BASE_STATION # ชื่อ station ที่ register ไว้
Username:    your@email.com  # สำหรับ RTK2GO
Password:    none            # RTK2GO ไม่ต้องใช้ password
GGA:         enable          # ส่ง position ของ Rover กลับไป Caster

การ enable GGA (NMEA sentence ที่มีตำแหน่ง Rover) สำคัญมาก เพราะ Caster ใช้ข้อมูลนี้เพื่อ route ไปยัง Base Station ที่ใกล้ที่สุดโดยอัตโนมัติ (VRS: Virtual Reference Station mode) และเพื่อ verify ว่า Rover อยู่ใน coverage zone

วิธีตรวจสอบ Coverage ก่อนติดตั้ง

ก่อนลงทุนซื้อ hardware ควรตรวจสอบก่อนว่ามี Base Station ครอบคลุมพื้นที่ที่ต้องการหรือไม่ ถ้าไม่มี — นั่นคือโอกาสสำหรับ DePIN operator ที่จะ earn rewards มากกว่า

Onocoy Coverage

onocoy.com/map

แผนที่ interactive แสดง active station ของ Onocoy network ทั่วโลก พื้นที่สีน้ำเงินหมายถึงมี coverage แล้ว พื้นที่ขาวคือโอกาสสำหรับ new operator

GEODNET Coverage

rtk.geodnet.com/coverage

แผนที่ hex grid ของ GEODNET แสดง reward multiplier แต่ละ cell สีเขียวเข้ม = reward สูง (มี station น้อย), สีเขียวอ่อน = reward ต่ำ (saturated area)

เจาะลึก: VRS (Virtual Reference Station) คืออะไร ต่างจาก Single Base อย่างไร?

VRS คือเทคนิคที่ใช้ข้อมูลจาก Base Station หลายตัวรอบๆ Rover เพื่อ interpolate correction ที่ "เหมือนกับว่า Base อยู่ตรงตำแหน่ง Rover เลย" ทำให้ precision สูงขึ้นโดยเฉพาะเมื่อ Rover อยู่ห่างจาก Base Station จริงมากกว่า 10 km

Network RTK อย่าง Onocoy และ GEODNET ให้ VRS correction โดยอัตโนมัติเมื่อมี Base หลายตัวในบริเวณ Rover ส่งผลให้ reliability และ accuracy ดีกว่า Single Base โดยเฉพาะพื้นที่ที่มี station density สูง

ข้อจำกัด: VRS ต้องการ Rover ส่ง GGA message กลับมาให้ Caster เพื่อรู้ว่า Rover อยู่ที่ไหน บาง firmware version ของ GNSS receiver ไม่ส่ง GGA โดย default ต้องเปิดใช้งานเอง

★ Intermediate takeaway: Enable GGA ใน NTRIP client เสมอ — Caster ต้องรู้ตำแหน่ง Rover เพื่อ route correction ที่ถูกต้อง ถ้า disable จะได้ correction จาก Base ที่ไม่ใช่ที่ใกล้ที่สุด

Section 4 — DIY Hardware · ส่วนประกอบ

สร้าง Base Station ด้วยตัวเอง: ESP32 + ZED-F9P <$300

Commercial RTK Base Station จากบริษัทอย่าง Leica, Trimble หรือ Septentrio มีราคา $700–$10,000 แต่ฟังก์ชันหลักที่ต้องการสำหรับ DePIN network นั้นสามารถ replicate ได้ด้วย hardware ราคาต่ำกว่า $300 ประกอบด้วย 4 components หลัก ที่แต่ละชิ้นทำหน้าที่เฉพาะและสามารถซื้อแยกได้ตามงบประมาณ

4 Components หลักของ DIY RTK Base Station

Component 1 — GNSS Receiver

u-blox ZED-F9P

Multi-band GNSS receiver ที่รับ L1+L2+L5 จาก GPS, GLONASS, Galileo, BeiDou พร้อมกัน ราคาประมาณ $180–$220 ถือเป็น standard สำหรับ RTK DIY community ให้ carrier phase measurement และรองรับ RTCM3 output โดยตรง

  • Interface: UART / USB / SPI / I2C
  • Update rate: ถึง 20 Hz
  • Fix time: ~10 s (multi-band)

Component 2 — Antenna

Multi-band Survey Antenna

Antenna ที่รองรับ L1+L2+L5 พร้อมกัน (เช่น u-blox ANN-MB-00 หรือ Matek M8Q-5883) สำคัญมากกว่า receiver เพราะ antenna คุณภาพต่ำทำให้ cycle slip บ่อย ลด reward และลด accuracy ราคา $30–$80 ต้องวางในที่ที่มี clear sky view 360°

  • Gain: >40 dBHz per satellite
  • Housing: weatherproof (IP67+)
  • Phase center: stable (<1 mm variation)

Component 3 — Microcontroller

ESP32 (WiFi + 4G option)

ESP32 ทำหน้าที่เป็น bridge ระหว่าง ZED-F9P กับ Internet: อ่าน RTCM3 จาก UART แล้วส่งขึ้น NTRIP Caster ผ่าน WiFi หรือ 4G modem Firmware ที่ใช้บ่อยคือ ESP32-RTK2GO (open source) ซึ่ง config ผ่าน web interface ไม่ต้องเขียนโค้ด ราคา $5–$15

Component 4 — Connectivity

WiFi หรือ 4G LTE Modem

Base Station ที่ติดตั้งถาวรนอกอาคารมักใช้ 4G LTE modem (เช่น SIM7600) แทน WiFi เพราะไม่ต้องพึ่ง router และ range ไม่จำกัด ค่าใช้จ่าย SIM card ประมาณ $5–$15/เดือน ขึ้นอยู่กับ data plan (ใช้ bandwidth น้อย ~5 MB/วัน)

Section 4 — DIY Hardware · ขั้นตอน Setup

ขั้นตอนการ Setup Base Station ตั้งแต่ต้น

เมื่อมี component ครบแล้ว การต่อและ configure ไม่ซับซ้อนอย่างที่คิด ต้องการเพียงการต่อสาย 4 เส้น, upload firmware, กรอก config ผ่าน web interface แล้วรอ survey-in ซึ่งเป็นกระบวนการที่ Base Station "วัดตำแหน่งของตัวเอง" ก่อนเริ่มส่ง Correction

ขั้นตอน Hardware Setup

  1. 1
    ต่อสาย ZED-F9P → ESP32 (4 เส้น)

    เชื่อมต่อ UART1 ของ ZED-F9P เข้ากับ ESP32: TX (F9P) → RX (ESP32 GPIO16), RX (F9P) → TX (ESP32 GPIO17), VCC (3.3V), GND Baud rate default คือ 38400 แต่ firmware จะตั้งให้อัตโนมัติ ไม่ต้องเปลี่ยน

  2. 2
    Flash ESP32 ด้วย RTK firmware

    Download firmware จาก project page แล้ว flash ด้วย esptool.py หรือ ESP32 Flash Tool (GUI) เมื่อ boot ครั้งแรก ESP32 สร้าง WiFi Access Point ชื่อ RTK_Setup เชื่อมต่อแล้วเปิด 192.168.4.1 เพื่อเข้า config UI

  3. 3
    กรอก NTRIP Credentials และ Network

    ใน web config UI กรอก: WiFi SSID + password (หรือ APN สำหรับ 4G), NTRIP Caster host/port/mountpoint/credentials ของ Onocoy หรือ GEODNET บันทึกแล้ว reboot — ESP32 จะเชื่อมต่อ internet และเริ่ม push RTCM3 ขึ้น Caster อัตโนมัติ

  4. 4
    รอ Survey-In (1–24 ชั่วโมง)

    Survey-in คือกระบวนการที่ Base Station เฉลี่ยตำแหน่ง GPS ของตัวเองนานพอจนได้ตำแหน่งที่เสถียร (accuracy <2 m หรือ ตามที่ตั้ง threshold) ZED-F9P รองรับ survey-in แบบ autonomous — ตั้ง minimum duration 300 วินาที + position accuracy 2 m ก็เพียงพอสำหรับ DePIN เพราะ absolute position ไม่สำคัญเท่า relative correction

  5. 5
    Verify บน Dashboard ของ Caster

    Login บน Onocoy หรือ GEODNET dashboard เพื่อยืนยันว่า station ของคุณ online และส่ง RTCM3 ถูกต้อง dashboard จะแสดง: uptime, satellite count, signal quality score, และ estimated reward สำหรับวันนั้น

Tips เพื่อ Maximize Rewards

Onocoy และ GEODNET คำนวณ reward จาก signal quality ซึ่งขึ้นอยู่กับหลายปัจจัย การ optimize แต่ละปัจจัยเหล่านี้จะเพิ่ม token ที่ได้รับต่อวันอย่างมีนัยสำคัญ:

DO ✓

ทำเพื่อ Reward สูง

  • วาง antenna ในที่ที่มี clear sky view >60° elevation
  • ใช้ multi-band antenna (L1+L2+L5)
  • ใช้ stable internet (uptime >99%)
  • ติดตั้งในพื้นที่ที่ยังไม่มี station ใน hex cell

DON'T ✗

หลีกเลี่ยงสิ่งเหล่านี้

  • วาง antenna ใกล้อาคารหรือต้นไม้ (multipath error)
  • ใช้ single-band antenna เพื่อลดต้นทุน
  • ใช้ WiFi ที่ไม่เสถียรหรือมี outage บ่อย
  • ติดตั้งในพื้นที่ที่มี station อิ่มตัวแล้ว (reward ต่ำ)
★ Intermediate takeaway: การต่อสายเพียง 4 เส้นและ upload firmware คือทั้งหมดที่ต้องทำด้าน hardware — ความซับซ้อนอยู่ที่การเลือกตำแหน่งติดตั้ง antenna และการ configure NTRIP credentials

Section 5 — DePIN Business Model · สัญชาตญาณ

DePIN: เปลี่ยน Physical Infrastructure จาก Cost Center เป็น Revenue Stream

DePIN (Decentralized Physical Infrastructure Networks) คือแนวคิดที่ใช้ Blockchain Token เป็น Incentive Mechanism เพื่อให้บุคคลทั่วไปช่วยกันสร้างและดูแลรักษา Physical Infrastructure เช่น WiFi hotspot, โครงข่าย sensor, หรือ RTK Base Station แทนที่จะให้บริษัทขนาดใหญ่ลงทุนเพียงรายเดียว วิธีนี้ทำให้ infrastructure สามารถ scale ได้รวดเร็วกว่าและมีต้นทุนน้อยกว่า เพราะกระจายทั้ง risk และ reward ไปยังผู้ใช้งานจริง

ทำไม DePIN ถึงใช้งานได้กับ RTK Network

RTK Network ต้องการ Base Station จำนวนมากกระจายตัวอยู่ทั่วโลก เพื่อให้ Rover ทุกตัวมี Base Station อยู่ในรัศมี 20 km การสร้าง network แบบนี้ด้วยวิธีดั้งเดิมต้องใช้เงินลงทุนหลายสิบล้านดอลลาร์และทีม operations ขนาดใหญ่ ซึ่งทำให้บริการ RTK Correction มีราคาแพงมาก

DePIN แก้ปัญหานี้ด้วยวิธีที่ elegant: แทนที่จะบริษัทเป็นคนติดตั้ง Base Station ทุกตัว บริษัทออก Token และสร้าง marketplace สำหรับซื้อขาย Correction Data แทน ผู้ใช้งานที่มีบ้านหรือออฟฟิศในพื้นที่ที่ยังขาด coverage สามารถซื้อ hardware ราคาต่ำกว่า $300 ติดตั้ง Base Station แล้วรับ Token เป็น reward ขณะที่ network ขายข้อมูล Correction ให้ enterprise

DePIN flywheel: Operator ส่ง RTCM3 → Network ขาย Correction ให้ Enterprise → รายได้ fund Token Rewards → Operator ได้ Token กลับมา

Section 5 — DePIN Business Model · Tokenomics

วิธีคำนวณ Reward และ Tokenomics ของ Onocoy & GEODNET

การทำความเข้าใจ reward mechanism ของแต่ละ network สำคัญมากก่อนตัดสินใจลงทุน hardware เพราะ reward ขึ้นอยู่กับหลายปัจจัยที่ควบคุมได้และควบคุมไม่ได้ ทั้ง signal quality, geographic location, network saturation, และ token price ซึ่งแปรผันตามตลาด crypto

Onocoy Reward Mechanism

Onocoy ใช้ reward model ที่ base บน "contribution score" ซึ่งคำนวณจากสี่ปัจจัยหลัก: จำนวนและคุณภาพของ satellite ที่รับได้ (multi-band = score สูงกว่า), uptime percentage, geographic uniqueness (พื้นที่ที่ไม่มี station อื่นใกล้ = score สูงกว่า), และ data quality (cycle slip rate ต่ำ = ดีกว่า) Reward จ่ายเป็น $ONO token รายวัน

กลไกสำคัญที่ทำให้ Onocoy sustainable: รายได้จาก enterprise subscription (ผู้ซื้อ Correction Data) ถูกใช้เพื่อซื้อ $ONO token จาก open market แล้วนำมา distribute ให้ operator — ทำให้ token มี real demand backing ไม่ใช่แค่ inflation

GEODNET มีทั้ง basic tier (ต้องการ hardware ผ่าน GEODNET official store ~$450) และ DIY tier (hardware ของตัวเอง แต่ต้อง verify compatibility ก่อน) reward ใน DIY tier อาจต่ำกว่าเล็กน้อยเนื่องจาก hardware quality verification

Section 6 — Use Cases

ประยุกต์ใช้ RTK ±2 cm ในโลกจริง

ความแม่นยำระดับเซนติเมตรเปิดประตูให้กับ application ที่เป็นไปไม่ได้ด้วย GPS ทั่วไป ตั้งแต่โดรนที่ต้อง land อัตโนมัติบน moving platform ไปจนถึงหุ่นยนต์เกษตรที่ต้องเดินตามแนวปลูกพืชอย่างแม่นยำ use cases เหล่านี้แสดงให้เห็นว่าทำไม RTK correction data จึงเป็น multi-million dollar market ที่ DePIN กำลัง democratize

Use Cases หลัก 4 ด้าน

Use Case 1 — Drones

โดรนสำรวจและโดรนเกษตร

โดรนที่ใช้ RTK สามารถ capture aerial photograph ที่มี geotag แม่นยำ ±2 cm โดยไม่ต้องใช้ GCP (Ground Control Points) ซึ่งปกติต้องวางด้วยมือก่อนบิน ประหยัดเวลาได้มาก สำหรับโดรนพ่นยา RTK ช่วยให้บินตาม waypoint ซ้ำได้อย่างแม่นยำทุกครั้งแม้ในสภาพลมแรง

Base Station ต้องวางใกล้พื้นที่บิน หรือใช้ NTRIP network ถ้ามี coverage แล้ว บาง setup ใช้ portable base station วางบนรถแล้วเชื่อมต่อกับ drone ผ่าน WiFi

Use Case 2 — Robotic Mowers

หุ่นยนต์ตัดหญ้าอัตโนมัติ

High-end robotic mower อย่าง Husqvarna CEORA ใช้ RTK โดยติดตั้ง Base Station ที่ dock ของหุ่นยนต์ ทำให้หุ่นยนต์รู้ตำแหน่งตัวเองบนสนามได้อย่างแม่นยำ ±2 cm ไม่ต้องพึ่ง perimeter wire หรือ beacon เหมือน mower รุ่นเก่า สามารถ map สนามได้แม้มีรูปร่างซับซ้อน

Use Case 3 — Precision Agriculture

เกษตรกรรมแม่นยำ

รถไถและเครื่องหว่านเมล็ดที่ใช้ RTK สามารถเดินตาม track ซ้ำได้อย่างแม่นยำทุกฤดูกาล ช่วยลด overlap ระหว่าง pass ทำให้ประหยัดเชื้อเพลิง ปุ๋ย และยาฆ่าแมลง ฟาร์มขนาดใหญ่ในออสเตรเลียและอเมริกาใช้ RTK มาหลายปีแล้ว แต่ DePIN ทำให้ฟาร์มขนาดเล็กในประเทศกำลังพัฒนาเข้าถึงได้ด้วย

Use Case 4 — Autonomous Vehicles

ยานพาหนะไร้คนขับ

ในสภาพแวดล้อม outdoor ที่ camera และ LiDAR อาจ confused ด้วยแสงหรือฝุ่น RTK GPS ให้ absolute position ที่เชื่อถือได้ตลอดเวลา ใช้เป็น fallback หรือ primary positioning สำหรับ AGV (Automated Guided Vehicle) ในเหมือง, port, และ logistics facility ที่ต้องการ precision สูง

เมื่อไหรที่ RTK ไม่ใช่คำตอบที่ดีที่สุด

RTK ทำงานได้ดีใน outdoor environment ที่มี clear sky view แต่มีข้อจำกัดในบางสถานการณ์ที่ควรรู้ก่อนเลือกใช้:

  • Indoor positioning: RTK ไม่ทำงาน indoor เพราะ GNSS signal ไม่ผ่านหลังคา ต้องใช้ UWB, WiFi RTT, หรือ LiDAR-based SLAM แทน
  • Urban canyon: ถนนที่ล้อมรอบด้วยตึกสูงทำให้ cycle slip บ่อย ทำให้ fix loss บ่อยและ accuracy ลด ในกรณีนี้ sensor fusion กับ IMU ช่วยได้
  • Real-time requirement ต่ำกว่า 5s: ถ้าต้องการ fix ทันทีโดยไม่รอ TTFF ต้องใช้ PPP (Precise Point Positioning) ซึ่ง fix ได้โดยไม่ต้องมี Base Station แต่ accuracy อยู่ที่ ~10 cm
★ สาระสำคัญ: RTK เหมาะที่สุดสำหรับ outdoor automation ที่ต้องการ ±2 cm และมีเวลา warm-up 5–60s — ทุกอย่างอื่นมี alternative ที่ดีกว่า

Section 7 — Caveats & FAQ

ข้อจำกัดและคำถามที่พบบ่อยจากการใช้งานจริง

RTK และ DePIN มีข้อจำกัดและ gotcha หลายอย่างที่ documentation มักไม่พูดถึงตรงๆ ส่วนนี้รวบรวมปัญหาที่ผู้ใช้มักเจอหลังติดตั้งไปสักพัก เพื่อให้สามารถวางแผนและออกแบบระบบได้อย่างถูกต้องตั้งแต่ต้น

Cycle slip คือสาเหตุหลัก เกิดขึ้นเมื่อ Rover สูญเสีย lock บน carrier wave ชั่วคราว (เช่น ขับรถผ่านใต้สะพาน, ต้นไม้บัง, หรือ signal interference จากอุปกรณ์ไฟฟ้า) เมื่อ cycle slip เกิดขึ้น Integer Ambiguity ที่ resolve ไว้แล้วจะ invalid ทันที ต้อง re-initialize ใหม่ซึ่งใช้เวลา 5–60 วินาที

วิธีแก้: ใช้ multi-band receiver (L1+L2) เพราะสามารถ detect และ repair cycle slip ได้เร็วกว่า single-band มาก; ใน mission-critical application ใช้ sensor fusion กับ IMU เพื่อ "bridge" ช่วงที่ fix หลุด; ตรวจสอบ CNR (Carrier-to-Noise Ratio) ที่ <35 dBHz บ่งบอกว่า signal อ่อนเกินไป

เมื่อไหรที่ cycle slip ไม่ใช่ปัญหา: application ที่ไม่ต้องการ continuous fix เช่น survey ที่ stop-and-go หรือ machine ที่ทำงานในพื้นที่โล่ง

Survey-in แบบ autonomous ให้ความแม่นยำของตำแหน่ง Base ประมาณ ±1–2 m ซึ่งเพียงพอสำหรับ relative positioning (Rover รู้ตำแหน่งของตัวเองเทียบกับ Base ได้ ±2 cm) แต่ไม่เพียงพอสำหรับ absolute positioning (ตำแหน่งบนแผนที่โลก)

ถ้าต้องการ absolute accuracy ±2 cm บน WGS84 coordinate system ต้อง "fix" ตำแหน่ง Base ด้วยวิธีอื่น: CORS network correction (ถ้ามีในประเทศ), PPP-AR service (เช่น Trimble CenterPoint RTX), หรือ occupying known benchmarkpoint สำหรับงานที่ต้องการ coordinate ที่แน่นอนอย่างงาน surveying มืออาชีพ

สำหรับ DePIN purposes: survey-in อัตโนมัติ ±1–2 m เพียงพอสมบูรณ์ เพราะ network ต้องการ relative correction ไม่ใช่ absolute position

"Float solution" หมายความว่า Rover ได้รับ Correction Data แล้วแต่ยังไม่สามารถ resolve Integer Ambiguity ได้ (accuracy อยู่ที่ ~10–30 cm แทน ±2 cm) สาเหตุที่พบบ่อยคือ: Base และ Rover ห่างกันเกิน 20 km, จำนวน common satellite น้อยเกินไป (<5 ดวง), หรือ ionospheric disturbance สูง (ช่วง solar maximum)

วิธีแก้: ตรวจสอบว่า mountpoint ที่เลือกนั้น Base อยู่ใน 20 km จริงหรือไม่; เพิ่ม elevation mask ให้ต่ำลง (จาก 15° เป็น 10°) เพื่อรับ satellite มากขึ้น; รอ ionospheric condition ดีขึ้น (โดยเฉพาะช่วงหัวเช้าหรือกลางคืน)

ขึ้นอยู่กับ 3 ปัจจัยหลัก: (1) location — พื้นที่ที่ยังไม่มี station ใน hex cell จะได้ reward สูงกว่า; (2) token price — ณ ต้นปี 2025 token reward อาจ cover hardware cost ใน 6–12 เดือนสำหรับ location ที่ดี แต่ token price สามารถลดลง 50% ได้ใน bear market; (3) hardware longevity — hardware ที่ดีอยู่ได้ 5+ ปี ทำให้ ROI long-term ดีกว่า

คำแนะนำ: ถ้าซื้อ hardware เพราะต้องการใช้ RTK เองอยู่แล้ว (เช่น สำหรับ drone หรือ robot project) การเชื่อมต่อกับ DePIN network เป็น "bonus income" ที่ risk ต่ำ แต่ถ้าซื้อ hardware เพื่อ mine token อย่างเดียว ต้องวิเคราะห์ ROI อย่างรอบคอบก่อน

⚠ Gotcha ที่สำคัญที่สุด: Survey-in ที่ไม่ครบเวลา (ต่ำกว่า 300 วินาที) จะทำให้ Base ส่ง Correction ที่มี systematic error ทุก Rover ที่ใช้ Correction จาก Base นั้นจะ offset ตำแหน่งไปพร้อมกัน ซึ่งตรวจจับยากมาก — รอ survey-in ให้ครบเสมอ
★ สาระสำคัญ: ปัญหา 80% ของ RTK setup มาจาก antenna placement (multipath), ระยะ Base-Rover (เกิน 20 km), หรือ survey-in ที่ไม่ครบ — แก้สามเรื่องนี้ก่อนอย่างอื่น

Closing — Key Takeaways

สิ่งที่ต้องจำและขั้นตอนถัดไป

หลังอ่าน explainer นี้จบ ผู้อ่านควรมีความเข้าใจครบถ้วนว่า RTK-GPS ทำงานอย่างไร จะ build DIY Base Station ได้อย่างไร และ DePIN เปลี่ยน economics ของ precision positioning อย่างไร

Key Takeaways

  • RTK วัด Phase ไม่ใช่แค่ Timing: ความแม่นยำ ±2 cm มาจากการวัด carrier phase (wavelength 19 cm) ผ่าน Integer Ambiguity Resolution ไม่ใช่แค่การ upgrade hardware ราคาแพงขึ้น
  • Base Station = กุญแจทั้งระบบ: Base ต้องอยู่ใกล้ Rover ≤20 km และต้องรู้ตำแหน่งตัวเองด้วย survey-in ที่ครบเวลา — ถ้า Base ผิด ทุกอย่างผิดหมด
  • Multi-band Antenna ไม่ใช่ option: Antenna L1+L2 คือ requirement สำหรับ reliable RTK fix time ที่ยอมรับได้ single-band อาจใช้ได้แต่ slow และ unreliable ใน challenging sky
  • DIY ทำได้ต่ำกว่า $300: ESP32 + u-blox ZED-F9P + multi-band antenna + 4G modem คือ stack ที่ทำงานได้จริงและ qualify สำหรับ DePIN rewards ไม่ต้องซื้อ commercial hardware $700+
  • DePIN เปลี่ยน cost เป็น revenue: Onocoy (ONO) และ GEODNET (GEOD) จ่าย token ให้ Base Station operator ตาม signal quality — ROI ขึ้นกับ location, token price, และ network demand
  • TTFF 5–60 วินาทีคือ fundamental limit: ต้องวางแผน workflow ให้ Rover warm-up ก่อนเริ่มงาน precision — ไม่มีทาง bypass กระบวนการ ambiguity resolution นี้
  • NTRIP = HTTP streaming สำหรับ GPS correction: Base push RTCM3 → Caster relay → Rover pull ผ่าน TCP connection เดียว ต้อง enable GGA ใน client เพื่อให้ Caster route ไปยัง Base ที่ใกล้ที่สุด
